# 02 · 모델 라우팅 거버넌스 (Model Routing Governance)

> 단일 출처: [`docs/_canon.md`](../_canon.md) §4 법령 · §8 운영 계약 · [데이터 거버넌스/PII](../02_제품/element-specs/07-data-governance-pii.md).
> 심사 대응: 2.4(규제·보안·개인정보·내부통제 인식), 5.5(개인정보·보안·책임소재 리스크 대응).

핵심 차별점: **외부 프런티어 LLM의 추론력은 쓰되, 고객 원본 PII·신용정보는 절대 외부로 반출하지 않는다.** 이를 데이터 등급제·토큰화·모델 라우팅·반출 스캔·감사 원장의 4중 방어로 보장한다. 본 문서는 그 운영 규칙을 표준화한다.

---

## 1. 처리 파이프라인 (tokenize → route → egress-scan → audit)

모든 외부 LLM 호출은 아래 4단계 관문을 순서대로 통과한다. 한 단계라도 실패하면 호출이 중단된다.

```
[Case 데이터]
   │
   1) classify  ── 필드별 데이터 등급 부여 (public/internal/confidential/restricted)
   │
   2) tokenize  ── restricted(PII)·confidential 필드를 토큰으로 치환
   │              토큰↔원본 매핑은 국내 PII 볼트(온프레)에만 존재
   │
   3) route     ── 작업 민감도로 국내·온프레 vs 외부 프런티어 모델 분기
   │
   4) egress-scan ── 외부행이면 전송 전 페이로드 스캔(주민번호/계좌/전화 정규식 + 토큰 검증)
   │              PII 패턴 적중 시 → 차단 + 에스컬레이션
   │
   5) audit     ── 모든 접근·전송·재식별을 감사 원장(무결성 해시)에 기록
```

```
원본:   "김민수(900101-1******) 010-1234-5678, 전주시 완산구 ..."
토큰화: "{{CUST_7F2}} {{PHONE_A1}} {{ADDR_3C}}"   ← 외부 모델은 이것만 본다
재식별: 국내 볼트에서만 토큰→원본 (사람 승인·감사 후)
```

---

## 2. 데이터 등급 규칙 (Data Tiers)

모든 필드에 등급을 부여하고, 등급이 외부 반출 가능 여부를 결정한다.

| 등급 | 예시 필드 | 외부 반출 |
| --- | --- | --- |
| `public` | 공시자료, 기사, 법령, 정책 | 허용 |
| `internal` | 케이스 코드, 리스크 점수, 상태 | 허용(비식별 상태) |
| `confidential` | 매출 추이, 대출 잔액(범위화) | **토큰화 후** 허용 |
| `restricted` (PII) | 성명, 주민번호, 계좌, 연락처, 주소 | **반출 금지** |

---

## 3. 라우팅 결정 표 (Routing Decision Table)

`route(taskKind, hasPII) → "onprem" | "external"`

| 작업 종류 | 최고 데이터 등급 | 라우팅 | 근거 |
| --- | --- | --- | --- |
| 원본 PII 필요한 식별·대조(재식별, 본인확인) | `restricted` | **국내·온프레** (외부 절대 금지) | PII 비반출 |
| 비식별 요약·분류·초안 작성 | `confidential`(토큰화 후) | **외부 프런티어** (토큰화 입력) | 토큰화로 원본 미노출 |
| 법령/정책/뉴스 등 공개정보 분석 | `public` | **외부 프런티어** (PII 없음) | PII 부재 |
| 내부 지표 집계·큐 건강도 | `internal` | 국내·온프레 또는 외부(비식별) | 비식별 상태 |
| 토큰↔원본 재식별 | `restricted` | **국내 볼트 전용** + 사람 승인·감사 | 신용정보법 §40조의2 |

> 기본 원칙: **PII가 한 필드라도 토큰화되지 않은 채 페이로드에 남으면 외부 라우팅 불가.** egress-scan이 최후 방어선이다.

---

## 4. 검증된 법령·규제 근거

> 은행/금융회사는 **신용정보법이 특별법으로 우선**(신용정보법 §3조의2). 개인정보보호법은 보충 적용한다. 금융사 신용정보 가명처리에 PIPA만 인용하면 감점 위험.

### 가명처리·토큰화 정당화

| 설계 요소 | 1차 근거 (특별법) | 보충 근거 (일반법) |
| --- | --- | --- |
| 토큰↔원본 키 **분리보관**·안전조치·처리기록 | **신용정보법 §40조의2 ①②⑧** | 개인정보보호법 **§28조의4** |
| **재식별 금지**·재식별 시 즉시 중단·회수·파기 | **신용정보법 §40조의2 ⑥⑦** | 개인정보보호법 **§28조의5** |
| 가명정보 결합은 지정 전문기관에서만 | — | 개인정보보호법 **§28조의3** |
| 동의 없는 가명정보 처리(통계·연구·공익) | — | 개인정보보호법 **§28조의2** |

### 외부/클라우드 AI 활용 정당화 (망분리 로드맵)

- **전자금융감독규정 §15조 ①항(3호·5호)** — 내부 업무망의 인터넷·외부망 분리(망분리), 데이터센터·개발/운영/보안 단말 물리적 분리. *(법령 개정 진행 중 → 제출 직전 law.go.kr 현행 항·호 번호 재확인)*
- **금융분야 클라우드컴퓨팅서비스 이용 가이드** (금융보안원 FSEC, **2025 개정**) — 전자금융감독규정 §14조의2 클라우드 이용 운영 가이드.
- **금융분야 망분리 개선 로드맵** (금융위원회 FSC, **2024-08-13**) — **다층보안체계(MLS)** 기반 **생성형 AI(클라우드/인터넷) 이용 허용** 및 SaaS/클라우드 범위 확대. → "외부 LLM을 쓰되 PII는 통제한다"는 설계의 정책적 정당화.

출처: law.go.kr, fsc.go.kr(2024-08-13 로드맵), fsec.or.kr(2025 가이드).

> JB금융그룹–네이버클라우드 MOU(2025-12-26)는 기업여신 상담→심사→사후관리 전반의 AI 적용을 **단계적으로 검토하는 추진 방향**이다. "이미 운영 중"으로 표현하지 않는다(캐논 §10).

## 5. 데이터 인터페이스 (갈아끼우기 가능)

```js
governance = {
  tiers: { fieldName: "public|internal|confidential|restricted" },
  classify(field) -> tier,
  tokenizePII(text) -> { masked, map },     // map은 볼트행, UI엔 미표시
  route(taskKind, hasPII) -> "onprem|external",
  egressScan(payload) -> { safe: bool, hits: [] },   // 외부 반출 스캔
  log(event) -> auditEntry
}
```

- 등급 규칙(`tiers`)·라우팅 정책(`route`)은 데이터/함수 교체만으로 변경.
- 토큰화 엔진은 인터페이스(`tokenizePII`)만 지키면 실제 비식별 엔진으로 교체 가능.

## 6. 운영자 책임소재

- **운영 조율 에이전트**가 승인 게이트를 운영하며, 외부 라우팅은 거버넌스 관문 통과 후에만 발생한다.
- 모든 외부 전송·재식별 시도는 감사 원장에 사용자·시각·해시와 함께 남아 사후 추적이 가능하다.
- 외부 모델은 토큰화된 입력만 보므로, 외부 LLM 사고 시에도 원본 PII는 국내 볼트 밖으로 나가지 않는다.
