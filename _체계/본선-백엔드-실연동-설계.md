## 1. 목표 / 제약
- **목표**: 정적 vanilla-JS MVP를 실제 백엔드와 연동해 "목업이 아니라 동작하는 서비스"임을 증명(심사 4.2/4.4), 발전경로(5.4) 내러티브 완성.
- **제약**:
  - 본선 현장 **오프라인**(네트워크·컴퓨팅 미제공) → 로컬 노트북에서 완전 구동 필수.
  - 일정 **D-8**(오늘 6/26 → 본선 7/4) → 신규 학습·재작성 최소화.
  - 기존 25항목 매핑·골든패스·시연 시나리오와 정합 유지.

## 2. 확정 결정 (브레인스토밍 결과)
| 항목 | 결정 |
|------|------|
| 배포 형태 | **로컬 + 클라우드 둘 다** (로컬=시연 안정성/폴백, 클라우드=운영 가능성 어필) |
| 스택 | **Node.js + TypeScript** (프론트·레포 코딩스타일과 일치, 풀스택 단일 언어) |
| LLM 연동 | **Rule Engine 기본 + 실 LLM 토글** (env). 실 LLM 경로엔 **PII 토큰화→반출 스캔→복원** 실제 구현. 로컬은 캐시 응답 폴백 |
| 접근법 | **🅐 계약 우선 씬(thin) 백엔드** — 기존 순수 함수 추출, 재작성 위험 최소 |

## 3. 아키텍처 (접근법 A)
- **단일 코드베이스**가 정적 프론트(`02_제품/app`)와 REST API를 함께 서빙. 로컬은 `node` 실행, 클라우드는 동일 서비스 배포.
- 기존 `app.js`의 **순수 함수 추출** → 서버 도메인 로직으로 이관:
  - `computeRiskDecision` · `buildDashboardData` · `auditChainRecords` · `moveCaseToColumn`
- **REST 표면(초안)**:
  - `GET /api/cases`, `GET /api/cases/:id`, `POST /api/cases/:id/agent-run`(실행), `POST /api/cases/:id/approve|reject`, `GET /api/audit/:caseId`, `GET /api/dashboard`
  - `POST /api/agent` — 거버넌스/LLM 엔드포인트. **env 토글**: `AGENT_MODE=rule|llm`. `llm` 모드는 PII 토큰화→외부 LLM(Claude)→복원, 응답에 토큰화 전/후·반출 스캔 결과 동봉.
- **DB**: 로컬 **SQLite**(파일, 오프라인), 클라우드는 동일 스키마(SQLite 파일 or 매니지드). ORM 경량(예: better-sqlite3 / drizzle) — 결정 보류(구현 단계).
- **프론트 연동**: 작은 **API 클라이언트** + `USE_BACKEND` 플래그. 백엔드 불가 시 기존 **localStorage 경로로 폴백**(시연 안전망).
- **오프라인 보장**: 정적 자산의 외부 CDN(Pretendard 폰트) 로컬화. LLM 토글 OFF면 네트워크 0 의존. LLM ON 시연은 **사전 캐시 응답** 우선, 실 호출은 클라우드/네트워크 가용 시에만.

## 4. 데이터 모델 (계약 유지)
`Case → AgentRun → Agent → Skill → Evidence → Approval → Audit` 를 테이블로:
- `cases`(id, status, riskScore, domain, …), `agent_runs`(id, caseId, steps, confidence, …), `approvals`(id, caseId, level L0–L4, actor, decision), `audit_events`(id, caseId, actor, action, evidenceId, hash, ts).

## 5. 심사 항목 매핑
- **4.2 실동작 MVP**: 정적 70% → 백엔드 실호출로 결정적 증명.
- **4.4 본선 고도화 기반**: 계약 함수의 서버 승격 자체가 기반.
- **5.4 발전경로**: "정적 PoC → 백엔드 승격 → 클라우드 운영" 단계가 실제 코드로 입증.
- **2.4 / 5.5**: PII 비반출 거버넌스(토큰화·반출 스캔·감사) 실구현으로 규제·리스크 대응 증명.

## 6. 범위 가드레일 (D-8)
- **In**: 계약 4함수 서버 이관, 핵심 REST 6종, SQLite, 프론트 API 클라이언트+폴백, Rule Engine, PII 거버넌스 구조, 로컬 구동.
