# F3 JBWC 기업여신·기술신용 — 오태림(JBG-205) 흐름

> **[대외비]** 인물·거래는 전부 합성(PII無). 본 문서는 [[_쇼케이스-빌드플랜]] F3 행의 상세 설계도다. 화면 자체의 와이어프레임/컴포넌트/데이터바인딩은 재정의하지 않고 [[01-메인-업무보드]]·[[02-케이스-상세]]·[[03-에이전트-실행뷰]]·[[04-승인-대기·결정]]·[[05-통계-추적-패널]]을 링크로 참조한다. 이 문서가 새로 정의하는 것은 **오태림 케이스 하나가 그 화면들을 어떤 순서·데이터·실동작 등급으로 관통하는가**다.

> **파일명 안내**: 이 파일은 과거 "F1 기업여신 딥 — 오태림" 설계도였으나, 2026-07-04 팀 결정에 따라 **F3 JBWC 흐름으로 재프레이밍**됐다(파일명 자체는 오케스트레이터가 `F3-JBWC-기업여신-딥-오태림.md`로 정리 예정). **딥 실LLM 히어로는 [[F1-CCL딥-김건우|F1 김건우(CCL-0001)]]**다 — PR#1이 이미 거기에 기배선돼 있다.

## 1. 한 줄 시나리오 + 페르소나

**오태림, 39세, 전북 군산시, 산업용 센서 제조 스타트업 대표.** JB우리캐피탈(JBWC) 콘솔 기업여신·기술신용 케이스, Risk 79, 상태 실행중. 대기업 납품계약은 확보했지만 양산 설비 리스 비용이 매출채권 회수보다 먼저 발생해 현금흐름이 얇고, 기술신용평가·특허·인증·납품계약서 근거가 흩어져 있어 심사가 지연될 위험이 있다. (원문: [[전라도-실감-페르소나-확장#JBG-205 — 오태림]])

**왜 이 케이스가 F3(실 LLM 2번째 후보 or 확장성 시연)인가**
- **딥 히어로는 아니지만 JBWC 콘솔의 실 LLM 확장성을 보여준다.** F1(김건우)이 전북은행 CCL 콘솔에서 딥 실LLM을 증명한다면, F3는 **같은 모델 선택 게이트웨이·같은 Zero-PII 원칙이 계열사(JBWC)를 바꿔도 그대로 작동한다**는 확장성을 보여준다 — "이식 가능한 아키텍처"라는 차별성 메시지에 정확히 들어맞는다.
- **텍스트 생성 가치가 여전히 선명하다.** 흩어진 기술신용 근거(특허·인증·납품계약)를 하나의 품의 초안으로 "패키징"하는 작업 자체가 텍스트 생성 문제이고, 규칙 기반 파이프라인으로는 흉내만 낼 수 있는 지점이다.
- **Zero-PII 티어 라우팅 시연에 최적이다.** 대표 개인보증·매출채권·계약 원문처럼 반출 위험이 높은 필드가 많은 도메인이라, "원문은 안 나가고 최소필드만 프런티어로 간다"는 서사가 억지스럽지 않고 케이스 성격과 맞아떨어진다.
- **검증 에이전트가 자기 오류를 잡는 장면을 자연스럽게 연출할 수 있다.** 계약일·금액·서류 누락 여부처럼 "숫자가 틀리면 바로 티가 나는" 필드가 많아, 준법 검토 에이전트의 재검이 시연에서 설득력을 갖는다.
- **빌드 순서상 2순위(Round 2, "왜 너희냐" 격파)**: F1(김건우)이 Round 1 코어 세트라면, F3는 Round 2 차별성 세트에서 "같은 파이프라인이 다른 계열사·다른 도메인에도 확장된다"를 증명하는 역할이다.

## 3. 실 LLM 포인트 상세

**어디서 실제로 도는가**: 스텝 4(품의/회신 초안 생성)가 유일한 실 LLM 지점이다. F1(김건우)과 **같은** 모델 선택 게이트웨이(`api-proxy.mjs` `handleLlm`, claude/codex/로컬)를 재사용한다 — F3는 새 게이트웨이를 만들지 않고, 계열사(JBWC)·도메인(기술신용)을 바꿔도 같은 파이프라인이 작동함을 보여주는 것이 목적이다.

**티어 라우팅**
- **분류·추출(로컬/저비용 티어)**: 계약서 필드 추출, 서류 체크리스트 매칭, 매출채권·리스 견적 일관성 판정 — 짧은 구조화 출력, 반복 호출량이 많은 구간.
- **판단·초안(프런티어 티어)**: 기술신용 근거 패키징 문장, 설비리스 조건안, RM 회신 초안 — 서술 품질이 승부처인 구간만 claude/codex 프런티어로 올린다.
- 원가 근거: 실측 1회 호출 in 49K(대부분 캐시)·out 3토큰·$0.115, break-even 대비 10배 여유([[Q13-토큰비용-유닛이코노믹스|Q13]]).

**Zero-PII 최소필드 페이로드 예시(합성)** — 프런티어로 나가는 실제 필드는 이 정도로 제한한다:

```json
{
  "caseId": "JBWC-CCL-0205",
  "riskLevel": "high",
  "affiliate": "jb-woori-capital",
  "sector": "산업용 센서 제조",
  "revenueTrend": "수주잔고 증가·입금 지연",
  "techCreditGrade": "T-3 (미확정)",
  "contractAmountBand": "3억~5억",
  "leaseAmountBand": "1억~2억",
  "guaranteeDependency": "high",
  "docsGap": ["기술신용평가서", "납품계약 부속서류"]
}
```
회사명·대표 성명·사업자등록번호·계약 원문·계좌번호는 이 페이로드에 없다 — `harnessGuardCheckPII()`가 반출 직전 이 계약을 강제한다.

**폴백 사다리 시연법** — [[Q14-오류로깅-폴백사다리|Q14]] 그대로 재현:
1. 정상 호출 — claude(또는 로컬모델)가 품의초안을 생성.
2. `timeoutMs`를 인위적으로 낮춰(예: `1`) 강제 실패시킨다 → 반대 엔진(codex 또는 다른 티어)으로 1회 폴백 재시도.
3. 재시도도 실패하도록 조작하면 `escalated: true, queue: "human"` 응답 → [[04-승인-대기·결정]] 승인 큐로 격상되어 사람 앞에 도착한다("AI가 조용히 사라지지 않는다"는 서사의 실물 증거).
4. 모든 시도(성공·실패 포함)가 `llm-runs.jsonl`에 한 줄씩 남는다(`runId, engine, tier, errorClass, retry, fallbackPath`).

**검증 에이전트가 자기 오류를 잡는 장면** — 품의초안이 계약일 또는 금액을 잘못 인용하면, 준법 검토 에이전트가 `PolicyCitationBox`/근거 대조에서 불일치를 발견해 해당 `AgentRunCard`를 `needsReview`로 되돌린다. 이 되돌림 자체가 [[03-에이전트-실행뷰]]의 `Shift+Enter`(반려/재검토, 사유 필수) 경로를 그대로 탄다 — 새 UI가 필요 없다.

## 4. 가드레일·승인·감사

- **PII 반출 차단**: `harnessGuardCheckPII()`(`harnessCore.js:74-110`, 정규식 기반 주민번호/전화번호/11자리 이상 숫자열 탐지) — `beforeCaseCreate`/`afterAgentRun`/`beforeCustomerMessage` 훅에서 강제. 위반 시 값이 아니라 "어떤 종류가 걸렸는지"만 표시(원문 미노출).
- **고위험 자동종결 차단**: `harnessGuardCheckAutoClose()` — riskLevel high/critical 케이스는 에이전트가 스스로 종결 상태로 못 넘어간다. 오태림 케이스(Risk 79 = high)는 이 가드에 항상 걸린다.
- **사람 승인(수정후승인)**: [[04-승인-대기·결정]] `S-04-detail` 3분할 — 설비리스 조건이 초안과 다르게 조정될 여지가 크므로, 이 케이스의 대표 데모 결정 경로는 "승인"이 아니라 **수정후승인**(`DraftEditor` 편집 → `DiffPreview` → 승인)으로 잡는다. L3이므로 Enter 단독 종료 금지(2단 확인 강제).
- **감사 봉인 — 정직 표기**: 감사 이벤트 자체(구조화 로그, `ccl_audit_logs`: actor/action/target/evidenceId)는 JB_project2에도 실재한다. **다만 해시체인 무결성 검증 UI(`prevHash→hash` 검증, `verifyAuditChain()`, `exportAuditJson()`)는 예선 MVP(`02_제품/app/`) 한정 구현**이며 JB_project2(제출 정본)에는 아직 이식되지 않았다 — F3 시연에서 감사 탭을 열 때 "해시체인 검증"까지 보여주려면 [[05-통계-추적-패널]]의 예선 MVP 화면을 별도로 인용해야 하고, JBWC 콘솔 화면만으로는 로그 존재까지만 보여줄 수 있다. 과장 금지.

## 7. 의존성

- **Gate 0 — 모델 선택 게이트웨이 JB_project2 이식**: F1(김건우)과 공유하는 선행 조건. 현재 `handleLlm`/`llm-runs.jsonl`/`liveLlmBlock`은 예선 `02_제품`에만 있다. F3의 실 LLM 지점(스텝 4)은 이 게이트웨이가 JB_project2(제출 정본)에 이식돼야 성립한다 — 미이식 상태에서는 F3 전체가 🟡로 강등된다.
- **F1과의 관계 — 별도 시드, 파이프라인은 공유**: PR#1(`feat/live-agent-ccl-financial`, River-181 fork, `LSB-afk/JB_project2#1`)은 **딥 히어로 [[F1-CCL딥-김건우|F1 김건우(CCL-0001)]]에 기배선**돼 있다. F3(오태림)는 김건우의 데이터를 갈아타는 것이 아니라, **같은 파이프라인 코드를 오태림 케이스로 별도 시드**해 확장성을 보여주는 것이 목적이다 — 파이프라인 자체를 다시 만들 필요는 없고, (a) PR#1 머지 후 (b) 오태림 케이스 데이터 시드 + JBWC 워크스페이스 라우팅 확인만 필요하다.
- **JBWC 워크스페이스 시드 정합**: 오태림은 JB우리캐피탈 콘솔(`jb-woori-capital` 워크스페이스, `jbWooriCapital*`/`wooricap*`) 소속이다. 기존 실 LLM 슬라이스가 은행 측 `ccl-financial`(기업여신) 롤에 붙어 있다면, JBWC 콘솔로 옮기거나 최소한 워크스페이스 간 케이스 라우팅을 확인해야 한다 — 계열사 정합 확인이 안 되면 스텝 2(케이스 생성)부터 어긋난다.
- **감사 해시체인 이식 여부**: §4에서 정직 표기한 대로, 해시체인 검증 UI가 JB_project2에 없다면 F3 데모의 감사 탭은 "로그 존재"까지만 보여줄 수 있고 "무결성 검증" 시연은 예선 MVP로 대체해야 한다 — 팀이 이식할지 정직 표기로 갈지 결정 필요.
