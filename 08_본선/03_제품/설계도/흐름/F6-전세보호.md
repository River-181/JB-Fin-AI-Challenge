# F6 전세보호 실 API — P6 전세(JPO) 흐름

> **[대외비]** 인물·거래는 전부 합성(PII無). 외부 공개 전 사람 최종승인 필요. 이 문서는 화면 자체를 다시 그리지 않고 [[01-메인-업무보드]]·[[02-케이스-상세]]·[[03-에이전트-실행뷰]]·[[04-승인-대기·결정]]의 S-ID를 그대로 참조한다.

> **인물 확정 상태 — 정직 표기**: P6 전세 페르소나는 **[확정필요]**다. JB_project2(`_vendor/JB_project2/app/jeonseProtection-db.js`)의 실 시드 `JEONSE-0001~0006`은 모두 **화곡동·신림동·개봉동·인천 미추홀·대전·부산** 케이스이며 **전북 소재 케이스가 하나도 없다**. 예선 `02_제품/app/app.js`의 데모 케이스(`seoul-jeonse-villa`)도 서울 강서구다. [[JB-콘솔-페르소나-로스터]]의 JBG-201(최은재, 전북 전주 완산구)이 페르소나 후보이지만, 실 데이터 시드에는 아직 매핑되지 않았다 — **전북 신규 케이스 캐스팅 또는 기존 JEONSE-0001의 전북 재배치가 팀 확정 사항**이다.

## 1. 한 줄 시나리오 + 페르소나

**[전세 페르소나, 전북 소재, 청년/신혼부부 전세 예정 — 확정필요].** 전북은행 JPO(전세보호) 콘솔 케이스. 신축 다세대·빌라 계약 전, 전세가율 과다·등기 미확인 위험이 겹치는 시점에 LocalGuard가 **방금 실제로 조회한 국토부 실거래가**를 근거로 케이스를 등록하고, 등기 확인·보증보험 가입 안내를 사람 승인 앞에 올려놓는다.

이 케이스가 F2(문서희, 기상특보)와 함께 "데이터 그라운딩 신뢰"를 보여주는 이유는 두 가지다. 첫째, F1·F3가 LLM 텍스트 생성 신뢰를 보여준다면 F6은 **위험 신호의 사실관계 자체가 정부 공식 실거래가**라서 환각 방어 서사가 직접 증명된다. 둘째, 전세사기는 본선 심사위원·관객 모두가 최근 뉴스로 체감하는 주제라 공감대 형성이 빠르다.

## 3. 실 API 포인트 상세

### 3-1. 데이터 소스와 검증 상태

| 소스 | 제공기관 | 확인 상태 | 근거 |
|---|---|---|---|
| **전세 실거래가**(`RTMSDataSvcRHTrade` 계열, `apis.data.go.kr`) | 국토교통부 | **확인됨** — 예선 `02_제품/scripts/api-proxy.mjs`의 `/jeonse?lawd=<법정동코드5>&ym=<YYYYMM>&kind=trade\|rent` 라우트가 이미 실 호출하는 코드에서 유일하게 검증된 실 fetch. `DATA_GO_KR_KEY` 1개로 인증 | `02_제품/scripts/api-proxy.mjs:33` |
| **jeonsePublicData.adapters.js**(`fetchJeonseMarketData`) | 국토교통부(MOLIT) 실거래 7종 + 서울 열린데이터 | **확인됨(JB_project2 정본)** — `?live=1` + 프록시 정상 시 `live_api`, 아니면 지역 스냅샷(`snapshot`), 둘 다 없으면 `fallback`(위험도를 낮게 확정하지 않고 최소 medium + "담당자 확인 필요") | `_vendor/JB_project2/docs/04-전세보호-역할-하네스.md` §4 |
| 공시가격 API | 국토교통부 | **미연동** — 매매 중앙값 기반 "추정" 공시가격만 표기(확정값 아님), 데모 대사에서 확정값처럼 말하지 않는다 | 상동 §4 |

**F6 핵심 경로는 이미 코드에 배선된 두 경로 중 하나를 그대로 쓴다.** 예선 `02_제품`의 `/jeonse` 프록시 라우트(간단·1개 키), 또는 JB_project2 정본의 `jeonsePublicData.adapters.js`(라이브/스냅샷/폴백 3단 sourceMode, 더 완성된 정본 경로) — 어느 쪽을 본선 데모 경로로 쓸지는 Gate 0 작업에서 확정한다.

### 3-2. 조회 흐름

```
브라우저(?live=1) → 로컬 proxy(api-proxy.mjs, :8020, /jeonse) → apis.data.go.kr (국토부 실거래가)
                                                                ↳ 키 없음/장애 시 → 사전 캡처 고정 실거래가 JSON 폴백(fallback)
```

- F2(기상특보)와 **동일한 `api-proxy.mjs`, 동일한 `DATA_GO_KR_KEY` 환경변수, 동일한 CORS 헤더 셋업**을 재사용한다 — 신규 프록시 서비스 도입이 아니다.
- data.go.kr 오픈API는 브라우저 직접 fetch 시 CORS가 대부분 막혀 있어 proxy 경유가 필수다.
- `?live=1`이 없으면 사전 캡처한 고정 실거래가 응답(거래시점·거래가·법정동)을 그대로 렌더 — 오프라인·키 없음 상황에서도 데모가 끊기지 않는다.

### 3-3. 케이스 위험신호·EvidencePack 바인딩

1. 프록시가 파싱한 실거래가 필드(거래금액, 거래일, 지역코드, 건물유형)를 그대로 `Signal` 레코드에 매핑한다 — 별도 LLM 가공 없이 **원문 그대로**가 근거의 핵심.
2. `Signal.source`에 `"국토부 실거래가(data.go.kr)"` 문자열과 조회 타임스탬프를 고정 표기 — 심사위원이 "이건 지어낸 숫자가 아니라 API 응답"임을 즉시 알 수 있게 한다.
3. `EvidencePack`은 이 Signal을 근거탭에 그대로 인용하고, 준법 검토 에이전트가 "전세가율 계산이 실거래가 원문과 일치하는가"만 재확인 — LLM이 실거래가 숫자 자체를 재작성하지 않는다(환각 방지의 핵심 설계).

### 3-4. 환각 방어 서사

F2(문서희, 기상특보)가 이미 증명한 패턴을 재사용한다: **LLM은 판단·초안 문구만 생성하고, 위험신호의 사실관계(실거래가·거래시점·지역)는 100% 외부 API 원문에서 가져온다.** 그래서 "AI가 전세사기를 예측했다"가 아니라 "AI가 정부 공식 실거래가를 놓치지 않고 즉시 케이스에 연결했다"는 정직한 주장만 한다.

## 4. PII·승인·감사

- 실거래가 데이터는 **지역·거래 단위 공개 데이터**이며 고객 개인정보가 아니다 — 조회·저장·화면 표시에 PII 비반출 원칙이 걸리지 않는다.
- 전세 페르소나 개인 식별정보(성명·계좌·연락처·계약 원문)는 어떤 단계에서도 외부 API·외부 LLM으로 나가지 않는다 — 실거래가 조회는 지역코드만 사용, 에이전트 판단 프롬프트는 Zero-PII 필드(케이스ID·전세가율·계약 만기일)만 사용.
- 등기 확인·보증보험 가입 안내, 법률 판단은 **사람 최종승인**([[04-승인-대기·결정]] Decision 패널)을 거쳐야 실행되며, AI는 후보 초안·확인 요청까지만 만든다(JB_project2 원칙: "전세사기 여부·법률·보증·피해자 결정은 항상 담당자가 판단").
- 감사 봉인은 실거래가 API 호출 시각·응답 요약·승인자·승인시각을 체인 기록에 남긴다 — 단, base app만 해시체인이 실재하므로([[_설계도-INDEX]] 실측 정합 주의) F6 데모의 감사 단계는 base app 경로로만 시연한다.

## 5. 데모 스크립트(구두, 30~60초)

1. (0~10초) "전북 [지역] 전세 예정 고객 케이스입니다. 화면에 방금 뜬 이 실거래가, 지금 이 순간 국토부 공공 API에서 실제로 가져온 실데이터입니다." — `?live=1` 토글 후 케이스 신규 등록 카드가 뜨는 순간을 보여준다.
2. (10~25초) 케이스 상세 위험신호 탭을 열어 "거래시점·거래가·지역"이 그대로 찍힌 화면을 가리키며 "AI가 지어낸 숫자가 아니라 정부 실거래가 원문입니다"라고 말한다.
3. (25~40초) 에이전트 실행뷰로 이동 — 전세위험 관리 에이전트 → 등기 권리 분석 에이전트 카드가 슬라이드되며 등기 확인·보증보험 안내 후보가 만들어지는 과정을 짧게 보여준다.
4. (40~55초) 승인대기함으로 이동 — "법률 판단과 보증 확정은 AI가 실행하지 않습니다. 사람이 최종 승인합니다"라며 3분할 결정 화면에서 승인 버튼을 누른다.
5. (55~60초) 감사기록 화면으로 컷 — "이 판단의 근거, 실거래가 조회 시각까지 전부 감사 로그에 봉인됩니다."

## 6. 수용 기준

- [ ] `?live=1` 지정 시 전북 대상 국토부 실거래가 API가 실제로 호출되고, 응답이 화면에 반영된다.
- [ ] `?live=1` 없이도(또는 API 키 미설정 시) 사전 캡처 고정 실거래가 데이터로 동일 흐름이 끊김 없이 재생된다.
- [ ] 위험신호 탭에 `Signal.source`로 API 출처명·조회 타임스탬프가 표기된다.
- [ ] 등기 확인 요청/보증보험 안내는 승인대기함을 거치지 않고는 "실행됨" 상태가 될 수 없다.
- [ ] 감사기록에 이 케이스의 실거래가 조회~승인 체인이 남는다(base app 경로 기준).
- [ ] 전세 페르소나 개인정보(성명·계좌·연락처 등)가 프롬프트 로그·네트워크 요청 어디에도 원문으로 나타나지 않는다.
- [ ] 데모 스크립트 30~60초를 실제로 리허설해 끊김 없이 재생된다.
- [ ] **[확정필요] 전북 전세 페르소나·케이스 ID가 팀 확정되어 위 항목을 실제 데이터로 재생 가능하다.**

## 7. 의존성·리스크

- **P6 인물·케이스 확정**: 전북 소재 신규 케이스 캐스팅 또는 JEONSE-0001의 전북 재배치가 우선 확정돼야 한다. 확정 전까지는 로스터 JBG-201(최은재, 전북 완산구) 바이오를 임시 참조로 쓰되, 화면에는 [확정필요] 표기를 유지한다.
- **API 실시간 도달성**: 조회 시점에 대상 법정동 실거래가 표본이 적을 수 있다(전세 실거래는 월별 변동이 큼). 대응: 사전 캡처한 "전세가율 위험 발견" 응답을 오프라인 폴백으로 반드시 준비.
- **CORS**: `apis.data.go.kr`는 브라우저 직접 호출이 막힐 가능성이 높다 — 반드시 `api-proxy.mjs` 경유(`/jeonse` 라우트, 이미 구현됨).
- **경로 선택**: 예선 `02_제품`의 단순 `/jeonse` 프록시와 JB_project2 정본의 `jeonsePublicData.adapters.js`(live/snapshot/fallback 3단) 중 본선 데모 경로를 Gate 0에서 확정 — 후자가 더 완성도 높은 정본 경로.
- **demo:proxy 설정**: `npm run demo:proxy`가 이미 `DATA_GO_KR_KEY` 환경변수로 기동하는 구조라 F6 전용 추가 설정은 지역코드·케이스 시드 외에 없다.
- **오프라인 폴백**: 발표 현장 네트워크 불안정 대비, 전세 실거래가 API 응답 샘플(고위험 케이스 1건 + 정상 케이스 1건)을 사전 캡처해 로컬 고정 JSON으로 보관.
