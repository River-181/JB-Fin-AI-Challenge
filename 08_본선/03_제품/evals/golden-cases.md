# Golden Cases — 반드시 성공해야 하는 대표 케이스(= 데모 케이스)

> 목적: 심사장에서 "좋은 답을 했다"가 아니라 **"사전 정의된 케이스가 `입력→근거→승인게이트→상태변화→감사로그` 프레임을 재현 가능하게 완주했다"**를 보인다(D13 [E2]). 이 문서의 케이스는 곧 **데모 케이스**이며, `evals/eval-plan.md`의 §4 테스트데이터·§6 Pass/Fail과 1:1로 물린다.
>
> **SSOT**: 케이스 데이터·에이전트·승인 라우팅은 제품 코드 `_vendor/JB_project2/app/`(역할축 콘솔)에 실재한다 — CaseID·AgentID·시그널은 지어내지 않고 코드에서 인용했다. 수치·"왜?"는 리서치 요약(D1a·D1b·D13)에서만 선별 주입.
>
> **근거등급(E0~E5)**: E0 가정 · E1 문헌시사 · E2 문헌다중근거 · E3 내부 설계·코드 근거 · E4 데모 실증 가능 · E5 운영실측. **목표 E2+, 데모 대상은 E4.**
> **신뢰마커**: **[확정]** 코드/canon 직접 · **[조건부]** 설계·팀결정 대기 · **[미검증]** 근거약함 · **[Open Question]** 팀 결정.

## 0. 공통 규약 (모든 골든 케이스에 적용)

| 규약 | 내용 | 근거 |
|---|---|---|
| 공통 프레임 | 모든 케이스는 `입력 → 근거(evidence) → 승인 게이트 → 상태 변화 → 감사로그` 5단을 완주한다. | D13 [E2] |
| PII 비반출 | 입력·저장·출력 전 구간에서 원본 PII를 외부 LLM에 반출하지 않는다. 케이스는 익명 참조ID(`BIZ-REF-*`/`CUST-*`)와 **구간(band) 지표**만 다룬다. | canon, 코드 주석 `cclConsole.data.js:1` [E3] |
| 승인 불변식 | 승인 토큰 없이는 고객 알림·약정·자금 release·차단 실행이 **절대** 일어나지 않는다(모델 성능과 분리한 시스템 불변식). | D13 [E2], eval-plan §3-B [E3] |
| Fail-closed | 신뢰도 낮음/근거 미달/데이터 시차 입력은 자동 확정하지 않고 **수기검토로 라우팅**한다. | D13·D14 [E2] |
| Attribution | 중요 claim에 `source_id, doc_snippet, retrieval_time, hash, policy_rule_id`를 붙인다. 근거 없는 claim은 차단. | D13 [E2] |
| 감사이벤트 | 케이스당 AuditEvent **≥4** 발생, `previousHash` 체인 무결(append-only). | eval-plan PF-1·PF-4 [E4] |
| 역할 축 | 승인 종착지는 두 갈래 — **RM(1차 승인)** / **준법·보안(고위험 2차 게이트, L3~L4)**. 전세·피싱은 별도 직군이 아니라 같은 2역할이 처리하는 케이스타입. | 키스톤-역할축-검증 [E3] |

> ⚠️ 절대 KPI는 **보장문이 아니라 범위+분모**로 진술한다: "승인 우회 0" → `critical flow e2e N건에서 승인 우회 0 관측`(D13 [E2], eval-plan P1).

## 3. Pass/Fail 요약 (eval-plan §6과 1:1)

| PF | 대상 | Pass 기준(범위+분모) |
