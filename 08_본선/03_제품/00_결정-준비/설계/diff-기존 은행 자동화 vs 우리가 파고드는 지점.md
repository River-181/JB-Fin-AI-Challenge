# 기존 은행 자동화 vs 우리가 파고드는 지점

# JB LocalGuard OS 차별 설계: “은행이 이미 다 하는 것”과 우리가 파고드는 지점

## 0. 한 줄 결론

기존 은행 자동화는 **개별 업무를 빠르게 처리하는 시스템**에 가깝다. JB LocalGuard OS는 **케이스 단위로 AgentRun, Evidence, Approval, Audit를 묶어 담당자의 판단·승인·감사 책임을 보조하는 운영 콘솔**이다. 자동심사, RPA, 문서작성 AI와 경쟁하는 것이 아니라, 그 결과물을 **케이스 메모리·근거·사람 승인·PII 비반출·감사원장** 안에 넣어 금융권이 실제로 구매 가능한 형태로 통제한다.

---

## 1. 설계 전제

SSOT 커널은 다음 흐름으로 고정한다.

`Case → AgentRun → Agent → Skill → Evidence → Approval → Audit`

콘솔 축은 확정 유력 A안 기준이다.

| 축 | 내용 |
|---|---|
| 계열사 | 전북은행, JB우리캐피탈 |
| 담당직군 | RM, 여신심사, 사후관리, 준법, AML |
| 도메인 | 여신, 전세, 피싱, 사후관리 |
| 제약 | 전세는 은행 전용, 캐피탈에는 부재 |
| 확장 방식 | 계열사별 도메인이 다른 모듈식 구조 |

현행 canon 기준은 **운영 에이전트 14 + 사람 승인자 2**이며, 히어로 케이스는 **전주 카페 여신심사 `JBG-104`, `risk88`**로 둔다. `(canon)`

---

## 2. 우리가 자동화하려는 것이 아니다

심사위원 질문은 이렇게 올 가능성이 높다.

> “대출 심사 시스템, 스코어링, RPA, 문서 자동작성 AI는 은행이 이미 하는 것 아닌가?”

맞다. 일부는 이미 하고 있다. D16 리서치에서도 은행권 자동화 사례가 확인된다.

| 기존 자동화 | 이미 확인된 내용 | 근거 |
|---|---|---|
| 기업여신 자동심사 | 우리은행은 기업여신 자동심사시스템 도입 사례가 있음 | `D16-geminipro`, 기업여신 자동심사 사례 |
| 신용평가/심사의견 AI | 하나은행은 심사의견 작성 시간을 `30분 → 10초`로 단축, 연 `27,000시간` 절감 기대 | `D16-geminiflash`, `D16-geminipro` |
| RPA | KB국민은행은 240개 업무에 RPA 적용, 직원 1인당 연 `170시간` 절감 언급 | `D16-geminiflash` |
| 여신 서류 반복 처리 | IBK RPA는 영업점 여신 서류 관련 업무를 일평균 `3,000건` 처리했다는 보도 근거 | `D16-gpt55high` |
| OCR/Non-typing | 기업여신 서류 입력·대조 시간을 크게 줄이는 벤더 사례 존재 | `D16-geminiflash`, `D16-geminipro` |

따라서 우리 메시지는 “은행이 못 하는 자동화를 우리가 처음 한다”가 아니다.  
정확한 메시지는 다음이다.

> 기존 자동화는 **업무 단위 효율화**이고, JB LocalGuard OS는 **케이스 단위 운영통제 레이어**다.

## 3. 차별 대조표

| 구분 | 기존 은행 자동화/RPA/AI | JB LocalGuard OS가 파고드는 지점 |
|---|---|---|
| 기본 단위 | 업무, 화면, 프로세스, 문서 | `Case` 단위 운영계약 |
| 자동화 목표 | 입력 단축, 심사의견 초안, 반복작업 제거 | 담당자가 케이스를 끝까지 통제하도록 보조 |
| 기억 구조 | 시스템별 이력, CRM, 단말 로그로 분산 | 케이스 통합 메모리 |
| 판단 근거 | 보고서나 입력 필드에 흩어짐 | `Evidence`로 출처·근거·판단 연결 |
| 사람 승인 | 결재 시스템에 존재하나 AI 출력과 분리될 수 있음 | `Approval L0~L4`를 AgentRun과 직접 연결 |
| 감사 대응 | 사후 로그 확인 중심 | `Audit` 원장에 실행·근거·승인 경로를 누적 |
| PII 처리 | 시스템별 보안정책에 의존 | 등급제·토큰화·모델라우팅·반출스캔+감사원장 |
| 외부 모델 사용 | 보안상 제한 또는 별도 검토 필요 | 원본 PII 외부 반출 금지, 모델 라우팅으로 통제 |
| 멀티에이전트 | 단일 업무 AI 또는 RPA 봇 중심 | 전문 Agent들이 Case 안에서 협업 |
| 담당자 경험 | 여러 시스템을 오가며 확인 | 직군별 콘솔에서 케이스·근거·승인·다음 액션 통합 |
| 구매 방어 논리 | “시간 절감” | “시간 절감 + 내부통제 + 감사 추적 + PII 비반출” |

## 4. 핵심 차별점 6개

### 4.1 케이스 통합 메모리

기존 자동화는 품의서 작성, 서류 입력, RPA 처리처럼 업무 조각별로 성과가 난다. 하지만 실제 RM 업무는 상담, 서류징구, 신용평가, 심사부 handoff, 사후관리, EWS, 고객 커뮤니케이션이 끊어지지 않는다. `D16-gpt55high`는 RM 업무 범위를 상담·서류수집·신용평가 입력·대출 의사결정 연결·사후관리·리스크 대응까지로 정리한다.

JB LocalGuard OS는 이 흐름을 `Case`에 묶는다. `JBG-104` 같은 케이스 안에 AgentRun, Evidence, Approval, Audit가 계속 붙는다. 담당자는 “어떤 AI가 어떤 근거로 어떤 제안을 했고, 누가 어디까지 승인했는지”를 한 화면에서 따라간다.

### 4.2 Evidence 추적

은행 자동화가 이미 심사의견 초안을 만들 수 있어도, 심사위원이 볼 핵심은 “그 말의 근거가 어디냐”다. D16은 handoff와 재작업의 원인을 문서 누락, 데이터 정합성 오류, 근거 불충분, 사후관리 기록 미흡으로 본다. `D16-gpt55high`는 금감원 제재·개선 요구에서 **진위확인, 공적장부 확인, 제3자 조사, 사후관리 근거 보관**이 중요하다고 정리한다. 신용정보법 §36조의2(자동화평가 결과에 대한 설명요구권), PIPA §37조의2도 Evidence 설계의 직접 근거다. (`근거팩/적법성-근거팩.md`, `D15-결과`)

JB LocalGuard OS는 AI 결과물을 바로 업무 처리하지 않는다. 먼저 `Evidence`를 붙인다.

예시:

| AgentRun 산출물 | Evidence로 붙어야 하는 것 |
|---|---|
| 여신심사 요약 | 재무제표, 세금계산서, 거래내역, 기존 여신이력 |
| 전세 피해 검토 | 등기부등본, 임대차계약서, 경공매 진행자료 |
| 피싱/FDS 대응 | 차단 로그, IP/기기/거래 패턴, 고객 확인 기록 |
| 사후관리 | 자금사용내역, 세금계산서, 입금내역, 점검 체크리스트 |

근거가 없는 판단은 “AI 의견”이 아니라 “검토 필요”로 남긴다.

### 4.3 사람 승인 게이트

금융위 AI 가이드라인은 여신심사에서 승인책임자와 사람에 의한 통제가능성을 요구한다는 근거가 있다. `D16-gpt55high`는 생성형 AI 정책자료도 최종 의사결정과 책임은 임직원에게 둔다고 정리한다. 신용정보법 §36조의2(자동화평가 결과에 대한 설명요구권), PIPA §37조의2도 승인우선·설명가능성 설계의 직접 근거다. (`근거팩/적법성-근거팩.md`, `D15-결과`)

따라서 JB LocalGuard OS의 승인 구조는 자동승인이 아니라 승인우선이다.

| 레벨 | 의미 | 승인자 |
|---|---|---|
| L0 | 조회·요약·내부 초안 | 시스템 기록 |
| L1 | 담당자 업무 보조 산출물 채택 | RM |
| L2 | 고객 안내, 케이스 진행 액션 | RM |
| L3 | 리스크·준법 영향 있는 액션 | RM + 준법 |
| L4 | 대외 반출, 고위험 처리, 예외 승인 | RM + 준법 2역할 승인 |

레벨별 상세 기준은 내부 규정 매핑이 필요하다. `(→담당자 확인)`

### 4.4 PII 4중 방어

PII는 제품의 핵심 방어선이다.

| 방어선 | 역할 |
|---|---|
| 등급제 | 데이터와 Evidence를 민감도별로 분류 |
| 토큰화 | AgentRun에는 원문 PII 대신 토큰 참조 전달 |
| 모델라우팅 | Claude, Codex, 로컬 모델 중 데이터 등급에 맞게 분기 |
| 반출스캔+감사원장 | 외부 전송 전 PII 포함 여부 검사, 결과를 Audit에 기록 |

원칙은 명확하다.

> 원본 PII는 외부로 반출하지 않는다.

외부 모델은 충분한 결과물을 낼 수 있지만, 원본 PII가 필요한 업무는 로컬 또는 내부망 경로로 라우팅한다. `(canon/커널)`

### 4.5 멀티에이전트 협업

사용자 비전의 핵심은 각 에이전트가 “뾰족한 전문가”로 회사 서버에 영속한다는 점이다.

각 Agent는 다음을 가진다.

| 구성요소 | 의미 |
|---|---|
| 전문 지식분야 | 여신, 전세, 피싱, 사후관리, AML 등 |
| 기업내규·규정 | 계열사·직군·업무별 규정 |
| 케이스 메모리 | 과거 Case와 AgentRun 이력 |
| 외부툴/Open API | 필요한 조회·검증 도구 |
| Skill | 문서대조, 요약, 품의서 초안, 근거 패키징 등 |

중요한 점은 “에이전트가 결론을 내린다”가 아니다.  
에이전트는 담당자가 결론을 낼 수 있도록 **결과물, 워크플로, 가이드, 제안**을 만든다.

### 4.6 직군별 콘솔

키스톤 A안은 도메인 중심이 아니라 `계열사 × 담당직군` 중심이다. 도메인은 직군이 처리하는 케이스다.

| 직군 | 대표 케이스 | 콘솔이 보여줘야 하는 것 |
|---|---|---|
| RM | 신규여신, 만기연장, 고객 안내 | 고객·서류·품의 초안·승인 대기 |
| 여신심사 | 심사의견, 조건변경, 예외검토 | Evidence, 리스크요인, 반려 사유 |
| 사후관리 | 자금용도 점검, 연체, EWS | 점검 체크리스트, 조치 이력 |
| 준법 | 고위험 승인, 반출 검토 | PII 스캔, 규정 위반 가능성, L3/L4 승인 |
| AML | 의심거래, 고객확인 | 이상징후, 고객확인 근거, 보고 후보 |

전세는 은행 전용 케이스로 두고, JB우리캐피탈 콘솔에는 동일 도메인을 억지로 넣지 않는다. `(canon/커널)`

## 5. “이미 자동화된 것”과 “우리가 필요한 이유”

### 5.1 자동심사 시스템이 있어도 필요한 이유

자동심사는 대출 가능성, 신용평가, 심사 효율을 높인다. 하지만 케이스의 전체 맥락과 사람이 승인해야 할 근거 패킷을 항상 만들어주지는 않는다. D16은 기업대출 프로세스에 상담, 신용평가, 대출 의사결정, 실행이 있고 서류 종류가 많다는 점을 근거로 제시한다. `D16-gpt55high`

JB LocalGuard OS는 자동심사 결과를 대체하지 않는다. 자동심사 결과를 `Evidence`와 `Approval` 안에 편입한다.

### 5.2 RPA가 있어도 필요한 이유

RPA는 반복 화면 작업에 강하다. D16의 KB, IBK, NH농협 사례도 RPA가 큰 시간 절감 효과를 냈음을 보여준다. 하지만 RPA는 “왜 이 케이스가 이렇게 판단됐는지”를 설명하는 운영계약이 아니다.

JB LocalGuard OS는 RPA 결과도 AgentRun 또는 Evidence로 흡수한다. RPA는 실행 수단이고, OS는 케이스 통제 레이어다.

### 5.3 문서 자동작성 AI가 있어도 필요한 이유

하나은행 사례처럼 심사의견 초안은 이미 AI가 매우 빠르게 만들 수 있다. 그러나 금융권 구매자가 우려하는 것은 초안 속도가 아니라 다음이다.

| 우려 | LocalGuard OS 대응 |
|---|---|
| AI가 틀린 근거로 썼는가 | Evidence 링크 |
| 누가 승인했는가 | Approval L0~L4 |
| PII가 외부로 나갔는가 | 모델라우팅, 반출스캔, Audit |
| 사후감사에서 설명 가능한가 | Audit 원장 |
| 여러 부서가 같은 맥락을 보는가 | Case 통합 메모리 |

즉, 문서 자동작성 AI가 있을수록 LocalGuard OS의 필요성은 커진다. 생성물이 많아질수록 통제·승인·감사가 더 중요해지기 때문이다.

## 6. 히어로 케이스: 전주 카페 여신심사 `JBG-104`

히어로는 `전주 카페 여신심사`, 위험도 `risk88`로 둔다. `(canon)`

### 케이스 흐름

1. RM이 `JBG-104` Case를 연다.
2. 여신 관련 AgentRun이 고객 자료, 재무자료, 거래이력, 기존 상담 메모를 정리한다.
3. Evidence가 재무제표, 세금계산서, 매출 증빙, 기존 여신 이력과 연결된다.
4. 여신심사 Agent가 심사의견 초안을 만든다.
5. 사후관리/준법 Agent가 자금용도, 고위험 문구, PII 반출 가능성을 점검한다.
6. RM이 L1/L2 승인으로 초안을 채택하거나 수정한다.
7. `risk88`이므로 준법 검토가 필요한 경우 L3/L4로 escalation한다. `(레벨 기준은 담당자 확인)`
8. 모든 AgentRun, Evidence, Approval이 Audit 원장에 남는다.

### 심사위원에게 보여줄 메시지

> “이 화면은 대출을 자동으로 승인하는 화면이 아닙니다. RM이 판단해야 할 케이스를 AI 전문가들이 근거와 함께 준비하고, RM·준법 승인 없이는 대외 액션이 나가지 않도록 막는 콘솔입니다.”

## 7. 근거 기반 ROI 메시지

D16 리서치에서 방어 가능한 근거는 다음이다.

| 항목 | 방어 가능한 주장 | 근거 |
|---|---|---|

`연간 절감 가능 원가 = RM 수 × 케이스/인 × 케이스당 절감시간 × 시간당 원가 × 실현율`

단, `케이스/인`, `케이스당 절감시간`, `실현율`은 공개자료가 약하므로 `[추정]`으로 둔다. `D16-gpt55high`

## 9. 심사 방어 문장

> 은행은 이미 자동심사, RPA, OCR, 문서작성 AI를 도입하고 있습니다. 그래서 JB LocalGuard OS는 “또 하나의 자동화 도구”가 아니라, 그 자동화 결과를 케이스 단위로 묶어 **근거 추적, 사람 승인, PII 비반출, 감사원장**까지 완성하는 운영 콘솔입니다. 금융권에서 중요한 것은 AI가 얼마나 빨리 쓰느냐가 아니라, 누가 어떤 근거로 승인했고 원본 PII가 어디까지 갔으며 사후감사에서 설명 가능한가입니다. 우리가 파고드는 지점은 바로 그 통제 레이어입니다.
