### Q1. 히어로가 둘로 갈렸다 — CCL-0001(전주 카페)을 유지할지, 오태림(F1)으로 완전히 넘어갈지, 병행할지?

- **근거**: decision-log는 "히어로ID CCL-0001 단일화… 완료(커밋 f75f20f)"라고 8개 파일 정정 완료를 기록했다. 그런데 시간상 그 이후(혹은 같은 세션 내) 작성된 [[_쇼케이스-빌드플랜]]은 "구 히어로 `JBG-104 전주 중앙로 카페(김건우)`는 **폐기**. 최신 히어로 = Figma… **5페르소나 쇼케이스**"라고 선언하며 F1(오태림)을 실 LLM의 유일한 지점으로 재지정했다. 둘 다 "이게 최종"이라는 어조지만 서로를 참조하지 않는다.
- **선택지**:
  - **(A) CCL-0001 유지, 5페르소나는 보조 레이어** — 이미 8개 핵심 스펙 문서(05_domain-model·06_prd·07_architecture·08_feature-spec·09_flow 등)가 CCL-0001로 정렬되어 있어 재작업이 가장 적다. 단점: 쇼케이스-빌드플랜·F1·F2 문서(가장 최신 설계도)가 이미 오태림·문서희 중심으로 전부 다시 쓰여 있어, 이 문서들을 다시 CCL-0001 세계관으로 되돌려야 한다.
  - **(B) 오태림(F1)으로 완전 전환, CCL-0001 폐기** — 쇼케이스-빌드플랜의 명시적 의도. 단점: 표 10의 15개+ 파일을 전부 재정렬해야 하고, PR#1(`feat/live-agent-ccl-financial`)이 이미 CCL-0001 데이터에 배선되어 있어 재시딩 작업이 필요하다(F1 §7 스스로 인정).
  - **(C) 병행** — CCL-0001은 "이미 실 LLM이 도는 검증된 파이프라인의 증거"로, 오태림은 "발표용 5페르소나 서사"로 역할을 나눠 둘 다 문서에 남긴다. 단점: 심사위원 앞에서 "히어로가 왜 두 개냐"는 질문에 대한 답을 준비해야 하고, 오히려 정합감사가 매번 반복해서 걸리는 문서 부채가 계속 남는다.
- **권고 판단 축**: 남은 개발 시간(제출 7/5 10:30)과 PR#1 재시딩 공수를 팀이 얼마나 감당할 수 있는지가 핵심 변수다.

### Q2. 오태림은 CCL 콘솔(전북은행 역할)에 넣을지, JBWC 콘솔(그의 실제 소속)로 옮길지?

- **근거**: `구현현황-JB_project2.md` §1에 따르면 CCL 콘솔은 "역할=기업여신 담당자"로 진입하며 계열사 축은 전체/전북은행일 때 열린다. JB우리캐피탈은 **역할 전환이 아니라 계열사 전환**에 걸리는 별도 JBWC 콘솔(`jbWooriCapital*/wooricap*`, 독립 DB `jbwc-ops-db-v3`, 13개 전용 에이전트)이다. 그런데 [[_쇼케이스-빌드플랜]] §1 표는 오태림(JB우리캐피탈 소속)의 "콘솔(역할축)"을 "CCL 기업여신"이라고 적었다. F1 문서 본문(§7 의존성)이 스스로 이 모순을 짚는다: "오태림은 JB우리캐피탈 콘솔… 소속이다. 기존 실 LLM 슬라이스가 은행 측 ccl-financial(기업여신) 롤에 붙어 있다면, JBWC 콘솔로 옮기거나 최소한 워크스페이스 간 케이스 라우팅을 확인해야 한다."
- **선택지**:
  - **(A) 오태림을 JBWC 콘솔로 이전** — 아키텍처상 정합하다. 단점: PR#1의 실 LLM 파이프라인이 CCL 코드(`cclConsole.*`)에 배선돼 있어, JBWC로 옮기려면 파이프라인을 다시 배선하거나 최소 워크스페이스 라우팅을 재검증해야 한다(F1 §7).
  - **(B) PR#1(CCL/전북은행) 파이프라인을 그대로 쓰고, 오태림 계열사 라벨을 전북은행으로 바꿔치기** — 개발량 최소. 단점: 페르소나 로스터·전라도 확장 문서가 이미 "JB우리캐피탈 기업여신·기술신용"으로 오태림을 여러 곳에 못박아 놓았고(캐논 라벨), 이 자체를 재정의해야 하는 연쇄 수정이 발생.
  - **(C) F1의 실 LLM 슬라이스는 CCL/전북은행에 남기되, 페르소나는 오태림이 아니라 기존 9건 로스터 중 전북은행·CCL에 이미 맞는 인물(예: JBG-105 정하린)로 교체** — 아키텍처 정합·개발량 최소를 동시에 만족. 단점: 쇼케이스-빌드플랜이 공들여 설계한 "오태림=기술신용 근거 패키징" 서사(LLM 텍스트생성 가치가 가장 선명한 이유)를 포기하게 된다.
- **시간이 촉박하므로 이 질문은 Q1과 함께 오늘 중 결정이 필요하다.**

### Q3. "2계열사만 시연" 결정과 5페르소나 쇼케이스의 광주은행 비중을 어떻게 맞출지?

- **근거**: 7/3 스프린트회의("전북은행이랑 우리캐피탈. (긍정), 두 개.")와 7/4 전략회의2("이거 두 개만 이렇게 만듭니다")가 **명시적으로 2계열사만** 확정했다. 그런데 [[_쇼케이스-빌드플랜]] §1의 5페르소나 중 문서희(F2)·송민재(F3)·윤복례(F4) **3명**의 계열사가 광주은행이다. 빌드플랜 스스로 "Figma 카드는 송민재·윤복례를 전북은행으로 표기(캐논=광주은행)… 문서희(P2)는 Figma도 광주은행 유지(3번째 계열사 카메오)"라고 인정한다.
- **선택지**:
  - **(A) 문서희·송민재·윤복례를 전북은행/JB우리캐피탈로 전원 재배치** — "2계열사" 발표 문구와 완전히 정합. 단점: F2·F3의 서사(완도 전복양식·해남 시설원예 — 기상/해양 특보와 강하게 결합된 전남 지역성)가 재배치 시 개연성을 잃는다. 기상청 특보 API 자체는 지역 무관하게 재사용 가능하지만 "전라도 실감" 컨셉의 설득력이 약해진다.
  - **(B) 발표 문구를 "핵심 2계열사 + 지역성 보강 카메오 1곳(광주은행)"으로 완화** — 페르소나 서사를 안 건드림. 단점: "2계열사만 시연"이라고 이미 여러 문서(HANDOFF-결정세션·키스톤-확정 등)에 박제된 문구와 발표 슬라이드를 다시 조율해야 하고, 심사위원이 "그럼 3계열사 아니냐"고 물었을 때 대응 논리가 필요.
  - **(C) Round 1(F1·F2·F6)만 최종 데모로 확정한다면 F2(문서희, 광주은행)가 이미 코어에 포함돼 있어 (A)/(B) 중 하나는 반드시 필요 — "만약 시간이 없어 Round1까지만 간다"는 폴백 시나리오에서도 이 질문은 회피 불가.**

### Q4. 전세(JPO) 도메인을 이번 데모/발표에 넣을지 뺄지?

- **근거**: [[_쇼케이스-빌드플랜]]은 9-flow·5-persona 어디에도 전세를 넣지 않고 "실 API 그라운딩은 기상/해양 공공API로 이동"이라 명시했다. 반면 7/3 스프린트회의에서 이승보가 "이 RM이랑… 전세, 전세사기, 그리고 보이스피싱 이 세 가지만 얘기를 해주면 어떨까?"라고 제안했고 김민주가 "좋아요, 그게 더 나을 것 같아요"라고 동의해 **전세사기가 3대 핵심 도메인 중 하나로 확정**된 바 있다(decision-log 2026-07-03 스프린트 회의 #3 항목 "D2 = RM(여신)·전세사기·보이스피싱 3핵심"). `06-정보체계-뷰-데이터바인딩-스펙.md`와 `_화면목록-뷰맵.md`도 여전히 전세(JPO, 29뷰로 콘솔 중 최다)를 상세히 다룬다.
- **선택지**:
  - **(A) 전세(JPO)를 5페르소나 쇼케이스에 F10으로 추가** — 7/3 결정과 정합, 이미 코드(JPO 콘솔 29뷰, `?live=1` 실거래가 API)가 가장 잘 구축된 도메인이라 개발 부담은 오히려 적을 수 있음. 단점: 9-flow 체계를 다시 늘려야 하고 발표 시간(예선 5분) 안에 더 욱여넣어야 함.
  - **(B) 전세를 공식적으로 데모 범위에서 제외, 부록/스트레치로만 언급** — 쇼케이스-빌드플랜의 현재 의도와 일치. 단점: 7/3 스프린트 회의 결정("전세사기는 꼭 가져가면 좋을 도메인")을 뒤집는 셈이라 팀 재확인 없이 진행하면 안 됨.
  - **(C) 전세는 발표 스크립트에 "이미 실 API로 검증된 배경 근거"로만 인용하고 라이브 쇼케이스에는 안 열되, 코드는 남겨 백업 슬라이드/Q&A 대응용으로만 유지.**

### Q5. Gate 0(`/llm` 게이트웨이 JB_project2 이식)가 제출 전까지 끝나지 않으면 F1 "실 LLM"을 어떻게 정직하게 표기할지?

- **선택지**:
  - **(A) 제출 전 이식을 완료해 F1을 진짜 🟢 실 LLM으로 시연** — 서사적으로 가장 강하나 남은 시간(현재 7/4, 제출 7/5 10:30) 안에 이식+재시딩+리허설까지 끝나야 하는 리스크가 크다.
  - **(B) 이식이 안 되면 F1을 🟡로 정직하게 강등하고, 예선 `02_제품/app`에서 실 LLM이 동작했던 스크린샷/영상을 "이미 검증된 슬라이스"로 보조 증거로만 쓴다** — 이미 F1 문서 자신이 이런 정직 표기 원칙("과장 금지")을 여러 번 명시했으므로 이 노선과 정합적이다.
  - **(C) 제출 정본과 별개로, 발표 당일 데모는 예선 `02_제품/app`(레거시)에서 실 LLM 부분만 열어 보여주고, 제출 GitHub 링크는 JB_project2로 유지** — 코드 정본과 데모 화면이 달라지는 리스크(심사위원이 "왜 두 개 앱이냐"고 물을 수 있음)가 있어 사전에 답변을 준비해야 한다.
