# 무제폴더 핵심이해 — `/Users/river/Downloads/docs/무제 폴더/` 정독 분석

> 대상 5개 파일(디스코드 대화 2건, 본선제출 최종안, HCI/UX 분석, 발표자료)을 전문 정독하고, 우리 볼트의 [[08_본선/03_제품/00_vision/차별성-경험레이어-서사|차별성-경험레이어-서사]]·[[키스톤-확정]]·[[08_본선/03_제품/00_결정-준비/설계/승보-프로토타입-반영|승보-프로토타입-반영]]과 대조했다.

## §1. 이 폴더가 담은 핵심 시스템/제품 관점

무제 폴더는 **"역할 기반 금융 AI 업무 콘솔 + Enter-First Case Flow UI"**라는 하나의 제안을 세 겹(발표자료·HCI 분석·최종안)으로 다듬어 쓴 결과물이다. 핵심 골격은 `계열사(기관) × 역할 × 케이스`를 1차 조직축으로 삼아, AI 에이전트가 케이스를 먼저 읽고 근거·위험신호·다음 액션·다음 담당 역할을 정리하면, 담당자는 화면을 훑고 `Enter`로 통과시키며 예외만 `Enter Enter`로 끌어올리고 책임 있는 최종 승인만 마우스 클릭으로 남기는 구조다. 이 위에 "AI가 업무를 대신하는 게 아니라 사람이 사람답게 판단할 여유를 만든다"는 AX(TX⊃UX·EX·CX·PX) 서사가 얹혀 있으며, KLM/Hick's Law/Fitts's Law와 생산성 실증연구(Fuster, Brynjolfsson, Noy&Zhang, Dell'Acqua)로 조작시간·효율 근거를 뒷받침한다. 디스코드 대화는 이 결과물이 나오기까지의 팀 내부 논의 과정(업무분장, PM 검토, 상호 피드백)을 담고 있다.

## §2. 사용자가 반복 강조했으나 우리 문서가 놓쳤을 가능성이 있는 것

**결론부터: 사용자가 강조한 "담당자/기관이 설정하는 스킬 + 외부 API/MCP + 회사 내부 DB 접근으로 근거를 높이는" differentiator는 무제 폴더 5개 문서 어디에도 이 구체성으로 존재하지 않는다.** 가장 가까운 흔적은 다음 세 곳뿐이고, 전부 "구성 요소 나열" 수준이지 "누가 설정하는가 → 어떻게 근거가 상향되는가"의 흐름으로는 쓰여 있지 않다.

1. **디스코드 대화 1, 김주용(업무분장 메시지)**: "그 외에 스킬이나 플러그인, 외부 플러그인 등 구상 준비 구현 할게요." — 스킬·플러그인·외부 플러그인이라는 단어는 등장하지만 "누가"(담당자/기관) 설정하는지, 그것이 "근거"를 어떻게 높이는지는 전혀 서술되지 않은 to-do 한 줄에 그친다.
2. **디스코드 대화 1, 영욱님 배정 메시지**: "은행에서 어떤 공공데이터들을 사용하는지... 에이전트가 은행에서 돌아가는데 더 강력한 에이전트가 되기 위해 실시간으로 가져와야 할 데이터는 무엇인가... 주식 쪽에서는 실시간 주식 데이터와 다양한 지표들... 실시간 뉴스나 X같은 소식들이나 종토방 커뮤니티 흐름... KIPRIS 상표검색, 전세/보이스피싱 쪽에 쓸 수 있는 공식 숫자나 출처" — 이것은 "외부 데이터로 에이전트를 더 강하게 만든다"는 방향성은 맞지만, **개별 리서치 과제**로 배정된 것이지 "담당자가 스킬을 설정한다"는 제품 기능/차별성 서사로 격상되어 있지 않다.
3. **본선제출 최종안·발표자료의 "데이터 및 기술 활용" 절**: 두 문서 모두 동일하게 "역할별 Skill 모듈 / RAG 기반 규정·근거 검색 / read-only DB/API 연동 / 위험 신호 룰과 모델 결합 / PII 마스킹 및 접근 권한 통제"를 기술 스택 리스트로만 나열한다. 이 목록은 사용자가 말한 재료(스킬·외부 API·내부 DB)와 표면적으로 겹치지만, **"기관·담당자가 직접 설정한다"는 주체성**과 **"판단 직전까지 근거를 끌어올린다"는 단계적 흐름**이 모두 빠져 있다 — 정적인 기술 스펙 나열일 뿐 차별성 논증으로 쓰이지 않는다.

추가로, **`MCP`라는 단어는 5개 문서 전체에서 단 한 번도 등장하지 않는다.** 사용자가 강조한 흐름의 핵심 재료 하나가 무제 폴더에는 아예 부재한다는 뜻이다.

즉 이 발견 자체가 중요한 시사점이다 — 사용자가 강조한 이 differentiator는 **무제 폴더 팀 문서에도, 우리 22종 문서에도 아직 정착되지 않은, 사용자 본인의 추가 통찰**일 가능성이 높다(§5 갭 참고).

## §3. '담당자 설정 → 근거 상향 → 판단 직전 보조' 흐름의 구체 단계 — 문서 근거

무제 폴더 문서에서 재구성 가능한 최대치는 다음과 같다(단, 이는 문서에 흩어진 조각을 이어붙인 것이며, 문서 스스로 이 흐름을 하나의 서사로 명시하지는 않는다).

| 단계 | 문서 근거 |
|---|---|
| 1. 케이스 접수 | "기관 -> 역할 -> 케이스 -> AI 에이전트 정리 -> 담당자 확인 -> 다음 역할 전달 -> 최종 판단 -> 감사 로그"(최종안 §제안 솔루션 개요) |
| 2. 근거 수집(스킬/RAG/DB·API) | "역할별 Skill 모듈 / RAG 기반 규정·근거 검색 / read-only DB/API 연동"(최종안·발표자료 공통, §데이터 및 기술 활용) |
| 3. 외부 데이터로 근거 강화(가장 근접한 실마리, but 리서치 과제 수준) | 영욱님 배정: "더 강력한 에이전트가 되기 위해 실시간으로 가져와야 할 데이터... KIPRIS 상표검색, 전세/보이스피싱 쪽에 쓸 수 있는 공식 숫자나 출처"(디스코드 대화 1) |
| 4. 근거 스트립으로 응축 | "긴 문서를 모두 열지 않아도 핵심 근거를 훑을 수 있게 보여준다"(최종안 §근거 스트립과 감사 로그) |
| 5. 판단 직전 보조(Enter 직전 화면) | "볼 것 정리 -> 기본 액션 제안 -> Enter로 진행 -> 예외만 깊게 보기 -> 최종 승인만 명시 클릭"(HCI/UX 분석 §2 UX 핵심 명제) |
| 6. 최종 판단은 사람 | "AI 에이전트는 케이스를 읽고 근거를 정리하지만... 최종 책임 판단은 사람이 맡는다"(최종안 §제안 솔루션 개요, 디스코드 대화 2와 동문) |

**빠진 연결고리**: "담당자가 스킬을 설정한다"(1단계 이전, 즉 configuration 단계)와 "기관마다 다른 스킬/커넥터 조합을 고른다"는 부분이 문서 어디에도 명문화되어 있지 않다. 위 표의 2·3단계는 "이미 만들어진 Skill/RAG/API가 자동으로 근거를 모은다"는 그림이지, "담당자/기관이 스스로 무엇을 연결할지 설정한다"는 그림이 아니다.

## §4. 디스코드 대화에서 나온 제품/UX 의사결정·논쟁·미결

**결정/합의된 것:**
- 업무분장: 김주용(문서/발표흐름/시연스토리/스킬·플러그인 구상), 이승보(프로토타입 공유 + 은행 DB 연동 설계 명문화), 김민주(디자인 톤/발표 화면/이름·슬로건·승인 UX), 영욱(외부 공공데이터·KIPRIS·전세/피싱 공식출처 리서치 + 자사 자동화 시스템과의 차별점 GPT 딥리서치).
- 핵심 UX 컨셉으로 **"역할 기반 금융 AI 업무 콘솔" + "Enter Flow UI"**를 PM(김민주로 추정)이 제안 → 김주용이 "필요하고 좋을 것 같다", "핵심을 엔터로 한 것도 좋다"고 확정적으로 동의.
- 경쟁력 문구 확정: "금융 업무의 실제 흐름을 역할 기반 게이트로 재구성하고, AI가 케이스를 정리하며, 사람의 감각을 최종 판단과 감지에 집중시킨다는 점... 비금융권 업무에도 확장될 수 있다"(김민주) — JB 대회 취지("금융의 현재를 개선하고 미래의 고객 경험을 새롭게 설계")와의 정합성까지 팀이 확인.
- TX⊃UX·EX·CX·PX·AX 프레이밍(김민주) → 김주용이 "라임 좋다", "애플처럼 하나의 생태계로 인식될 것"이라 강하게 지지.
- "기술보다 가치를 본다"는 팀 정체성 재확인(김주용): "저희보다 더 뛰어난 엔지니어들이 더 잘 구현할 것이다. 그러나 이 전체를 아우르는 경험과 가치를, 우리는 보았고 도전한다."

**제안됐지만 채택 여부 불명(미결):**
- 김주용의 단축키 제안: "컨트롤 1/2/3/4/5 이렇게 누르면 순서대로 각 뷰들로 넘어갈 수 있고, Tap을 누르면 엔터와는 다른 다음 흐름으로" — Enter/Enter Enter 2단 체계 외에 **Ctrl+숫자 뷰 전환**과 **Tab 키 대체 흐름**을 추가 제안했으나, 이후 문서(최종안·발표자료·HCI 분석) 어디에도 이 제안이 반영된 흔적이 없다. 채택 여부가 대화 안에서 마무리되지 않음.
- 김주용이 지적한 리스크: "이때 AI를 얼마나 신뢰할 수 있고, 정확한지에 대한 추가적인 근거와 설명이 필요할 것 같아요." — 이후 최종안에 KLM/생산성 연구 인용이 추가되긴 했으나, 이것이 "신뢰성/정확성 근거" 요구에 정확히 대응하는지는 대화 내에서 재확인되지 않음.
- 김주용이 "이 가설(AI-사람 판단배치)에 대한 기존 연구가 있는지 찾아보겠다"고 자원 — 이는 우리 볼트의 [[08_본선/03_제품/00_결정-준비/근거팩/서사-학술규제-근거팩|서사-학술규제-근거팩]] 작업으로 이미 흡수·적대검증된 것으로 보이며 대화상 미결이 해소된 사례.
- 승보 프로토타입 공유 방식(zip vs GitHub 새 repo) — 방식 미확정, 공유 요청만 있고 응답은 대화에 없음.
- 은행 자동화 시스템과의 차이("대출 과정 자동화와 우리는 뭐가 다른가")를 영욱에게 GPT 딥리서치로 조사 요청 — 완료 여부가 이 대화 안에는 없음.

## §5. 우리 현재 22종 문서 세트와의 갭

1. **"담당자/기관이 설정하는 스킬 + 외부 API/MCP + 내부 DB 접근"이라는 configuration-first differentiator 자체가 문서화되어 있지 않다.** 무제 폴더에도 없고(§2), 우리 볼트에서도 [[08_본선/03_제품/00_결정-준비/설계/승보-프로토타입-반영|승보-프로토타입-반영]]의 "외부 데이터 연결"(JBWC 뷰 목록의 `integrations`, `external_connectors` 5건, degraded/down 상태 관리)이 가장 근접하지만 이것도 **운영자가 상태를 보는 화면**이지 "담당자가 스킬·커넥터를 직접 설정해 근거를 끌어올리는" 사용자 흐름으로 서술돼 있지 않다. **→ 이 흐름을 별도 문서(예: `02_agent-design/스킬-구성가능성.md` 또는 `00_vision/` 하위)로 신규 작성해 명문화할 것을 권고.** MCP는 우리 문서에도 전혀 등장하지 않으므로, 채택 여부부터 팀 결정이 필요하다.
2. **Enter/Enter Enter/Mouse Click 3단 에스컬레이션 메커니즘의 구체 설계가 축약돼 있다.** 우리 [[08_본선/03_제품/00_vision/차별성-경험레이어-서사|차별성-경험레이어-서사]] §4·§7은 "감각 중심으로 구성된 UX(Enter-first)"라고만 짧게 언급하고, "⛔ Enter 49% 빠름 등 성과 수치 주장 금지"라는 가드레일까지 걸어뒀다. 반면 무제 폴더의 HCI/UX 분석·최종안은 KLM(K/P/H/M) 계산으로 **"Enter 진행 1.55초 vs 마우스 진행 3.05초, 약 49% 단축, 하루 200건 기준 약 5분"**이라는 구체 수치와 "습관적 Enter·AI 과신·Enter Enter 모호성·최종 승인 남발·빠른 오판"의 **5개 리스크-방어 설계 표**를 포함한다. **우리 가드레일(49% 주장 금지)과 무제 폴더 문서(49% 명시 주장)가 정면으로 충돌** — 어느 쪽이 최종 발표 원고가 될지, 수치를 "설계근거"로만 쓸지 "성과 수치"로 쓸지 팀 확정이 필요하다.
3. **Ctrl+1~5 / Tab 단축키 제안(김주용, 디스코드)이 어느 문서에도 반영되지 않았다.** 채택할지 폐기할지 결정 자체가 없어 유실 위험이 있다.
4. **"비금융권(민원 처리, 계약 검토, 보험 청구, 공공 행정, 병원 원무)까지 확장"이라는 최종안의 강한 확장 주장**이 우리 [[키스톤-확정]]·[[08_본선/03_제품/00_vision/차별성-경험레이어-서사|차별성-경험레이어-서사]]의 "계열사·비계열사까지"보다 훨씬 더 넓다. 심사 발표에서 이 정도까지 주장할지(과대 주장 리스크) 여부가 우리 문서에는 아직 결정돼 있지 않다.
