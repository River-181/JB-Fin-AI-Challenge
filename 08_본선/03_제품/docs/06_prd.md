## 1. Product Summary

**JB LocalGuard OS**는 지역 금융(전북은행+JB우리캐피탈) 고객의 위험 신호를 `Case`로 모으고, 14+종 전문 AI Agent(canon §2 로스터)가 스킬을 장착해 판단→행동 초안→검증을 수행하되, 고객 대상 행동은 사람 승인(L0~L4, 준법 게이트 L3~L4) 전까지 자동 실행되지 않는 금융 AI Agent 운영 콘솔이다. 핵심 차별점은 **PII 4중 방어**(데이터등급제·토큰화·모델 라우팅·반출 스캔·감사원장)로, 외부 프런티어 LLM의 추론력을 쓰면서도 고객 원본 PII·신용정보를 절대 외부로 반출하지 않는다는 것을 실동작으로 증명하는 데 있다(canon §0, Core Bet Solution) [E4 히어로/E2+ 규제매핑].

> **히어로 케이스 = `CCL-0001`(구 `JBG-104`)**(전주 중앙로 카페 개인사업자 운전자금 검토, corporate-credit 콘솔, `cclConsole.data.js` seed). 히어로 콘솔의 에이전트셋은 corporate-credit **8종**(표면 5: intake·financial·repayment·doc·memo + policy·reply·supervisor), 케이스당 활성 3~5(승보-프로토타입-반영 §7.5) [E4]. 예선 표시코드 `JBG-104`는 동일 "전주 카페" 히어로의 구 코드로, **히어로 ID는 `CCL-0001`로 단일 확정(2026-07-04)** — 본선 전 산출물은 CCL-0001로 통일한다 [확정, definitions §4].

> **실측 정합(2026-07-04)** — 코드 실측 인벤토리([[08_본선/03_제품/구현현황-JB_project2|구현현황-JB_project2]])와 대조한 정정치, 세부는 링크 문서 참조:
> - "14+종 전문 AI Agent"는 canon 메인 로스터(설계) 표기다. **[실측]** 코드에는 이와 별개로 콘솔별 확장 에이전트 40종(CCL8·FDR8·JPO11·JBWC13)·스킬 28종이 존재하며, canon 14종과의 통합 명명은 `[Open Question]`.
> - "역할 콘솔"은 **[실측]** 4개(CCL/FDR/JPO/JBWC)가 아니라 base app·rm-dashboard까지 합쳐 **6개 진입점**, 총 **91개 해시라우트 뷰**로 코드에 존재한다. JBWC는 역할 전환이 아니라 계열사 전환으로 진입한다.
> - 감사 해시체인(`auditChainRecords`)은 **base app에만** 구현돼 있고 CCL/FDR/JPO/JBWC 4개 역할 콘솔의 감사로그는 평문 리스트다 — "전 콘솔 탬퍼-에비던트 감사체인" 주장은 미충족이므로 목표 5(원본 PII 비반출 실증)·성공지표(케이스당 감사이벤트 4건 이상)를 base app 시연 범위로 한정해 표기한다.
> - "AI Agent가 판단·행동초안"은 **[실측]** 전부 JS 템플릿 문자열이며 LLM API 실호출은 0건(코드 전체 `fetch()` 2건은 전세 `?live=1` data.go.kr 공공데이터 조회뿐). 라이브 추론은 §7 2.1.1·2.3.1대로 `[목표/7-4]`.
> - Case는 마크다운 파일이 아니라 **순수 JS 객체 + localStorage**(콘솔별 5개 독립 DB)로 구현돼 있다 — FR-08(§ feature-spec 6.1.1)은 미구현·설계 단계.

## 3. Problem Statement

RM은 계정 행동·서류 상태·정책 제약·공개 시세 등 흩어진 신호를 여러 시스템에서 직접 모아야 하고, 반복 행정(초안작성·증빙대조·누락탐지·기록·handoff 재작업)에 시간을 태운다. 국내 공개 근거로는 대출 만기연장 **1건 30분+**(D16, 단일 2차 출처 → 신규여신 평균으로 일반화 금지 `[약근거]`)이며, RM 1인당 월 케이스 수·handoff 재작업 분은 공개 실측이 없어 `[미검증/가정]`이다(Core Bet Problem). 이 시간이 회수되지 않으면 정작 필요한 최종 판단(승인·조건 조정)에 쓸 여력을 잃는다. 같은 병목이 **사후관리**에서 규제 리스크로 이미 현실화됐다 [E2] — 2026-06-30 시행 "자금용도외 유용 사후점검준칙" 개정으로 의무 점검 대상이 하향(개인사업자 1억→5,000만원 등)돼 케이스가 폭증하는데 RM 인력은 축소 추세다(D2: 5천만원 초과·실행 후 3개월 이내 건은 세금계산서·임대차계약서·송금영수증 건별 대조가 강제 — 캐피탈 적용성은 [미결/7-4]). 일반 AI 챗봇으로는 이 문제를 풀 수 없다 [E2] — 근거 없는 블랙박스 위험점수, 승인 게이트·감사로그 부재는 금융권 실도입에서 규제 위반·책임소재 불명으로 직결된다(심사기준 5.5, D3c·D18).

## 4. Goals

- Triage 시간 단축: 위험 인지→대응 착수 시간 **50% 단축**(canon §3).
- 모든 AI 판단·추천이 근거(Evidence)에 연결됨 — **traceability 100%**.
- 고객 대상 행동은 승인 게이트 통과 전 자동 실행 **0건**(Approval safety 100%).
- 원본 고객 PII를 외부 LLM에 반출하지 않는다는 것을 정적·런타임 스캔으로 실증.
- 계열사(전북은행/JB우리캐피탈)별로 도메인팩만 스왑되는 모듈식 확장성을 실동작 또는 코드 근거(승보 프로토타입)로 증명.

## 5. Non-Goals

- 신용평가·대출 승인·금리/한도 확정 판단을 AI가 대신하지 않는다 — AI는 권고, 심사역이 실무책임, 센터장이 최종책임(AI기본법 §34, 금융위 2026 가이드라인 RACI).
- 법률·규정 확정 판단, 전세사기/피해자결정/보증가입 확정 등 확정적 단정을 AI가 내리지 않는다.
- 승인 없는 자동 고객 접촉·발송이 없다(fraud 케이스는 전 구간 승인 필수).
- 로컬모델 채택의 명분을 "비용 절감"으로 세우지 않는다 — 실제 근거는 원본 PII 비반출·규제 준수다.
- 실제 코어뱅킹/외부 계열사 시스템과의 프로덕션 연동(MVP는 mock 커넥터·시뮬레이션 — §7 그룹5).

## Data Requirements

> 상세는 [[08_본선/03_제품/docs/18_data-strategy|Data Strategy]]·[[08_본선/03_제품/docs/05_domain-model|05_domain-model]]가 SSOT. 여기서는 PRD가 소비하는 요건만 요약한다.

- **데이터 등급제 5단계**(G0 원본 개인신용정보 ~ G4 내부 생성물/Zero-PII 파생)에 따라 AI 입력 가능 여부와 라우팅 대상이 결정된다(data-strategy §데이터 등급제).
- **데이터 레인 4단 분리**(L-실명/L-가명/L-결합/L-외부) — 같은 그룹이라도 원본을 섞지 않고, 외부 LLM에는 L-외부(토큰·위험코드·근거 포인터)만 흐른다.
- **Key Data Inventory**(DATA-001 전세등기, DATA-002 카드매출·상환, DATA-003 CB스코어·여신원장, DATA-004 공공API)는 조달 3층(무상 공개형/유료 건별/기관계약형)에 따라 MVP 편입 여부가 다르다 — DATA-002·003은 [미검증/최영욱 확인 대기], DATA-004는 E4(데모 실동작).
- **원장 비접촉 연동**: 읽기는 CDC(로그 기반), 쓰기는 RM 승인 후 MCI/EAI 표준 전문 경유 — AI가 직접 Write하지 않는다.
- **보존정책**: 개인신용정보 처리기록 3년, 접속기록 1~2년, 전자금융거래기록 5년, RM 초안·중간 프롬프트는 단기 파기 원칙(신용정보법 통제).
- **저장 3단 옵션(opt-in) [E4, 8c274b5]**: 코드 기본은 여전히 `localStorage`이나, `JB_DB_DRIVER` 환경변수로 파일 기반 JSON DB·Supabase 저장을 선택할 수 있는 서버 백엔드가 신설됐다(연결·마이그레이션은 미배선).

## AI Requirements

> 상세는 [[08_본선/03_제품/07_architecture|07_architecture]] §5(모델/API 라우팅)가 SSOT. LLM은 판단을 대신하지 않는다는 원칙이 핵심.

- **4층 모델 라우팅**(계층1 로컬 8B~32B / 계층2 국산 후보 / 계층3 오픈웨이트 / 계층4 외부 premium API) — 승격 기준은 난이도가 아니라 **데이터 등급**이며, 민감정보가 섞이면 로컬로 강등한다.
- **LLM 역할 고정**: LLM은 신용평가·여신심사·FDS 탐지 점수를 직접 산출하지 않는다. 신용/여신 점수화는 특화모델(XGBoost·LightGBM), FDS는 규칙엔진+GNN, 문서 정보추출은 LayoutLMv3·Donut에 라우팅하고, LLM은 의도 분류→근거 범주 결정→도구 호출→결과 요약만 담당한다.
- **결정형 게이트 우선**: Policy Engine(규칙 게이트)이 최종 allow/block/require_approval/escalate를 결정하고, LLM 출력은 그 위에 얹히는 보조 계층이다.
- **PII 비반출**: `restricted`/`confidential` 입력은 로컬모델 경로로만, 반출 스캔을 통과한 비식별 컨텍스트만 외부 LLM으로 전달된다.
- **폴백**: 라이브 LLM이 미연결이거나 실패하면 결정형 골든패스로 자동 전환되어 데모·운영이 끊기지 않는다(feature-spec F-2.1.1 Edge case).
- **런타임 토글(opt-in) [E4, 8c274b5]**: RM 하네스는 "모의 실행" 기본값에 더해 로컬 Ollama 프록시(:8030) 실행을 사용자가 켜는 opt-in 토글을 제공한다(`agentModelSettings.js`) — 기본 판단 루프는 그대로 결정형이다.

## Admin/Operation Requirements

- **승인 게이트 L0~L4**: `approvalLevelMatrix` — L3(80~89점)는 RM+준법 공동 결재, L4는 발송보류/차단. 실 승인 주체는 [Open Question](domain-model §9).
