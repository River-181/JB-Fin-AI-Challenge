# Business Model (DDBM) — JB LocalGuard OS

> 이 문서는 정의서·기능명세서 입력으로 재사용 가능한 수준까지 확장한 DDBM(Deep-Dive Business Model) 초안이다. 7/3 종합 근거(키스톤·ROI·적법성 근거팩)와 **7/3 밤 팀 확정**(하이브리드 도메인 옵션3 + 사후관리 편입 + 에이전트 14 + Enter Flow, [[08_본선/03_제품/01_결정-준비/결정-현황-종합|결정-현황-종합]] ✅)을 반영한다. 남은 미결(데모②전세/피싱 택1, 승인직군 확대 여부, 실 데이터 제공 범위, seat 단가·OPEX 실단가)은 하단 Open Questions에 `[미결/7-4]`로 명시한다.

## Judge-Facing Thesis

JB LocalGuard OS는 전북은행·JB우리캐피탈 두 계열사의 담당 직군(RM·여신심사·사후관리·준법·AML)이 매일 다루는 위험 신호를 `Case → AgentRun → Agent → Skill → Evidence → Approval → Audit` 단일 운영계약으로 묶어, AI는 판단·행동 초안까지만 만들고 사람이 승인 게이트(L0~L4)를 통과시켜야 고객 대상 행동이 나간다. 이 모델의 데이터 우위는 범용 LLM이 흉내 낼 수 없는 지점에서 나온다 — 원본 PII·신용정보는 데이터 등급제·토큰화·모델 라우팅·반출 스캔·감사원장의 4중 방어로 로컬/온프레 모델에만 두고, 외부 프런티어 LLM은 비식별·공개 데이터만 처리한다. 신용정보법 §40조의2·§36조의2, 전자금융감독규정 §15조, 금융위 2026 「금융분야 AI 가이드라인」의 보조수단성·RACI, AI기본법 §34 고영향 AI 사람관리·감독 요건을 설계 시점부터 만족시키므로, 규제 준수 자체가 경쟁사가 단기간에 복제하기 어려운 진입장벽이다[E5].

이 모델은 단발성 자동화 판매가 아니라 **나선형 성장(spiral growth) 구조**를 판다: 케이스가 처리될수록 판단·근거·승인·반려 로그가 쌓이고(Audit), 그 로그로 에이전트·스킬이 정교화되며, 결국 조직 담당자 개인에게만 머물던 암묵지(노하우)가 AI 학습 데이터로 축적되어 3년 내 그룹 AX(AI 전환)의 완전 준비 상태에 도달한다는 것이 팀의 장기 가설이다(회의록 2026-07-04 전략회의, 김주용 "루프를 도는 거예요. 케이스가 생기고 업무가 처리되고 결과물이 나오고 피드백하고 다시 돈다" — `[가설/E1]`, 정량 근거는 파일럿 로그로 사후 확보 필요). JB가 이 나선형 구조를 필요로 하는 이유는 편의가 아니라 재무 KPI다: JB는 2026년을 AX 내재화 원년으로 선언했고, 투자 판단 기준은 `ROE 13+α%`(중장기 15%), `RORWA 1.91%`, `CIR 38.8%`, `CCR 0.86% 이하`, `CET1 12.58%`다[E3, D3b] — 심사 오류·누락 감소는 CCR 방어로, 서류 작업 단축은 CIR 절감으로, 나은 자산 선별은 RORWA 개선으로 직결되어야 "편의 도구"가 아니라 "자본효율 개선 도구"로 통한다.

**차별성의 두 축**은 각각 "왜"와 "어떻게"를 담당한다. [[08_본선/03_제품/00_vision/차별성-경험레이어-서사|차별성-경험레이어-서사]](왜) — 업무를 기능이 아니라 직원·조직·그룹·고객의 경험 흐름(TX⊃UX·EX·CX·PX, AX가 전환 레이어)으로 재해석해 역할 기반 AI Agent 게이트로 재구성했다는 것. [[08_본선/03_제품/00_vision/차별성-설정근거상향-흐름|차별성-설정근거상향-흐름]](어떻게) — 담당자가 케이스·역할·계열사에 맞게 스킬·외부 API/MCP·내부 DB 접근을 **직접 설정**해서 케이스의 **근거를 상향**시키고, AI는 그 근거로 **판단 직전까지만** 준비한다는 메커니즘. 두 문서 모두 "판단과 최종 책임은 사람"이라는 동일한 결론으로 수렴한다.

## 고객 ↔ 사용자 분리 (돈 내는 자 vs 혜택 받는 자)

DDBM 설계에서 가장 먼저 고정해야 할 축은 "누가 지갑을 여는가"와 "누가 매일 화면을 쓰는가"가 다르다는 것이다. 이 둘을 하나로 뭉개면 Value Proposition과 Revenue가 서로 다른 근거를 요구한다는 사실이 가려진다.

| --------------------------- | ---------------------------------------------------------------- | ------------------------------------------------------------------------------------------ | -------------------- |

## DDBM 11 Blocks

### 1. Mission
지역 금융(전북은행·JB우리캐피탈) 담당 직군이 위험 신호를 놓치지 않고, 사람 승인 아래에서만 고객에게 행동하는 AI 운영 콘솔을 제공한다. 단기 미션은 "반복 행정 시간을 회수해 최종 판단에 쓸 여력을 만드는 것"이고(canon §0), 장기 미션(3년 지평)은 "케이스 처리 로그가 쌓일수록 에이전트·스킬이 정교화되고, 조직의 암묵지가 AI 학습 데이터로 전환되어 그룹 AX가 완전히 준비된 상태에 도달하는 것"이다 — **사회적 가치**(취약 소상공인·전세임차인 보호, 담당자의 인간다운 판단 여유 회복)와 **실행가능성**(코드로 선행 실증된 4개 콘솔, canon §9 4함수계약)을 동시에 건다.

### 3. Key Activities
운영 계약은 신호 수집(Evidence) → 계열사×직군별 Agent 판단 → 행동 초안 생성 → RM/준법 승인(L0~L4) → 감사원장 기록의 단일 루프다(canon §8). 하이브리드 확정 도메인 조합에 따라 계열사별 활동이 모듈식으로 스왑된다.

### 4. Key Data — **경쟁우위 자산** (있으면 좋음이 아니다)
Key Data는 부가 기능이 아니라 이 사업의 **해자(moat)**다. 범용 LLM 벤더는 은행 내부 원장·등기부·CB 스코어를 원본 형태로 접근할 권한이 구조적으로 없다 — 우리는 로컬/온프레 라우팅으로 원본을 다루면서도 신용정보법 §40조의2 요건을 만족시키는 유일한 설계를 코드로(harnessVerification 24체크) 선행 실증했다(E4). 이 데이터 접근권 자체가 경쟁사가 계약·규제 준수 트랙레코드 없이는 단기간에 복제할 수 없는 자산이다.

| RISK-AFFIL-001 | Key Partners / Key Data | 계열사(전북은행↔JB우리캐피탈) 식별형 고객원장 상시 통합 | 3단 고정: 고객응대·실행=계열사별 식별형, 그룹 모델·리스크=가명형 결합, 대외·외부 LLM=Zero-PII 파생물 | 준법 + 데이터 거버넌스팀 | 공동 솔트 구조 기본안 제외 |
| RISK-DEP-001 | Key Data / Open Questions | Key Data(카드매출·CB·여신원장) 실 데이터 미확보 | Fallback: 원본 처리는 로컬 모델 라우팅/수기 입력, 공공 API는 사전 확보 스냅샷으로 대체 | 데이터 엔지니어 + 최영욱 | Q-BM-005 |

전체 12개 리스크(RISK-PRIVACY-001~003·BIAS-001·DATA-001~002·ACTION-001·GOV-001~004·PERF-001·AFFIL-001·CLAIM-001~003·DEP-001)의 상세 Likelihood/Severity/Evidence는 [[08_본선/03_제품/docs/19_risk-impact-register|Risk & Impact Register]] 본표를 참조 — 이 표는 그중 DDBM 블록에 직접 연결되는 핵심 5건만 발췌했다.
