# 배포-스택-리서치 (2026-07-03)

> 본선 MVP(`02_제품/app/` 정적 바닐라JS)를 **공개 라이브 데모 URL + 누구나 볼 수 있는 온라인 DB**로 확장하기 위한 배포/호스팅/DB 스택을 Codex로 직접 웹 리서치한 결정 문서. SSoT는 각 도구 공식 문서(Cloudflare Pages/Workers, Supabase). 실제 배포·공개는 **외부공개 게이트 — 사람 승인 후**에만 실행.
> 트리거: 사용자 지시 — "본선 안내를 읽고, 이 MVP를 공개 라이브 데모 URL + 온라인 DB로 누구나 볼 수 있게 하는 데 필요한 플러그인·스킬·CLI·MCP를 Codex로 리서치하라. 손 쓰는 작업은 Sonnet/Haiku, 토큰 무거운 작업은 Codex에 위임."

---

## ⚠️ CRITICAL 프레이밍 (본문 전체의 전제)

- **오프라인 온사이트 심사 경로가 PRIMARY다.** 본선 시연은 팀 노트북에서 네트워크 없이 진행되고, 외부 LLM 호출은 금지된다(6/30 확정 시연원칙 — 로컬 모델 실동작·PII 비반출). 이 배포 스택은 그 경로를 **대체하지 않는다**.
- **공개 URL + 온라인 DB는 SECONDARY·부가 자산**이다 — 접근성, PoC→파일럿 증빙, 심사축 **5.4 확장성**에 대응하는 보조 증거일 뿐 심사 당일 시연 경로가 아니다.
- **온라인 DB는 합성데이터 전용 + RLS read-only**여야 한다 — 심사축 **2.4/5.5**(고객 PII 비반출)를 온라인 자산에서도 어기지 않기 위한 하드 게이트. 실제 배포 실행은 대외공개 행위이므로 **사용자 명시 승인 없이 자율 실행하지 않는다**(AGENTS 3-way 결정모델 — 외부 제출/공개/고객 대상 = 사람만).

## A. 정적 호스팅 — Cloudflare Pages ★검증 후 사용

**무엇**: `02_제품/app/`을 그대로 정적 배포하는 Cloudflare의 정적 사이트 호스팅. 번들러 없는 바닐라JS 구조와 그대로 맞는다(빌드 스텝 불필요, 디렉터리 그대로 업로드).

```sh
npx wrangler pages deploy 02_제품/app --project-name jb-localguard-os
```

**무료 한도**: 월 500 빌드, 커스텀 도메인 100개. 배포마다 고유 프리뷰 URL 자동 발급 + 프로덕션 URL 고정.

**왜**: 현재 MVP는 `index.html`/`app.js`/`modules.js`/`styles.css` 정적 파일 그대로 서빙 가능한 구조라 프레임워크 마이그레이션 없이 배포 가능. `verify_static.py`가 요구하는 파일 구조·문자열 계약에 영향 없음(배포는 산출물을 옮길 뿐 소스를 바꾸지 않음).

## B. 온라인 DB — Supabase (Postgres + Realtime) ★검증 후 사용

**무엇**: Postgres 기반 BaaS. Realtime(변경 구독)으로 "누구나 지켜보는" 라이브 데모에 적합 — Case가 컬럼을 이동하거나 승인되는 순간을 브라우저에서 실시간으로 보여줄 수 있다.

```sh
npm i -D supabase
npx supabase login
npx supabase link --project-ref <ref>
npx supabase db push
```

브라우저 CDN(번들러 불필요, 기존 `<script>` 로딩 방식과 동일):
```html
<script src="https://cdn.jsdelivr.net/npm/@supabase/supabase-js@2"></script>
```

**무료 한도**: DB 500MB, Realtime 동시접속 200 peak, 메시지 200만/월.

**RLS(Row Level Security) 원칙**: `anon`(비로그인 공개 조회자) 롤은 **SELECT만** 허용. 쓰기(Case 이동·승인)는 오퍼레이터 토큰 또는 RPC 함수를 통해서만.

**합성데이터 전용 강제**: PII 비반출은 UI 규칙이 아니라 **DB 제약(CHECK constraint)**으로 강제한다 — 주민등록번호·전화번호 정규식에 매칭되면 INSERT 자체가 거부된다. 아래 [스키마](#최소-sql-스키마) 참조.

## C. 백엔드 — 별도 서버 없음 (Supabase RPC) ★검증 후 사용

**결론**: Node/Hono 같은 상시 구동 백엔드 서버를 새로 두지 않는다. `computeRiskDecision`/`buildDashboardData`/`auditChainRecords`/`moveCaseToColumn` 4함수 계약은 **클라이언트 순수 함수로 유지**하고, 쓰기가 필요한 지점(케이스 컬럼 이동)만 Postgres 함수(RPC)로 감싼다.

```sql
-- move_case_to_column: 이동 + 감사기록을 하나의 트랜잭션으로 원자화
create or replace function move_case_to_column(p_case_id uuid, p_to_column text)
returns void language plpgsql security definer as $$
begin
  update demo_cases set column_status = p_to_column, updated_at = now() where id = p_case_id;
  insert into audit_ledger (case_id, action, detail) values (p_case_id, 'move_column', p_to_column);
end;
$$;
```

**왜**: `moveCaseToColumn`이 이미 "이동+감사"를 한 단위로 취급하는 함수 계약이므로, RPC 하나로 그 계약을 그대로 승격할 수 있다 — 서버 프로세스·배포·모니터링 부담 없이 원자성만 얻는다.

**폴백(미채택, 필요 시만)**: Hono on Cloudflare Workers — `npm create hono@latest` → `npm run deploy`. Workers 무료 100k req/day. RPC로 안 되는 로직(외부 API 프록시 등)이 실제로 생기면 그때 검토.

| 도구 | 설치/와이어링 | 인증 | 용도 |

## F. 오프라인↔온라인 아키텍처 — 단일 DataSource 인터페이스

바닐라JS 무프레임워크 원칙을 유지한 채 두 배포 타깃을 스왑하는 최소 인터페이스:

```js
// DataSource 계약 — 4함수(computeRiskDecision/buildDashboardData/auditChainRecords/moveCaseToColumn)는
// 이 인터페이스 위에서 동일하게 동작한다.
interface DataSource {
  loadState(): Promise<State>
  saveState(state: State): Promise<void>
  subscribe(onChange: (state: State) => void): () => void  // unsubscribe 반환
  appendAudit(entry: AuditEntry): Promise<void>
}
```

- **LocalStorageDataSource** — 현행 그대로. 오프라인 온사이트 심사 데모(Rule Engine 기본, 선택적 Ollama/EXAONE 로컬 추론, JSON 감사 export). 네트워크 요구 없음 — 심사 당일 기본 경로.
- **SupabaseDataSource** — `loadState`=SELECT, `saveState`/`appendAudit`=RPC 호출, `subscribe`=Supabase Realtime 구독. 공개 온라인 데모 전용.

둘 다 미구현(설계만) — 구현 착수는 별도 승인 후.

## 최소 SQL 스키마

```sql
create table demo_cases (
  id uuid primary key default gen_random_uuid(),
  title text not null,
  column_status text not null default 'intake',
  risk_score int,
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now(),
  -- synthetic_only: 주민등록번호(6자리-7자리) · 국내 전화번호 패턴을 차단
  constraint synthetic_only check (
    title !~ '\d{6}-?\d{7}' and
    title !~ '01[0-9]-?\d{3,4}-?\d{4}'
  )
);

create table audit_ledger (
  id bigint generated always as identity primary key,
  case_id uuid not null references demo_cases(id),
  action text not null,
  detail text,
  created_at timestamptz not null default now()
);

alter table demo_cases enable row level security;
alter table audit_ledger enable row level security;

-- anon(공개 관람자) = 조회만
create policy "public read demo_cases" on demo_cases for select to anon using (true);
create policy "public read audit_ledger" on audit_ledger for select to anon using (true);
-- 쓰기는 RLS로 막고 move_case_to_column() RPC(SECURITY DEFINER)로만 우회 허용
```

## Realtime 브라우저 스니펫

```html
<script src="https://cdn.jsdelivr.net/npm/@supabase/supabase-js@2"></script>
<script>
  const supabase = window.supabase.createClient(SUPABASE_URL, SUPABASE_ANON_KEY);
  supabase
    .channel('demo_cases_changes')
    .on('postgres_changes', { event: '*', schema: 'public', table: 'demo_cases' }, (payload) => {
      // 관람자 브라우저에서 케이스 이동/승인을 실시간 반영
      console.log('case changed', payload);
    })
    .subscribe();
</script>
```

## AI 실행성 (사람↔AI 경계)

- **교정 배경**: 앞선 Codex 리서치가 `wrangler login` / `supabase login`을 권장했으나, 둘 다 브라우저 OAuth **대화형** 명령이라 AI 에이전트가 완결할 수 없다. 교정 원칙 = **비대화형만 사용**.
- **정적 공개데모(사람 개입 0)**: `gh` CLI(이미 인증됨, 토큰·브라우저 불필요)만으로 배포. 저장소 `LSB-afk/JB-Fin-AI-Challenge`는 PUBLIC이나 인증 사용자(River-181)가 팀 origin의 admin이 아니라 Pages를 켤 수 없음 → **사용자 fork `River-181/JB-Fin-AI-Challenge`(PUBLIC·admin ✓)에서 배포**. 대상 URL: `https://river-181.github.io/JB-Fin-AI-Challenge/`.
  - 실행: `OWNER=River-181 bash 08_본선/_system/tools/deploy-public-demo.sh`
  - 자가검증: 배포 후 URL을 폴링해 `index.html`의 `<title>` 로드베어링 문자열 `"JB 금융안전 업무지원"`이 실제로 서빙되는지 확인.

- **신설 스크립트**: [[deploy-public-demo.sh]] — `--check`(읽기전용 프리플라이트: gh 인증·repo admin·앱 디렉터리·검증 문자열 확인) 프리플라이트가 fork 대상 GREEN, `bash -n` 문법 검사 통과. 실제 배포는 미실행(외부공개 게이트 — 사용자 승인 대기).
