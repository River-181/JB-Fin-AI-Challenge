# 케이스 스터디 — JB LocalGuard OS

> 목적: 프로젝트 전체를 처음 보는 사람(채용 담당자·심사위원·다른 AI)에게 배경→문제→해결 방향→만든 것을 한 호흡으로 설명하는 문서. 개인 경험담은 [[08_본선/06_회고와포트폴리오/03_포트폴리오/경험-에세이|경험 에세이]], 근거 조회는 [[08_본선/06_회고와포트폴리오/03_포트폴리오/증거지도|증거지도]] 참고.

## 한 줄

금융 AI 에이전트를 고객 접점 자동화가 아니라 `Case → AgentRun → Evidence → Approval → Audit` 운영 루프로 설계해, 민감정보 비반출과 사람 승인 게이트를 중심에 둔 내부 업무 콘솔을 만들었다.

## 배경

JB금융그룹 Fin:AI Challenge는 청년(만 19~34세) 대상 AI 서비스 개발 경진대회로, 자유주제 트랙에서 금융과 AI 에이전트를 결합한 서비스를 겨루는 자리였다(총 상금 1억원, 대상 5천만원). 팀 망상궤도(대회 등록명 JByond) 4인은 지역 금융(전북은행·JB우리캐피탈) 고객의 위험 신호를 다루는 운영 콘솔을 선택했다. 소상공인 자금압박·전세사기·보이스피싱은 고객과 은행 모두에게 손실을 만들지만, 현장 담당자는 흩어진 신호를 한 화면에서 판단하기 어렵다는 문제의식이었다.

## 문제

은행 내부 담당자는 여러 시스템과 문서를 오가며 고객 위험을 파악한다. 단순 챗봇은 답을 줄 수 있지만, 금융권에서는 고객 대상 행동·개인정보·승인 책임·감사 기록이 더 중요하다. 그래서 우리가 실제로 풀어야 했던 문제는 "AI가 답을 잘 말하는가"가 아니라 "AI가 판단과 행동 초안을 만들 때 무엇을 근거로 했고, 누가 승인했으며, 어떤 정보가 외부로 나가지 않았는가"였다.

## 해결 방향

| 설계축 | 결정 |
|---|---|
| 제품 형태 | 내부 운영 콘솔 (챗봇 아님) |
| 핵심 단위 | 고객 위험을 모은 `Case` |
| AI 역할 | 판단 · 행동 초안 · 검증 · 다음 확인 항목 제안 |
| 통제 | 고객 대상 행동은 사람 승인 전 자동 실행 금지 |
| 보안 | 원본 PII·신용정보 외부 반출 금지, 모델 라우팅으로 스코프 격리 |
| 증거 | 모든 판단은 Evidence·Audit으로 연결 |

## 강조할 점

- 금융 AI는 모델 성능만으로 설득되지 않는다. 개인정보·승인권한·감사·책임소재를 제품 구조로 먼저 풀어야 한다.
- 48시간 안에 전부 실구현하지 못했지만, 실제 구현된 것과 목업인 것을 코드 기준으로 구분해 정직하게 남겼다 — [[08_본선/03_제품/reports/구현현황-JB_project2|구현현황 리포트]].
- 결과와 별개로 문제를 보는 기준이 분명했다: 기술 데모보다 실사용 가능성과 운영 리스크를 더 중요하게 봤다.

## 과장하면 안 되는 점

- 모든 에이전트 출력이 실시간 LLM 호출로 생성되는 것은 아니다. 구현현황 리포트 기준 판단 루프 상당수는 결정론적 mock이고, 로컬 모델(Ollama) 경로는 opt-in 버튼을 눌러야 실행된다.
- 본선 제품 코드는 이 모노레포(문서·기획 정본)와 별도 레포([[08_본선/06_회고와포트폴리오/00_개요/링크허브|링크 허브]] 참고)에 있다.
- ROI(연 7.66억)는 추정·조건부 수치이지 실측이 아니다 — [[08_본선/06_회고와포트폴리오/03_포트폴리오/증거지도|증거지도]] 참고.
- 팀 결과는 라운딩 평가 상위 5팀 미포함(확정)이며 최종 등급·상금은 미확정이다 — [[08_본선/06_회고와포트폴리오/00_개요/팩트시트|팩트시트]] 참고.
