## 4. 로컬모델 opt-in

| 항목 | 내용 |
|---|---|
| 무엇을 | 초기(D4)에는 "하이브리드"(클로드/코덱스 연결 + 로컬모델 실동작, 이승보 PC에서 최종테스트)로 확정했으나, 최종 구현에서는 로컬모델(ollama)이 opt-in 토글로 정착 — 기본 흐름은 mock, 사용자가 켜야 로컬 실행 |
| 왜 | PII 스텝만 로컬 실행으로 보안 서사를 지키면서도, 데모 완주 안정성을 위해 기본값은 안전한 목업으로 유지 |
| 근거 | [[08_본선/04_증빙/04_회의록/회의록-2026-07-04-스코프확정|회의록-2026-07-04-스코프확정]]:47-51,73,84 (초기 D4 결정), `session-log.md:269-271`("리터럴 기본값 ollama ≠ 기본 흐름 mock" 정밀 판정, ollama=opt-in 확정) |
| 결과 | 보안 서사(로컬 실행 가능)는 유지됐지만, "하이브리드 실동작"이라는 초기 표현과 실제 구현(opt-in 토글) 사이에 표현 차이가 있어 발표 자료에서 정정이 필요했다 |
| 지금 다시라면 | opt-in 자체는 합리적인 안전판이다. 다만 "하이브리드"라는 초기 문구를 쓸 때부터 "기본값은 목업, 로컬은 옵션"이라는 조건을 함께 못박아, 나중에 표현을 정정하는 수고를 줄인다 |

## 6. 회의 기록 체계 개정 — 원문+정리본 2파일 → 정리본 1파일

| 항목 | 내용 |
|---|---|
| 무엇을 | 6/29에 정한 "원문(gitignore)+정리본(추적)" 2파일 구조를, 정리본 1파일 안에 "발췌(줄글)+구조화 요약"을 함께 담는 구조로 재개정 |
| 왜 | gitignore된 원문 폴더는 AI가 git-only clone 같은 다른 실행 환경에서 못 읽을 수 있어, 정작 풍부한 내용이 AI에 안 보이고 부실한 요약만 보이는 구조적 결함이 있었다("이러니까 AI가 이해를 못하지" — 사용자 지적) |
| 근거 | `decision-log.md:150-154` |
| 결과 | 회의록 9건을 1파일 구조로 재작업, `_회의록-INDEX.md` 재구성까지 이어졌다 — 재작업 비용은 있었지만 이후 회의록이 프롬프트 인풋으로 바로 쓸 수 있는 완전성을 갖추게 됐다 |
| 지금 다시라면 | gitignore가 "AI가 읽을 수 있는 범위"에 영향을 준다는 점을 회의 기록 체계를 처음 설계할 때부터 고려했어야 한다 — 대외비 보호와 AI 판독성은 별개 축으로 다뤄야 한다 |
