# 협업 방법론

> 목적: "우리가 실제로 어떻게 일했나"를 다음 프로젝트나 다른 팀이 그대로 가져다 쓸 수 있는 **재사용 플레이북**으로 정리한다. [[08_본선/06_회고와포트폴리오/02_종합/미시-회고|미시-회고]]가 "무엇이 잘못됐나(교훈)"를 다룬다면, 이 문서는 "어떻게 하면 되나(절차)"만 다룬다 — 같은 사건을 다시 서술하지 않고 필요할 때 미시-회고를 링크로 참조한다.

## 1. 병렬 AI 세션 오케스트레이션

본선 48시간 동안 팀은 한 사람이 순서대로 작업하는 대신, 여러 AI 세션을 동시에 돌리고 한 세션이 "총괄" 역할로 조율하는 방식을 실전 규모로 운용했다. `session-log.md`에 기록된 대표 사이클 두 개를 그대로 인계 가능한 패턴으로 뽑으면 다음과 같다.

**패턴 A — 총괄 1 + 실무 N (제출물 조립형)**: 제출물 9종을 병렬로 조립할 때 "Fable 총괄 + Sonnet 6팀" 구조를 썼다(`session-log.md:284-290`). 총괄이 먼저 산출물 목록을 쪼개 팀별로 폴더·파일 단위 소유권을 배정하고, 각 팀이 독립적으로 작업한 뒤 총괄이 마지막에 정합성만 훑는다. 팀 이름을 알파벳(A~N)으로 붙여 "누가 무엇을 끝냈는지"를 로그 한 줄로 추적할 수 있게 했다(`session-log.md:245-282`의 A~L 트랙).

**패턴 B — 다트랙 동시전(마감 압박형)**: 마감 직전 "총력전 4트랙"(R1~R4 + fixer 2기)으로 점수 개선·README 본선화·최종 패키징을 동시에 돌렸다(`session-log.md:299-303`). 트랙마다 결과물 영역이 겹치지 않게 나누고(코드 픽스 / README / 패키징), 완료 시점에만 총괄이 병합했다.

**모델 선택 기준**: "하이쿠가 쉽고 반복적인 거 하면 된다"는 실제 지시(`session-log.md:202`)가 그대로 원칙이 됐다 — 판단 없이 반복 가능한 작업(색인 정리, bare 링크 경로 지정)은 Haiku 병렬 4개, 조립·정합성 판단이 필요한 작업은 Sonnet, 전체 조율은 상위 모델(Fable)로 위임했다.

**재사용 체크리스트**
- 시작 전에 산출물/영역을 겹치지 않게 쪼갠다 — 같은 파일을 두 세션이 동시에 건드리게 두지 않는다.
- 반복·기계적 작업만 가벼운 모델에, 판단이 필요한 조립·검증은 무거운 모델에 배정한다.
- 세션마다 이름(알파벳이든 역할명이든)을 붙여 로그에서 추적 가능하게 한다.
- 총괄은 중간 산출물을 만들지 않고 배정·병합·검증만 한다.

## 2. Capture-by-default 3계층 로그

팀은 결정과 행동을 사후에 재구성하지 않고, 진행 중에 그때그때 세 계층으로 나눠 기록했다.

| 계층 | 파일 | 담는 내용 |
|---|---|---|
| 행동 로그 | `08_본선/04_증빙/01_핵심로그/session-log.md` | 세션마다 "한 일 / 결과물 / 다음"을 append-only로 누적 |
| 결정 로그 | `08_본선/04_증빙/01_핵심로그/decision-log.md` | "선택 / 이유 / 대안 / 상태" 형식으로 의사결정만 별도 추적 |
| 원문 로그 | `08_본선/04_증빙/01_핵심로그/프롬프트-로그.md` | 실제 프롬프트 원문을 손대지 않고 보존 |

세 계층을 분리한 이유는 용도가 다르기 때문이다 — "무슨 일이 있었나"(session-log)와 "왜 그렇게 결정했나"(decision-log)를 섞으면 나중에 결정 하나를 찾으려고 긴 행동 로그를 처음부터 훑어야 한다. 실제로 두 로그가 중복되기 시작했을 때(회의 결정 내용이 decision-log와 회의록 양쪽에 똑같이 적힘) 발견 즉시 decision-log 쪽을 포인터로 축약하고 원 내용은 회의록에만 남기는 정리를 했다(`decision-log.md:150-154`).

이 체계가 실제로 작동했다는 증거는 이 회고 문서 자체다 — [[08_본선/06_회고와포트폴리오/01_회고/팀-회고|팀 회고]]와 [[08_본선/06_회고와포트폴리오/02_종합/거시-회고|거시-회고]]의 모든 인용이 추측이 아니라 `session-log.md`·`decision-log.md`의 실제 줄 번호에서 나온다.

**재사용 체크리스트**
- 로그를 최소 3계층(행동/결정/원문)으로 나눈다 — 하나로 합치면 반드시 역할이 섞이고 드리프트가 생긴다.
- 결정 로그는 "선택/이유/대안/상태" 4항목 고정 포맷을 쓴다 — 나중에 "왜 그때 그렇게 했나"를 재구성할 때 이유·대안 항목이 없으면 복원이 불가능하다.
- append-only 원칙을 지킨다 — 과거 로그를 지우거나 고쳐쓰지 않는다(이 원칙이 왜 중요한지는 §3의 벌크 치환 사고에서 확인된다).

## 3. 병렬 세션 충돌 처리 프로토콜 — 되돌리지 않고 이어받기

여러 AI 세션이 같은 저장소를 동시에 건드리면 충돌이 난다. 이번 프로젝트에서 실제로 검증된 원칙은 "충돌을 감지해도 다른 세션의 작업 자체는 되돌리지 않는다"였다(사고 상세는 [[08_본선/06_회고와포트폴리오/02_종합/미시-회고|미시-회고]] #2 참고, `session-log.md:204`).

절차로 정리하면:

1. **감지**: 검증 지표가 갑자기 튄다(예: 도달성 미달 2건→118건). 내 작업 실수로 단정하지 않고 원인부터 조사한다.
2. **판별**: 원인이 다른 동시 세션의 의도된 작업(예: 폴더 구조 재편)인지, 진짜 회귀인지 구분한다.
3. **대기**: 의도된 작업으로 확인되면 롤백하지 않고 그 세션이 자체적으로 마무리할 때까지 기다린다.
4. **재확인·잔여 처리**: 완료된 뒤 재검증해서, 남은 미처리분(그 세션이 놓친 것)만 수동으로 마저 고친다.

이 원칙이 실패했던 사례도 로그에 정직하게 남아 있다 — 다른 세션의 벌크 치환 실수(위키링크 `[[ ]]` 경계 밖 텍스트까지 문자열 그대로 치환)로 38개 파일 113곳이 오염된 사고(`session-log.md:217-222`)는 "되돌리지 않기" 원칙과는 별개로, 애초에 벌크 편집 전 "이 경로를 다른 세션이 건드리고 있는가"를 확인하지 않아 발생했다. 복구 자체는 셸 치환(sed) 1차 → 실패한 11개 파일에 한해 Node.js `String.normalize('NFC')` 적용 2차로 마쳤다.

**재사용 체크리스트**
- 검증 지표 급변을 발견하면 먼저 "내 작업 문제"가 아니라 "동시 세션 충돌"을 의심하고 원인부터 찾는다.
- 다른 세션의 의도된 리팩터는 롤백 대상이 아니다 — 완료를 기다렸다가 잔여만 흡수한다.
- 벌크 편집(특히 문자열 치환) 전에는 위키링크 `[[ ]]` 경계를 매치 범위에서 제외하고, 대상 파일이 유니코드 정규화(NFC/NFD)가 섞여 있을 수 있다는 걸 전제한다.

## 4. 근거 정직성 규율 — `[추정]` 표기

발표 자료에 들어가는 모든 수치에 신뢰 수준을 스스로 표시하는 규율을 지켰다. ROI 수치(연 7.66억)를 실측치가 아니라 `[추정/조건부]`로 명시한 채 사용한 것이 대표 사례다. 발표 직전에는 인용 근거 33건을 전수 재검증해 환각 0건을 확인하고, 그 과정에서 발견한 정정 6건(경찰청 단독 수치를 전체 수치처럼 쓴 것 등)을 반영한 뒤에야 자료를 확정했다(`session-log.md:294`).

이 규율의 핵심은 "한 번 적어둔 숫자도 재검증 대상"이라는 점이다 — 팀이 스스로 문서화한 에이전트 수·콘솔 판정 같은 사실도 코드 재실측과 주기적으로 맞대봐야 드리프트를 잡을 수 있었다(예: "RM=단일 페이지"로 적었던 판정이 실측 후 "11에이전트 완전 하네스"로 정정됨, `session-log.md:263-265`).

**재사용 체크리스트**
- 실측이 아닌 수치는 문서에 적는 순간 `[추정]`/`[조건부]` 태그를 붙인다 — 나중에 태그를 떼는 건 검증 후, 붙이는 건 작성 즉시.
- 발표·제출 직전에는 인용 근거를 전수 재검증하는 단계를 별도로 둔다(건수 세고 결과를 로그에 남긴다).
- 팀이 과거에 적어둔 "확정" 표기도 코드/현장과 주기적으로 재대조한다 — 문서가 SSOT가 아니라 코드/현장이 SSOT다.
