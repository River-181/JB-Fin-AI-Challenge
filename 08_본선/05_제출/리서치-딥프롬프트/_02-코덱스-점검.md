## 총평(3줄)
- `19종`은 파일 수 기준·심사 항목 매핑 기준 모두 충족(경로상 19개 D* 파일 존재)하며, 도메인 분해/경계 규칙/출처 가드가 일관적입니다.
- 다만 실무 적용성 관점에서는 `심사 4(검증)`와 `심사 5(운영리스크)`의 상호 연동이 다소 분리되어, “파일럿 KPI 증명”이 하나의 완결 패키지로 묶이지 않습니다.
- 표본 점검(D1a/D5b/D10/D13/Dplus-a)에서 큰 결함은 없으나, 고가치 구매결정(수억 원 투자) 판단에 필요한 “가격-효과 민감도 + 감사/운영 증거 파이프라인”은 아직 분산되어 있습니다.

## 누락·갭(우선순위순)
1. **최우선: 투자 의사결정 패키지(ROI)의 실증 축이 분산**
   - 원인: `Dplus-a`가 ROI를 수식화했지만, `D13`은 MVP 증빙, `D10`은 보안, `D12`는 인증 경로에 흩어져 있음.
   - 처방: `Dplus-a`에 `사전/사후 파일럿 KPI`, `감사 증빙`, `보안 실패사례 비용` 입력변수까지 묶은 별도 부록 1개 추가 (출처 분리: 1차=근거값, 2차=산식 가정).
2. **심사 4의 실패/운영 리스크 시나리오가 약함**
   - 원인: `D13`은 성공기준 중심으로 설계되어 있지만 "허용안함·오차단·긴급정지·오탐 연쇄" 같은 운영 실패 케이스가 보강 부족.
   - 처방: `D13`에 `부정합/오탐/중단` 케이스를 포함한 데모·파일럿 실패 프로토콜을 강제 칼럼으로 추가.
3. **심사 5의 운영리스크 정량화가 일부 간접**
   - 원인: `Dplus-b`가 운영리스크를 다루더라도 비용·시간 기반 정량(예: 운영인건비 절감, 보안사고 회복시간)과 심사 연결성이 프롬프트에서 덜 직접.
   - 처방: `Dplus-b`에서 `RTO/RPO`, `SLO`, `인력투입시간`, `규제 대응 지연`을 출력 형식에 의무화.
4. **경쟁·유사 대안의 실제 가격-도입 장벽 비교 미완성 위험**
   - 원인: `Dplus-a`의 가격 항목이 공개 비율 한계에 민감하고, JB 내부 규제/승인 제약 반영이 약함.
   - 처방: 공개 조달·입찰·공시 기반으로 `가격 하한선/상한선` 대역(구축형/SaaS형) 산출 지시를 추가, `규제제약 비용`을 별도 항목으로 분리.

## 중복·경계
1. **D5b ↔ D10 (허용/기술 경계)**
   - 고정: `_00`에서 명시한 경계가 좋지만, 각 파일의 실제 질의문에서 정책 근거를 기술 실행 항목과 반복 인용하는 경향이 남음.
   - 처리: `D5b`는 조문·정책 판단/조건만, `D10`은 위협·게이트웨이·아키텍처만 추적하도록 실행 전 `스코프 라인`으로 강제.
2. **D1a/D1b ↔ D2 (수치 vs 메커니즘)**
   - 고정: 분해 상 문제 의식은 명확.
   - 처리: D2에서 "비용/손실 산식"이 사실값 재수집으로 넘어가면 중복 발생하므로, D1류 수치 재확인 시 D2는 “작동 단계 맵만”으로 한정.
3. **D8 ↔ D13 (생산성 일반 vs 제품 검증)**
   - 장점: 경계 노트로 분리했으나, 실제 실행에서 D13이 성공기준 산출 시 D8의 지표 정의를 과도 재해석할 위험.
   - 처리: `D13`에서 D8 수치 인용 시 반드시 `출력형식` 칼럼 `소스도메인(외부/제품내)`를 요구.
4. **D6 ↔ D7a**
- 장점: 경계 노트가 강함.
- 처리: `D6`의 “이식지점” 항목에서 D7a 취득데이터만 쓰고, 내부망 구조 추정은 명시적으로 금지하는 문구를 1회 더 고정.

## 신규 프롬프트 후보
1. `D14-도입 KPI·가격 민감도 실증 (파일럿 운영)`
   - 수치 기반 파일럿 산식(기대절감액, 오탐비용, 인력재배치효과, 보안 사고 회피비용)과 최소 3개 표본 시나리오(보수/기준/낙관) 템플릿.
   - 적합 심사: 4번/5번, 바이어 설득력 극대화.
2. `D15-승인우선 루프 사고 대응 연습 및 감사 증거`
   - 승인 루프 실패, 오탐 오탑, 롤백, human override 실패 대응 프로토콜을 실측 로그 산식으로 정리.
   - 적합 심사: 5.5(운영리스크), 4.2(실동작 검증).
3. `D16-데이터 품질·개입 거버넌스(실무성)`
   - 소상공인/전세/피싱 각 도메인의 입력 데이터 결측·오염·지연이 의사결정 정확도에 미치는 영향과 보정규칙.
   - 적합 심사: 3번(기술 실현성), 4번(MVP 신뢰도), 5번(안전성).

## 엔진배정 의견
1. **유지 추천: `D10`/`D11`/`D9`는 현재와 같이 Codex**
   - 이 세 개는 기술/설계/학술 인용 적합성이 높고, Perplexity보다 텍스트 합성-구조 분석이 유리합니다.
2. **재고려 제안: `D5b`를 Perplexity 우선으로 이동**
   - 근거: 법·정책·조문 검증, 시행일 재확인, 시행규칙 교차 검증 성격이 높고, 출처 정확도 요구가 큼.
   - 현안 제약: Perplexity 월 5–10 제한이므로 상위 우선순위 5건만 보장 실행 시 1차 투입 대상.
3. **재고려 제안: `D2`는 Gemini 유지 가능, 또는 소수 핵심 항목만 Perplexity**
   - 현행: Gemini로 충분하나 D2 일부는 RM 업무 메커니즘 수치 기반 근거가 있어 항목 분해만 단순히 하면 Gemini/GPT에서 처리 가능.
4. **유지 추천: `Dplus-a`는 Gemini 유지**
   - 경쟁/시장/ROI는 다중 도메인 통합 추론이 유리하고, 단순 수치 인용보다 추정모형 프레임 구성에 강함.

## 한 줄 결론
현재 19종은 큰 틀에서 완성도가 높아 점검 프레임 자체는 유효하지만, 수억 단위 구매의심사(심사 4·5) 대응력을 위해선 `ROI·실패모드·운영증거`를 한 번 더 묶는 보완 프롬프트 한 세트가 즉시 유효합니다.
