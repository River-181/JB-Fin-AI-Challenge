# D6 — 은행시스템 · 코어뱅킹 · 연동표준

## 메타
- **도메인**: 실제 은행 시스템 구성 — 코어뱅킹·계정계/정보계·채널(앱/창구/콜)·표준 연동(오픈뱅킹·금융결제원·마이데이터). 우리 콘솔(JB LocalGuard OS)을 **어디에 어떻게 이식·연결할지** 어필할 근거.
- **티어**: ★★
- **심사항목**: 3 (3.3 기술접근 구체성 · 3.4 시스템 구성도 구현가능성)
- **실행 배치**: 배치 2 (정합·규제·구현)
- **경계 메모**:
  - **D6(여기) ↔ D7a**: *행 안에서 흐르는 데이터=D6(내부 연동·표준 인터페이스) / 밖에서 가져오는 외부 데이터=D7a(공공·금융 API 취득).*
  - **D6 ↔ D10**: *시스템 구성·연동 표준=D6 / 보안 아키텍처·망분리 기술구현·위협 방어=D10.* 보안은 여기서 다루지 않는다.
  - **D6 ↔ D9**: *은행 측 표준 인터페이스·아키텍처=D6 / 우리 구현 스택·LLMOps=D9.*

## 딥리서치 프롬프트 (이 아래 통째로 붙여넣기)

당신은 **한국 은행 IT 아키텍처 분석가**다. 목표는, 외부 스타트업이 만든 **RM용 거버넌드 AI Agent 콘솔(가칭 JB LocalGuard OS)** 을 한국 시중·지방은행의 실제 시스템에 **어느 지점에 어떻게 이식·연결할 수 있는지**를 공개 자료만으로 입증할 수 있게, 한국 은행 시스템의 표준 구성과 연동 인터페이스를 깊이 있게 조사하는 것이다. 이 콘솔은 위험 케이스를 모아 여러 AI 에이전트가 판단·행동초안·검증을 수행하되 고객 대상 행동은 사람 승인 전까지 차단하는 운영 콘솔이며, **고객 원본 PII·신용정보를 외부로 반출하지 않는** 설계를 전제한다.

### 알아내야 할 것 (구체 질문)
1. **계정계(Account/원장)·정보계(Information/분석)·채널계의 분리 구조**: 한국 은행 시스템에서 계정계(코어뱅킹 원장·거래처리), 정보계(데이터웨어하우스·MIS·분석), 채널계(앱·인터넷뱅킹·창구단말·콜센터 IVR)는 각각 어떤 역할을 맡고 어떻게 분리·연동되는가? 계정계가 실시간 거래의 SoR(System of Record)이고 정보계가 배치/분석용이라는 일반 통념이 실제로 어떻게 구현되는가?
2. **코어뱅킹 현황과 현대화 흐름**: 한국 은행권 코어뱅킹의 일반적 구성(메인프레임 vs 유닉스 다운사이징 vs 차세대 시스템), 차세대(Next-Gen) 프로젝트의 전형적 범위, 마이크로서비스/API화·계정계 개방 동향. 벤더(예: 삼성SDS, LG CNS, SK C&C, 오라클 FLEXCUBE, Temenos 등) 백서가 제시하는 현대 코어뱅킹 참조 아키텍처.
3. **표준 연동 인터페이스 (행 밖과의 접점이지만 표준화된 채널)**:
   - **오픈뱅킹 공동업무** (금융결제원 오픈뱅킹공동망): 제공 API(잔액조회·거래내역·계좌실명·출금이체·입금이체 등), 참가 자격, 인증·전문 규격의 공개 범위.
   - **금융결제원(KFTC)** 공동망·표준 전문(예: 펌뱅킹/CMS, 계좌이체, 실시간 송금).
   - **마이데이터(본인신용정보관리업) API** 표준: 정보제공자·마이데이터사업자 간 표준 API, 금융보안원/마이데이터 표준 규격이 공개하는 인터페이스 항목.
   - 표준은 어디까지가 공개 규격이고 어디부터가 참가기관 비공개 협약 영역인가?
4. **EAI/ESB·API 게이트웨이 계층**: 은행 내부에서 채널·외부 핀테크를 계정계/정보계에 연결할 때 통상적으로 거치는 통합 미들웨어(EAI/ESB), 대외계(對外系) 게이트웨이, API 관리 계층의 일반 구조. 외부 솔루션이 붙는 **표준 진입점**은 보통 어디인가(대외계 게이트웨이 / API 게이트웨이 / 정보계 read 인터페이스)?
5. **우리 콘솔의 이식 인터페이스 지점**(핵심): 위 구조를 전제로, ① 정보계(분석·DW)에서 위험신호를 read-only로 받는 지점, ② RM 채널(창구단말·RM 워크벤치·내부 포털)에 콘솔 화면을 끼우는 지점, ③ 승인된 행동만 계정계/채널계에 write하는 지점(예: 알림 발송, 케이스 상태 갱신) — 이 3개 이식 지점이 한국 은행 아키텍처상 **어디에 매핑되는지**, 그리고 PII 비반출 전제에서 토큰화/가명처리 경계를 어디에 두는 것이 자연스러운지.
6. **지방은행 특이사항(있으면)**: 전북은행·광주은행 같은 지방은행이 시중은행과 다른 시스템 구성/공동 인프라(예: 지방은행 공동 IT, 코어뱅킹 공동화) 사용 여부에 대한 **공개된** 일반 사실. (특정 은행 비공개 내부망 구성은 추정하지 말 것.)
7. (덤 유도) 위 외에, 외부 AI Agent 콘솔이 은행 시스템에 붙을 때 한국 현장에서 **현실적 마찰점**(배치 윈도우, 계정계 변경 동결, 대외계 인증, 망 경계)으로 자주 거론되는 것이 공개 자료/사례에 있으면 정리하라.

### 출처 규칙 (반드시 준수)
- ① 발견마다 **URL + 원문 짧은 인용 + 발행처·날짜 + 신뢰도(1차/2차/[미검증])** 를 붙인다.
- ② 핵심 사실(특히 오픈뱅킹/마이데이터 표준 API 항목, 코어뱅킹 구조)은 **2개 출처로 교차**한다.
- ③ 링크에 실제 접속·확인하지 못하면 인용하지 말 것. 미접속 추정은 [미검증].
- ④ 한국 맥락·최신 우선(가능하면 2022년 이후). 글로벌 일반론은 한국 적용 가능성을 명시.
- ⑤ 본 프로젝트 정본(_canon.md)과 충돌하면 정본 우선(SSOT) — 단, 이 도메인은 외부 사실 수집이 주목적이다.
- **도메인 가드(저위험·이 도메인 전용)**: **공개 표준·벤더 백서·금융결제원/금융보안원 공식 규격·일반 참조 아키텍처로만** 한정한다. **특정 은행(전북은행 등)의 비공개 내부망 실구성·서버 배치·실 IP/대역은 추정·기술 금지.** 그런 항목은 "공개되지 않음"으로 명시하고 일반 표준으로 대체하라. **보안 아키텍처·망분리 기술·위협 방어는 이 리서치 범위 밖(다른 문서가 담당)** 이므로 다루지 말 것 — 단, 연동 표준이 보안 토큰/인증을 "요구한다"는 사실 언급은 허용.

### 한국 맥락·기간
- 대상: 한국 시중·지방은행, 금융결제원·금융보안원 표준. 지역 거점 지방은행(전북·광주) 관점 우대. 2022년 이후 현대화 동향 우선.

### 출력 형식 (반드시 이 구조)
1. **핵심 발견** — 계정계/정보계/채널계 구조, 표준 연동 인터페이스, 이식 지점 요약(불릿).
2. **근거표** — | 주장 | 출처 URL | 발행처·날짜 | 신뢰도(1차/2차/[미검증]) | 원문 인용(짧게) |
3. **아키텍처 매핑** — 한국 은행 표준 계층(채널 / 대외계·게이트웨이 / 정보계 / 계정계 / EAI·ESB)별로, **우리 콘솔의 이식 인터페이스 지점**을 표로 매핑: | 은행 계층 | 표준 인터페이스 | 우리 콘솔 연결 방식(read/write/UI) | PII 경계(토큰화·가명 위치) | 근거 |
4. **제품 적용 시사점** — 본선 기능명세서 "시스템 구성도"에 그대로 쓸 수 있는 이식 서사 3~5개.
5. **갭 / [미검증]** — 공개되지 않아 확인 불가한 항목, 추정 회피한 지점.

### 딥리서치 깊이 지시
표준 규격 원문(금융결제원 오픈뱅킹·마이데이터 표준 API 명세 페이지, 금융보안원 가이드), 코어뱅킹 벤더 백서, 차세대 프로젝트 공개 사례를 폭넓게 검색하고 교차 확인하라. 표면적 정의 나열이 아니라 **"외부 AI 콘솔이 실제로 붙는 표준 지점이 어디인가"** 라는 이식 관점으로 구조화하라. 확인 불가한 내부 구성은 단정하지 말고 갭으로 남겨라.

## Opus 점검 로그 (2패스)
- **패스 1 (경계·환각·심사 적합성 적대적 비판 → 보완)**: 초안이 (a) D10 보안 영역(망분리·위협)으로 번질 위험, (b) 특정 은행 내부망을 추정해 환각을 부를 위험, (c) "표준 정의 나열"에 그쳐 심사 3.3/3.4 어필력이 약할 위험이 있었다. → 도메인 가드에 "특정 은행 내부망 추정 금지·보안은 범위 밖" 명문화, 질문 5번을 **이식 인터페이스 3지점(read/UI/write)** 으로 구체화, 출력 형식에 "아키텍처 매핑 표(은행 계층↔콘솔 연결)"를 강제해 곧장 기능명세서 시스템구성도로 전환되게 했다.
- **패스 2 (D7a 중복·실행가치 재비판 → 보완)**: 오픈뱅킹·마이데이터 API가 "외부에서 가져오는 데이터"인 D7a와 겹쳐 보일 수 있었다. → 경계 메모와 질문 3번에서 **여기서는 "표준 인터페이스 규격·연결 지점"으로만 다루고, 한도·라이선스·비용 같은 취득 조건은 D7a가 담당**함을 명시. 또한 지방은행(전북·광주) 공동 인프라 질문(6번)과 현장 마찰점(7번 덤)을 추가해, 일반론을 넘어 JB 맥락·실무 설득력을 확보했다. PII 비반출 전제를 매핑 표의 "PII 경계" 열로 끌어와 우리 차별점과 결속.
