# D17 · 계열사 데이터 공동이용·처리위탁·클린룸 운영모델

> ⚠️ 대외비 — JB LocalGuard OS 본선 리서치 프롬프트. 외부 공유 금지.

**도메인** 계열사 데이터 거버넌스·클린룸 운영 · **티어** ★★(P0) · **심사** 2(금융업무·고객가치)·3(AI Agent 구현)·5(확장·운영리스크) · **배치** 2 · **엔진** 🟠 GPT
**경계**: D5a=신용정보법·개인정보보호법 조문 자체 / D5b=망분리·금융클라우드·생성형 AI 허용 정책 / **D17=전북은행·광주은행·JB우리캐피탈 간 데이터 이동 가능/불가 경계와 운영모델**. 여기서는 "외부 LLM 비반출"과 "그룹 내 과잉공유 금지"를 동시에 방어한다.

## 딥리서치 프롬프트 (이 아래 통째로 붙여넣기)

당신은 **금융그룹 데이터 거버넌스·신용정보 처리·클린룸 운영모델 리서처**입니다. 제품은 **JB LocalGuard OS**입니다. 이 제품은 RM용 거버넌드 AI 콘솔로, 지역 금융 고객의 위험 신호를 Case로 만들고 여러 AI Agent가 판단·행동 초안·검증을 수행하되 고객 대상 행동은 사람 승인 전까지 차단합니다. 핵심 차별점은 **고객 원본 PII·신용정보를 외부 LLM으로 반출하지 않고**, 동시에 **그룹 내부 계열사 사이에서도 목적을 벗어난 과잉공유를 막는 것**입니다.

목표는 전북은행·광주은행·JB우리캐피탈을 중심으로, JB LocalGuard OS가 멀티도메인(소상공인 여신·전세 위험·보이스피싱/이상거래)으로 확장될 때 **어떤 데이터는 계열사 간 이동·공동이용·위탁·가명결합이 가능하고, 어떤 데이터는 금지 또는 별도 동의·절차가 필요한지**를 실무 운영모델로 정리하는 것입니다. 단순 법조문 요약은 D5a에 맡기고, 여기서는 **PoC/파일럿에서 실제로 어떤 데이터 경계와 클린룸·토큰화 구조로 운영해야 하는지**를 도출하세요.

### 핵심 질문

**A. 계열사 간 데이터 이동 가능/불가 경계**
- 전북은행·광주은행·JB우리캐피탈 사이에서 고객정보·개인신용정보·기업여신정보·거래정보·연체/상환정보·상담기록·사후관리 이벤트를 공유하거나 결합할 때 필요한 법적 근거와 제한은 무엇인가?
- 금융지주/계열회사 관계에서 **공동이용**, **제3자 제공**, **업무위탁/처리위탁**, **가명정보 결합**, **내부 통계·연구 목적 처리**가 어떻게 구분되는가?
- "같은 그룹이므로 자유롭게 공유 가능"이라는 주장을 반박할 수 있는 금지·제한·목적 제한·최소 필요 원칙은 무엇인가?

**B. 동의·고지·위탁·공동이용 체크리스트**
- 고객 동의가 필요한 경우와, 동의 없이 가능한 경우(예: 법령상 근거, 계약 이행, 가명처리 통계·연구 등)를 금융권 맥락에서 구분하라.
- 공동이용 동의서, 업무위탁 공시/고지, 제3자 제공 동의, 마케팅 동의, 신용정보 조회·집중·활용 동의가 각각 어떤 데이터 흐름에 대응하는지 표로 정리하라.
- PoC 단계에서 실명 고객정보를 쓰지 않기 위해 필요한 **동의 최소화·목적 제한·접근권한·처리기록** 요구사항을 정리하라.

**C. 클린룸·토큰화·가명결합 운영모델**
- JB LocalGuard OS가 외부 LLM을 활용하더라도 원본 PII·신용정보를 외부로 보내지 않기 위한 구조를 제안하라: 내부 클린룸, 토큰화/가명키, 원본-토큰 매핑키 분리보관, 반출 스캔, 프롬프트 프록시, 감사로그, 승인 게이트.
- 계열사별 원천 시스템은 분리하고, 공통 Case/AgentRun/Evidence/Audit에는 어떤 최소 필드만 올릴 수 있는지 설계하라.
- 지정 결합전문기관 또는 데이터전문기관을 통한 가명정보 결합이 필요한 경우와, 그룹 내부 통계/연구용 가명처리로 충분한 경우를 구분하라.
- 클린룸에서 허용되는 분석 결과의 반출 기준(집계치, 위험 스코어, 설명근거, 샘플 수 기준, 재식별 위험 점검)을 정리하라.

**D. PoC/파일럿 허용 데이터 범위**
- 본선 MVP·PoC에서 안전하게 사용할 수 있는 데이터셋을 등급별로 제안하라: 합성 데이터, 공개 데이터, 익명/집계 데이터, 가명 데이터, 실명 데이터.
- 소상공인 여신·전세 위험·보이스피싱/이상거래 각각에서 필요한 최소 데이터 필드와 금지 필드를 구분하라.
- RM 콘솔이 "실제 운영 가능"해 보이려면 어떤 데이터는 내부망/계열사 내부에서만 처리하고, 어떤 요약·근거만 외부 LLM/RAG로 보낼 수 있는가?

**E. 덤 발견 유도**
- 위 질문 밖이라도 금융그룹 데이터 공동활용, 신용정보 데이터 결합, 데이터 클린룸, 연합학습, 개인정보보호 강화기술(PET), 토큰화, 내부통제 사례 중 JB LocalGuard OS의 심사 2·3·5번 방어에 도움이 되는 근거를 찾아라.
- 국내 금융권 사례를 우선하되, 국내 자료가 부족하면 해외 금융권 클린룸·데이터 협업 사례를 보조 근거로 제시하고 한국 적용 한계를 명시하라.

### 출처 규칙
1. 발견마다 **URL + 원문 짧은 인용 + 발행처·날짜 + 신뢰도(1차/2차/[미검증])**를 반드시 붙인다.
2. 핵심 수치·제도 적용 주장·사례는 가능하면 **2개 이상 출처로 교차확인**한다. 단일 출처이면 `[미검증]` 또는 `[2차]`로 낮춰 표기한다.
3. 링크를 실제로 접속하지 못했거나 원문을 확인하지 못한 경우 **인용하지 않는다**.
4. 한국 금융 맥락과 최신 자료를 우선한다. 법령·가이드는 law.go.kr, 개인정보보호위원회, 금융위원회, 금융감독원, 한국신용정보원, 금융보안원 등 1차 출처를 우선한다.
5. 신용정보법이 금융회사 신용정보 처리에서 특별법으로 우선한다는 점을 확인하고, 개인정보보호법은 보충 근거로 다룬다.
6. `_canon.md`와 충돌하면 `_canon.md`가 우선한다. **핀크(Finnq)를 JB 계열로 표기하지 말 것.** 네이버클라우드 MOU는 **공표된 추진방향**으로만 표현하고, 이미 운영 중이라고 쓰지 말 것. 그룹 총자산은 `[미확정]`으로 다루고 새 수치를 단정하지 말 것.
7. D5a/D5b와 중복을 피한다. 조문 자체의 장문 해설이나 망분리 정책 일반론은 최소화하고, **계열사 데이터 경계·위탁·공동이용·클린룸 운영 설계**에 집중한다.

### 출력 형식
1. **핵심 발견** — 계열사 데이터 경계 / 동의·위탁·공동이용 / 클린룸·토큰화 / PoC 허용범위 4축으로 요약.
2. **근거표** — `주장 | 출처 URL | 발행처·날짜 | 신뢰도 | 원문 인용`.
3. **제품 적용 시사점** — JB LocalGuard OS에 적용할 데이터 경계도, 권한·승인·감사 요구사항, 외부 LLM 비반출 설계, 그룹 내 과잉공유 방지 설계.
4. **운영 체크리스트** — `데이터 흐름 | 가능/조건부/금지 | 필요한 동의·위탁·고지 | 처리 위치 | 반출 가능 산출물 | 감사증거`.
5. **갭·[미검증]** — 공개자료로 확인 불가한 항목, 법무 확인 필요 항목, PoC 전에 확인해야 할 질문.

### 깊이·범위 지시
ChatGPT Pro Deep Research로 수행한다. 국내 금융 법령·감독 가이드·신용정보 데이터 결합 제도·금융권 클린룸/PET 사례까지 깊게 조사한다. 최소 12개 독립 출처를 목표로 하되, 출처 수보다 **원문 확인·신뢰도 라벨·제품 적용성**을 우선한다. 결론은 "공유 가능/불가" 이분법이 아니라, **목적·동의·처리위치·가명화 수준·반출 산출물·감사증거** 조건으로 제시하라.
