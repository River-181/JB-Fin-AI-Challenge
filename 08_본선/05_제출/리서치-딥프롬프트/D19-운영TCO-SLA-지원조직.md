# D19 · 운영 TCO·SLA·지원조직·장애 대응모델

> ⚠️ 대외비 — 본선 제출 전 내부 리서치 프롬프트.

**도메인** 운영 TCO·SLA·지원조직·장애 대응모델 · **티어** ★(P1) · **심사** 4(MVP 완성도·검증 가능성)·5(확장성·운영리스크) · **배치** 3 · **엔진** 🟠 GPT  
**경계 메모**: D9=레퍼런스 아키텍처·LLMOps 기술스택 / D10=보안 위협·PII 유출 차단 기술 / D+a=시장·가격·ROI 상위 논리. **D19는 "수억 구매안이 운영 가능한 서비스 계약인가"** 를 입증한다. 외부 LLM 호출비, 프록시/DLP/로그 저장, 감사원장 보관, 운영인력, 12개월 TCO, SLA, RTO/RPO, 모델 버전 통보, 장애 등급·복구·지원조직만 다룬다.

## 딥리서치 프롬프트 (이 아래 통째로 붙여넣기)

당신은 **금융권 B2B SaaS/AI 서비스 운영·SLA·TCO 리서처**입니다. 제품은 **JB LocalGuard OS** — RM용 거버넌드 AI 콘솔입니다. 핵심 설계는 **PII 비반출 × 승인우선 × 멀티도메인(소상공인·전세·사기)** 입니다. 이 리서치의 목표는 기술 아이디어가 아니라, JB금융그룹 같은 금융사가 **수억 원을 지불해도 되는 "운영 가능한 서비스 계약"** 으로 보이게 만들 근거를 찾는 것입니다.

핵심 질문은 하나입니다. **"JB LocalGuard OS를 PoC에서 파일럿·운영 서비스로 전환할 때, 12개월 동안 얼마가 들고, 어떤 SLA·지원조직·장애 대응 책임을 계약서에 넣어야 구매자가 안심하는가?"**

### 알아내야 할 것

**A. 12개월 운영 TCO 구성**
1. 금융권 AI/LLM 애플리케이션 운영 TCO의 비용 항목을 분해하라: 외부 LLM/API 호출비, 임베딩·RAG 인덱스, 프록시/API 게이트웨이, DLP·PII 스캔, 로그·감사원장 저장, 관찰성(APM/log/trace), 보안 모니터링, 백업·DR, 취약점 점검, 운영자 콘솔, 정기 리포팅.
2. 각 항목의 가격 단위와 산식 예시를 찾아라: `월 사용자 수`, `RM 수`, `케이스 수`, `LLM token`, `문서 페이지`, `로그 GB/day`, `보관 개월`, `감사 조회 건수`, `온콜 시간`, `SLA 등급` 등.
3. 공개 단가가 있으면 출처와 함께 정리하고, 공개 단가가 없으면 **추정 산식만** 제시하라. 단일 벤더 가격은 [미검증] 또는 [2차]로 낮추고 2출처 교차가 없으면 핵심 숫자로 쓰지 말라.
4. "라이선스 가격"과 "운영 가능 서비스 가격"의 차이를 설명하라. 특히 금융권에서는 보안점검·감사대응·장애대응·로그보관·지원조직 비용이 왜 별도 원가가 되는지 근거를 찾아라.

**B. SLA·RTO/RPO·지원 범위**
5. 금융권 B2B SaaS/AI 서비스 계약에서 자주 쓰는 SLA 항목을 수집하라: 가용성, 응답시간, 장애 접수·초동응답, 복구목표시간(RTO), 복구목표시점(RPO), 정기점검, 유지보수 공지, 보안패치, 데이터 백업, 로그 보존, 리포트 제공.
6. PoC·파일럿·운영 단계별 SLA 차이를 정리하라. PoC는 best-effort로 충분한가, 파일럿부터 어떤 최소 SLA가 필요한가, 정식 운영은 어떤 수치와 책임분계가 필요한가.
7. RM용 내부 콘솔 특성상 **고객 접점 자동실행은 승인 전 차단**되지만, 위험 알림·사후관리 워크플로 지연은 업무 리스크가 된다. 이 특성에 맞는 SLA 우선순위를 제안하라.
8. RTO/RPO는 "코어뱅킹" 수준으로 과장하지 말고, **내부 RM 의사결정 보조 콘솔**로서 적정 수준의 후보 범위를 근거 기반으로 제시하라. 확정 수치가 어려우면 보수/기준/강화 3단계로 제안하라.

**C. 지원조직·운영 역할표**
9. 운영에 필요한 역할을 분해하라: 서비스 매니저, L1 헬프데스크, L2 애플리케이션 운영, L3 AI/LLMOps 엔지니어, 보안/DLP 담당, 데이터 거버넌스 담당, 준법·감사 대응, 고객 성공/교육, 온콜 책임자.
10. 각 역할이 주당/월간 수행해야 하는 운영 작업을 찾아라: 모델·프롬프트 버전 관리, 장애분석, 비용 모니터링, 로그 샘플링, 권한검토, 승인정책 변경관리, 릴리즈 노트, 사용자 교육, 월간 운영 리포트.
11. "작은 스타트업 팀이 금융사 파일럿을 운영할 때" 현실적인 최소 운영조직과, 정식 운영 전환 시 필요한 증원·아웃소싱·파트너 조합을 제안하라.

**D. 장애 등급·복구·커뮤니케이션 모델**
12. 금융 AI 콘솔에 맞는 장애 등급을 정의하라: 예) P1 서비스 전체 중단, P2 특정 도메인/케이스 처리 장애, P3 응답 지연·부분 오류, P4 표시·리포트 오류, 보안/PII 의심 이벤트 별도 최고등급.
13. 등급별 초동응답·상황공유·우회절차·복구목표·사후보고(RCA) 기준을 조사하라. 금융권/클라우드/SaaS의 공개 SLA·incident management 문서를 우선 참고하라.
14. 외부 LLM 장애·모델 품질 저하·응답 지연·가격 급등·모델 폐기/deprecation·버전 변경이 발생했을 때 고객에게 어떤 통보와 대체 경로가 필요한지 정리하라.
15. "모델 버전 변경 통보"와 "프롬프트/정책 변경 승인"의 운영 절차를 제안하라. 누가 승인하고, 언제 통보하고, 어떤 로그를 남겨야 하는가.

**E. 규제·감사·보관 비용**
16. 감사원장·판단근거·승인 로그·운영 로그의 보관기간과 비용 요인을 조사하라. 법정 보관기간이 명확한 영역과, 내부통제/감사 편의상 설정하는 영역을 구분하라.
17. 금융 클라우드·망분리 개선·AI 활용 환경에서 운영계약에 들어가야 할 감사권, 보안점검권, 위탁/재위탁 통지, 데이터 반출 금지, 침해사고 통보 조항을 찾아라.
18. PII 비반출 제품이라도 **토큰화 키 관리, 로그 마스킹, 감사조회 권한, 보관·파기 정책**이 비용과 책임으로 남는다는 점을 근거화하라.

**F. 범용 덤 발견**
19. 위 질문 외에 구매자가 "이 정도면 수억 원 계약이 운영 가능한 서비스다"라고 판단하게 만드는 근거를 찾아라: 표준 계약 조항, 운영 리포트 샘플, SLA 크레딧, 보안 서약, 월간 QBR, 교육·전환관리, 종료/반출/삭제 절차.
20. 반대로 "좋은 데모지만 운영계약으로는 부족하다"는 실패 신호를 찾아라: SLA 부재, 온콜 없음, 로그 보관비 누락, 외부 API 장애 대책 없음, 모델 변경 통보 없음, 단일 담당자 의존.

## Opus 점검 로그 (2패스)

### 패스 1 — 적대적 자기검토 → 보완
- **비판 ① D9·D10·D+a와 중복될 위험.** TCO를 묻는 순간 아키텍처 상세, 보안통제, ROI 시장논리로 흘러 기존 도메인과 겹친다.
  - **보완**: 상단 경계메모와 출처규칙 8번에 "운영비·계약·SLA·지원·장애대응으로 환산"을 고정했다. 질문도 프록시/DLP/로그저장 자체 기술이 아니라 가격 단위·보관·운영책임·장애대응으로 묶었다.
- **비판 ② 수억 구매안을 가격표 몇 개로 방어하려는 약점.** 공개 API 단가만 모으면 "운영 가능한 서비스 계약"이 아니라 단순 원가표가 된다.
  - **보완**: 12개월 TCO 외에 SLA 초안, 지원조직 RACI, 장애 등급·복구 플레이북, 감사·보관 조항을 출력 형식으로 강제했다. 라이선스와 운영 서비스 가격의 차이를 별도 질문으로 추가했다.
- **비판 ③ RTO/RPO 과장 위험.** 은행 코어시스템 기준을 그대로 끌어오면 RM 보조 콘솔의 현실성과 비용감이 깨진다.
  - **보완**: "내부 RM 의사결정 보조 콘솔"로 적정 수준을 보수/기준/강화 3단계로 제시하라고 제한했다.

### 패스 2 — 재비판 → 보완
- **재비판 ① 외부 LLM 운영 리스크가 빠질 수 있음.** 일반 SaaS SLA만 보면 모델 deprecation, 버전 변경, 호출비 급등, 외부 API 장애 같은 AI 특유 리스크가 누락된다.
  - **보완**: 질문 14~15와 장애 플레이북 필수 항목에 외부 LLM 장애·모델 품질 저하·가격 급등·모델 폐기/deprecation·버전 변경 통보를 넣었다.
- **재비판 ② PII 비반출이면 보안운영 비용이 작다고 오해할 수 있음.** 원본 PII를 안 보내도 토큰화 키, 로그 마스킹, 감사조회 권한, 보관·파기가 남는다.
  - **보완**: 질문 16~18에 감사원장·승인로그·토큰화 키 관리·로그 마스킹·파기 정책의 비용과 책임을 명시했다.
- **재비판 ③ JB 사실 제약 누락.** 운영 도메인이라도 JB 계열·MOU·총자산 표현을 잘못 쓰면 전체 정합성이 깨진다.
  - **보완**: 출처규칙 7번에 `_canon.md` 우선, 핀크 JB계열 표기금지, MOU는 추진방향만, 총자산은 [미확정] 처리 규칙을 넣었다.
