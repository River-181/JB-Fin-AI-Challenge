# D9 — 구현워크플로 · 레퍼런스아키텍처

## 메타 (프롬프트 밖 — 생성·운영용)
- **도메인**: 금융권 AI 시스템 구현의 **외부 리서치 가능 부분** — 레퍼런스 아키텍처, LLMOps/MLOps, RAG·Rule Engine·멀티에이전트 오케스트레이션 패턴, 기술스택 벤치마크·사례
- **티어**: ★ (외부 가치 제한적 — 풀 빠듯하면 Claude 내부 작성 우선, 외부는 보강용)
- **심사 항목**: 3(AI Agent 설계·기술 구현 가능성 — 아키텍처·기술접근의 현실성) · 4(MVP 완성도 — 구현 기반 충분성)
- **실행 배치**: 배치 3 (종합·차별 — 배치1·2 인용)
- **경계 메모**:
  - **D9 ↔ D4**: *코드·도구·기술스택·아키텍처 = 여기(D9) / 사람·직무·조직·권한 = D4.* 직무·협업·승인권한은 D4 담당 — 침범 금지.
  - **내부결정 제외**: 우리가 확정할 스택(우리 제품의 채택 결정)은 본 리서치 대상이 아니다. **외부 사례·패턴·레퍼런스 아키텍처만** 수집한다. "우리는 X를 쓴다"가 아니라 "업계는 X 패턴을 이렇게 쓴다".
  - **D9 ↔ D10**: 일반 구현 아키텍처·오케스트레이션 = D9 / 보안 위협·인젝션·데이터유출·모델 거버넌스·망분리 기술구현 = D10. 보안은 D10으로 양보.
  - **D9 ↔ D13**: 구현 패턴·아키텍처 = D9 / 평가지표·검증·파일럿 KPI = D13.
- **제품 연결**: JB LocalGuard OS의 본선 목표(정적 함수 계약 → 서버 API 승격, RAG·Rule Engine·멀티에이전트 오케스트레이션)가 **업계 레퍼런스 아키텍처와 정합**함을 입증하기 위한 외부 패턴 베이스. "현실적으로 구현 가능"(심사 3.4) 방어.

## 딥리서치 프롬프트 (이 아래 통째로 붙여넣기)

당신은 **금융·엔터프라이즈 AI 시스템의 구현 아키텍처와 운영(LLMOps/MLOps)을 정리하는 기술 아키텍처 리서처**다. 목표는 "RM용 거버넌드 AI Agent 콘솔" 같은 시스템을 만들 때, **업계가 실제로 채택하는 레퍼런스 아키텍처·구현 패턴·기술스택·운영 방식**을 외부 공개 자료(레퍼런스 아키텍처 문서, 사례연구, 오픈소스 프레임워크 문서, 기술 백서)에 근거해 정리하는 것이다.

**스코프 경계(엄수)**: (1) **코드·도구·기술스택·아키텍처 패턴만** 다룬다 — 사람·직무·조직·권한은 다루지 않는다(다른 도메인 담당). (2) **우리가 무엇을 채택할지(내부 결정)는 다루지 않는다** — "업계가 어떤 패턴을 어떻게 쓰는가"의 외부 사례·레퍼런스만 정리한다. (3) **보안 위협·데이터 유출·프롬프트 인젝션·모델 거버넌스·망분리 기술구현은 별도 도메인(D10) 담당** — 여기서는 일반 구현 아키텍처·오케스트레이션에 집중하고 보안 심화는 양보한다.

### 알아내야 할 것 (핵심 질문)

**A. 엔터프라이즈/금융 AI 레퍼런스 아키텍처**
1. 클라우드 벤더·프레임워크가 공개한 **생성형 AI/에이전트 애플리케이션 레퍼런스 아키텍처**의 공통 구성요소: 데이터 인입·전처리 → 임베딩/인덱싱 → 검색(RAG) → LLM 추론(모델 라우팅) → 후처리·검증 → 액션/도구호출 → 로깅·감사. 각 레이어가 무엇을 책임지는가.
2. **금융권 특유의 아키텍처 고려사항**(보안 심화 제외): 온프레미스/프라이빗 클라우드 배치, 모델 라우팅(외부 프런티어 LLM vs 내부 모델 혼용), 토큰화/마스킹 전처리 단계의 *아키텍처상 위치*(상세 보안기법은 D10), 감사·추적성(audit trail) 설계. 금융사가 공개한 AI 도입 사례의 아키텍처 패턴.

**B. RAG (검색증강생성) 구현 패턴**
3. **RAG 파이프라인의 표준 구성과 변형**: 청킹 전략, 임베딩·벡터DB 선택지, 하이브리드 검색(키워드+벡터), 리랭킹, 컨텍스트 구성. **공공/내부 문서(규정·법령·심사자료) 위 RAG**의 알려진 패턴·함정.
4. **RAG 품질·정확도(faithfulness/groundedness)를 높이는 구현 기법**: 출처 인용(citation) 강제, grounding 검증, 환각 억제 패턴(다만 이를 "측정"하는 지표는 D8/D13 담당 — 여기선 *구현 기법*만).

**C. Rule Engine + AI 결합 (하이브리드 의사결정)**
5. **결정형 규칙 엔진(Rule Engine)과 LLM/ML을 결합하는 아키텍처 패턴**: 어디까지 결정형 규칙으로 게이팅하고 어디서 LLM 판단을 쓰는가, 승인·차단 같은 **결정적 게이트를 규칙으로 처리**하는 사례(금융 의사결정·여신·컴플라이언스 분야의 rule+ML 하이브리드). 규칙 우선(rule-first) vs 모델 우선 패턴.

**D. 멀티에이전트 오케스트레이션**
6. **멀티에이전트 시스템 오케스트레이션의 대표 패턴**: 오케스트레이터-워커, 역할 분담(specialist agents), 계획-실행-검증 루프, 핸드오프·도구 사용, human-in-the-loop 승인 게이트의 *구현* 방식. 대표 오픈소스/프레임워크가 제시하는 오케스트레이션 패턴(프레임워크별 접근 차이).
7. **에이전트 신뢰성·거버넌스 구현 패턴**: 도구 호출 권한 제한, 실행 전 승인(approval gate) 구현, 추적성(각 판단의 근거·도구·입출력 로깅), 실패·재시도·가드레일. "고객 대상 행동은 사람 승인 전 차단" 같은 통제를 *기술적으로* 어떻게 구현하나.

**E. LLMOps/MLOps 운영**
8. **LLMOps/MLOps 운영 패턴**: 모델·프롬프트 버전관리, 평가·회귀 테스트 파이프라인, 모니터링·관측(observability), 비용·지연 관리, 모델 라우팅/폴백, 배포(블루그린·카나리)·롤백. 엔터프라이즈가 권장하는 운영 라이프사이클.
9. **기술스택 벤치마크·선택지 비교**: 오케스트레이션 프레임워크, 벡터DB, 평가/관측 도구, 게이트웨이 등 각 카테고리의 **대표 선택지와 트레이드오프**(성숙도·라이선스·금융권 적합성). *권고가 아니라 비교 사실*로 — 우리가 무엇을 고를지는 우리가 결정.

**F. 사례·덤**
10. *(범용 덤 유도)* 위 외에, **금융사가 실제로 공개한 AI 시스템 구현 사례**(아키텍처 다이어그램·기술선택·교훈)나, "정적 함수 계약을 서버 API로 승격" 같은 **점진적 구현 경로(MVP→프로덕션 마이그레이션) 패턴**이 있으면 추가 수집하라.

### 한국 맥락·기간
- **한국 금융권 클라우드/AI 도입 제약**(망분리·금융클라우드 환경에서의 아키텍처 적용)을 고려한 패턴을 우선하되, 정책·법 근거 자체는 D5b/D10 담당 — 여기선 "그 환경에서 쓰이는 아키텍처 패턴"만.
- 기간: 생성형 AI 아키텍처·LLMOps는 **2024~2026 우선**(빠른 변화 영역). 도구·버전 명시 필수.

### 출력 형식 (이 구조로 답하라)
1. **핵심 발견 요약** — A(레퍼런스 아키텍처)·B(RAG)·C(Rule+AI)·D(멀티에이전트)·E(LLMOps)·F(사례·덤) 블록별 서술.
2. **레퍼런스 아키텍처 표** — | 레이어/구성요소 | 책임 | 대표 패턴/도구 | 출처 |
3. **기술스택 비교 표** — | 카테고리(오케스트레이션/벡터DB/관측 등) | 대표 선택지 | 트레이드오프 | 금융권 적합성 | 출처 | (권고 아닌 비교 사실)
4. **근거표** — | 주장(패턴/사례) | 출처 URL | 발행처·버전·날짜 | 신뢰도 | 원문 짧은 인용 |
5. **제품 적용 시사점** — 이 패턴들이 JB LocalGuard OS의 본선 목표(정적 함수 계약→서버 API 승격, RAG·Rule Engine·멀티에이전트 오케스트레이션, 승인 게이트·감사 추적)와 어떻게 정합하며 "구현 가능"(심사 3.4)을 뒷받침하는가 (4~7개). **단, 최종 스택 채택은 우리 결정 — 여기서는 "정합하는 외부 패턴" 제시까지만.**
6. **갭 / [미검증]** — 금융권 공개 사례의 희소성, 버전 불확실, 보안/평가(D10·D13)로 넘긴 항목, 확정 못 한 부분.

### 딥리서치 깊이 지시 (Gemini / Perplexity Deep Research)
- 단순 도구 나열이 아니라 **"왜 이 레이어/패턴이 필요하고 어떤 트레이드오프가 있는가"**까지 추적하라. 특히 RAG·Rule+AI·멀티에이전트 오케스트레이션의 *결합 지점*(어디서 결정형 게이트, 어디서 LLM 판단)을 명확히.
- **버전·날짜를 반드시 명시**하라(프레임워크는 빨리 바뀜). deprecated 패턴을 최신처럼 인용하지 말 것.
- **벤더 추천을 "권고"가 아니라 "비교 사실"로** 정리하라 — 우리가 무엇을 채택할지는 외부 리서치 범위 밖이다.
- 보안 심화(인젝션·유출·망분리 구현)와 평가지표(KPI·벤치마크)는 다른 도메인이니 **깊이 들어가지 말고 경계에서 멈춰** "D10/D13 참조"로 표기하라.
- 제출 전 자체 점검: "이 패턴들이 실제 금융권 제약에서 구현 가능한가 / 버전·출처가 최신인가 / 사람·보안·평가 도메인을 침범하지 않았는가."

## Opus 점검 로그 (2패스)

### 패스 1 — 적대적 비판 → 보완
- **비판 ① 경계 3중 누수(D4·D10·D13).** 구현을 다루면 자연히 직무(D4)·보안(D10)·평가(D13)로 흘러 4개 도메인이 중복된다.
  - **보완**: 스코프 경계문·도메인 가드·깊이지시에 "사람=D4, 보안위협·망분리=D10, 평가=D13 침범 금지, 경계에서 멈추고 참조 표기"를 반복 명시. RAG 질문(B4)도 "측정은 D8/D13, 여기선 구현 기법만"으로 한정.
- **비판 ② 내부결정 오염.** "우리가 무엇을 쓸지"로 흘러 외부 리서치가 자가추천이 되면 가치 하락·환각.
  - **보완**: 역할문·경계·기술스택 표·시사점·깊이지시 5곳에 "우리 채택 결정 아님, 업계 외부 패턴·비교 사실만"을 박음. 출력형식 3을 "권고 아닌 비교"로 명시.
- **비판 ③ 버전 환각.** LLMOps·프레임워크는 분기마다 바뀌어 구버전·폐기 패턴을 최신처럼 인용하기 쉬움.
  - **보완**: 도메인 가드에 "버전·발행일 명시·deprecated 구분" 규칙, 출력 근거표에 "버전" 컬럼 추가, 깊이지시에 "deprecated를 최신처럼 인용 금지" 명시.

### 패스 2 — 재비판 → 보완
- **재비판 ① 제품 정합 연결 약하면 ★티어 가치마저 소멸.** 외부 패턴만 모으고 우리 본선 목표(함수계약→API 승격 등)와 안 이으면 심사 3.4 기여 미약.
  - **보완**: 제품 연결 메타·C5(Rule+AI 게이트)·D7(승인 전 차단 구현)·F10(MVP→프로덕션 마이그레이션 패턴)을 _canon의 본선 목표·운영계약(승인 게이트·감사)과 직결. 출력형식 5에 정합 매핑 의무화.
- **재비판 ② 벤더 마케팅 표류.** 구현 도메인은 벤더 블로그가 풍부해 성능 과장에 휩쓸리기 쉬움.
  - **보완**: 도메인 가드에 "벤더 추천=비교 사실로만·[2차], 성능 과장=[미검증]" 규칙. 1차(공식 레퍼런스·동료평가)와 2차(블로그·발표)를 신뢰도 컬럼으로 분리.
- **재비판 ③ 핵심 차별(Rule+AI·승인 게이트 구현)이 일반론에 묻힐 위험.** 우리 제품의 코어(승인우선 루프)를 구현 관점에서 정당화하는 부분이 약하면 손해.
  - **보완**: C5(rule-first 게이팅)·D7(approval gate·도구권한 제한·추적성 구현)을 별도 질문으로 강조하고, 시사점에 "결정적 게이트를 규칙으로, 사람 승인 전 차단을 기술적으로 구현하는 외부 패턴"을 명시 요구.
