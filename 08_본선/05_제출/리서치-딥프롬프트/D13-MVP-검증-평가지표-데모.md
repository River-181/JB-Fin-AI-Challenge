# D13 — MVP · 검증 · 평가지표 · 데모

## 딥리서치 프롬프트 (이 아래 통째로 붙여넣기)

**역할**: 당신은 **AI 제품의 MVP를 "검증 가능한 형태로 증명"하는 제품 평가·실험설계 전문 리서처**다. 목표는 핀테크/AI-Agent MVP가 심사·투자·파일럿 검증을 통과하기 위해 **(1) 어떤 평가지표로 (2) 어떤 데모 시나리오 성공기준으로 (3) 어떤 파일럿 KPI·표본·기간·통계 설계로 "실제 동작하고 효과가 있다"를 증명하는가**를, 실제 학술·산업·공모전 선례에서 구체적으로 끌어내는 것이다.

**이 리서치가 답해야 할 단 하나의 질문(반드시 정조준)**: **"발표장에서 심사위원에게 보여줄 3개 데모 시나리오(소상공인 상환위험 분류 / 전세사기 위험 차단 / 보이스피싱 이상거래 차단)의 성공기준과 실측 지표는 무엇이며, 그것이 '실제 동작 가능한 MVP로 검증 가능'을 어떻게 입증하는가?"**

**중대 전제**:
- 이 리서치는 **우리 제품의 KPI를 검증·측정하는 *방법*과 *선례*** 를 찾는 것이지, AI 생산성 일반론의 효과 크기를 재수집하는 게 아니다(그건 별도 트랙). 모든 발견은 "우리 데모/파일럿에 적용 가능한 측정 설계"로 환산하라.
- 우리 제품엔 **고정 목표 KPI**가 있다: Triage 시간 50% 단축 · 근거 추적성 100% · 승인 안전 100% · 사후관리 누락 0건 · 전세 안전계약 연결 100%. 이 값들을 **어떻게 측정·정의·검증**해야 신뢰받는지가 핵심이다. **수치를 부풀리지 말고, "이 KPI를 이렇게 측정한다"는 검증 가능한 방법**을 찾아라.

### 알아내야 할 것 (핵심 질문 — 1~8 필수, 9~11 덤 발견 유도)

**A. AI Agent 제품 평가지표 — 무엇을·어떻게 재나**
1. **AI Agent/LLM 애플리케이션의 표준 평가지표**: 과업 성공률(task success rate), 정확도(accuracy/precision/recall), **근거연결률·인용 정확도(citation/attribution accuracy, groundedness)**, **환각율(hallucination rate)·사실충실성(faithfulness)**, 도구 호출 정확도, end-to-end latency. 각 지표의 정의·산식·측정법과, **에이전트형(다단계·도구사용)** 평가의 추가 고려사항(궤적 평가 trajectory eval 등).
2. **"승인우선 루프"의 검증 지표**: human-in-the-loop 시스템에서 측정하는 지표 — 승인 게이트 통과율, **승인 루프로 인한 지연(approval latency)**, 자동 차단 정확도(차단해야 할 건 차단·정상 건 오차단율 false block rate), 인간 검토 부담(검토 시간·overrides). 우리의 "승인 전 고객행동 차단" 설계를 정량 검증할 지표.
3. **"PII 누출 0건"을 검증하는 방법**: 데이터 유출 0건을 **어떻게 측정·증명**하는가 — 합성 PII 주입 테스트(canary/honeytoken), 반출 트래픽 감사, 레드팀 유출 시도 성공률, de-identification 정확도(재식별 위험 측정). "0건"이라는 절대값을 신뢰 가능하게 만드는 검증 프로토콜.
4. **AI Agent 평가 프레임워크·벤치마크·도구**: 산업·학술의 에이전트 평가 프레임워크(예: LLM-as-a-judge의 신뢰성과 한계, 평가 자동화 도구, 표준 벤치마크). LLM-as-judge를 평가에 쓸 때의 **편향·신뢰성 한계**와 보정법을 반드시 포함.

**B. 데모 시나리오 성공기준 설계 — 발표장에서 무엇을 보여주나**
5. **데모/PoC 시나리오의 성공기준(acceptance criteria) 설계법**: 짧은 발표 데모에서 "동작한다"를 설득력 있게 보이는 시나리오 구성 원칙 — 골든패스 deterministic 시연, 사전·사후 비교(Before/After), 실패 케이스 의도적 노출(차단이 작동함을 보임), 측정 가능한 성공 판정 기준. **AI/핀테크 제품의 좋은 데모 시나리오 vs 나쁜(체리피킹·과장) 데모를 가르는 기준.**
6. 위를 우리 **3개 시나리오 각각**에 적용해 **성공기준·실측 지표·검증 방법**을 설계할 근거를 수집하라:
   - **① 소상공인 상환위험 분류**(전주 카페, riskScore·운전자금·매출둔화 → 정책금융 매칭 초안): 위험분류 정확도, 근거연결률, RM 검토시간 단축 측정법.
   - **② 전세사기 위험 차단**(등기 권리관계·전세가율·보증보험 체크리스트): 위험탐지 적중·놓침(재현율), 안전계약 연결 완결성 측정법.
   - **③ 보이스피싱 이상거래 차단**(이상거래 탐지→승인 전 차단): 차단 정확도·오차단율, 차단 지연시간 측정법.
   각 시나리오에서 **무엇을 측정해 "성공"이라 부를지**의 정량 기준을 선례 기반으로 제시하라.

**C. 파일럿 KPI·통계 설계 — 효과를 어떻게 입증하나**
7. **핀테크/금융 AI PoC·파일럿의 KPI·실험 설계**: 실제 금융권 AI 파일럿이 채택한 KPI(처리시간 단축·정확도·전환율·리스크 감소 등)와, **표본 수·기간·대조군·통계적 유의성** 설계 관행. 소표본 PoC에서 효과를 신뢰 가능하게 주장하려면 최소 표본·기간·비교설계가 어때야 하는가. A/B·전후비교의 함정(학습효과·선택편향).
8. **유사 공모전(핀테크·금융 AI·해커톤·정부 AI 공모) 수상작의 MVP 검증 방식**: 수상작들이 "동작하는 MVP"와 "검증 가능성"을 어떻게 입증했는가 — 데모 구성, 제시한 지표, PoC/파일럿 증거, 발표 전략. 평가기준에서 "MVP 완성도·검증 가능성"이 어떻게 채점됐는지의 패턴. (가능하면 국내 금융·핀테크 공모전.)

**D. 범용 덤 유도**
9. *(덤)* **재현 가능한 검증 산출물(evidence artifact)**: 심사·투자자에게 신뢰를 주는 검증 산출물 형태 — 테스트 리포트, e2e 테스트 커버리지, 벤치마크 데이터셋, 감사 로그 샘플, 평가 대시보드. 우리가 `scripts/verify_static.py`·Playwright e2e(19테스트)를 보유한 맥락에서, 이를 "검증 증거"로 제시하는 모범.
10. *(덤)* **AI Agent 평가의 최신 동향**(2024~2026): 에이전트 평가가 단일 정답 정확도에서 궤적·도구사용·안전성 종합 평가로 이동하는 흐름, 평가의 표준화 시도가 있으면 정리.
11. *(덤)* **"검증 가능성"을 발표에서 언어화하는 프레임**: 측정 못 한 것을 솔직히 "갭"으로 두면서도 검증 설계의 신뢰성을 보이는 커뮤니케이션 패턴(과장 없이 신뢰 주기).

### 출처 규칙 (반드시 준수)
- **공통 5가드**: ① 발견마다 **출처 URL + 원문 짧은 인용 + 발행처·날짜 + 신뢰도(1차/2차/[미검증])**. ② 핵심 평가지표 정의·산식과 파일럿 표본·통계 관행은 **2출처 교차**. ③ **링크 미접속 시 인용 금지.** ④ **한국 맥락·최신 우선**(에이전트 평가는 빠르게 진화 → 2023~2026 우선). ⑤ 의심 시 보수적 [미검증]. 내부 정본(`_canon.md`)과 충돌하면 _canon 우선(SSOT) — 특히 데모 3종·고정 KPI는 _canon이 승리.
- **도메인 가드**:
  - **1차로만 인정**: 동료평가 논문·저명 워킹페이퍼, 표준기관·학술의 평가 방법론, 공공/금융기관 PoC 보고서, 공모전 주최 측 공식 평가기준·수상작 공식 발표. 평가지표 정의는 **산식·정의 원문**과 함께.
  - **[2차/미검증]로 격하**: 평가 도구·플랫폼 벤더의 자가 우월성 주장("우리 eval이 최고"), 마케팅 블로그, 출처불명 "AI Agent KPI 10선" 리스트. 본문 인용 시 "[2차/미검증·벤더]" 꼬리표 + 1차 표와 분리.
  - **"0건·100%" 절대값 가드**: "PII 누출 0건" "근거추적 100%" 같은 **절대값 목표는 측정·검증 방법 없이는 [미검증]** 으로 취급. 이 리서치의 임무는 절대값을 주장하는 게 아니라 **"그 절대값을 어떻게 검증 가능하게 측정하나"** 의 프로토콜을 찾는 것. 검증 불가능한 절대값 주장은 오히려 심사 감점 위험임을 명시.
  - **체리피킹 가드**: 데모 시나리오는 "잘 되는 것만 보여주기"가 아니라 **실패·차단 케이스도 시연해 신뢰를 얻는** 설계가 정석. 과장된 데모 설계를 권하지 말 것.
- **경계 가드**: AI 협업 생산성 일반론·효과 크기 수치(D8 소관)·우리 KPI의 외부 학술 측정 프레임 정의(D8) 재수집 금지 — 본 파일은 **우리 MVP의 검증 설계·데모 KPI·파일럿 설계**에 한정.

### 한국 맥락·기간
- 한국 핀테크·금융 AI 공모전·PoC 사례를 우선. 국내 직접 사례가 희소하면 글로벌 AI Agent 평가·핀테크 PoC로 보강하되 "국내 직접 사례 부재/존재"를 갭으로 명시.
- 기간: AI Agent 평가지표·프레임워크는 **2023~2026 최신 우선**. 통계·실험설계 고전(효과크기·표본설계)은 검증된 것 허용하되 연도 표기.

### 출력 형식 (이 순서로)
1. **핵심 발견 요약** — A(평가지표)·B(데모 성공기준)·C(파일럿 KPI·통계)·D(덤) 블록별. 각 지표는 **"정의 + 산식 + 측정법 + 한계"** 로.
2. **평가지표 카탈로그 표**: `지표명 | 정의·산식 | 무엇을 검증하나 | 측정 방법·도구 | 한계·함정 | 출처 URL | 발행처·날짜 | 신뢰도`
3. **데모 3시나리오 성공기준 설계표**(이 표가 본 파일의 핵심 산출): `시나리오(SME/전세/피싱) | 보여줄 동작(골든패스) | 성공기준(정량) | 실측 지표 | 측정·검증 방법 | 의도적으로 노출할 실패/차단 케이스 | 근거 출처`
4. **근거표(1차 전용)**: `주장(지표/방법/선례) | 출처 URL | 발행처·날짜 | 신뢰도(1차) | 원문 짧은 인용`
5. **[2차/미검증·벤더] 별도표**: `주장 | 출처 | 발행처 | 왜 격하했나`.
