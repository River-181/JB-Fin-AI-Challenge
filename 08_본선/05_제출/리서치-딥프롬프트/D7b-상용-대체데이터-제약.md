# D7b — 상용 · 대체데이터 · 제약

## 메타
- **도메인**: 상용·대체데이터·문서·제약 — 신용평가사(CB)·통신·카드매출·상권·시세 등 **유료 상용 데이터**, 라이선스·요금·이용조건·반출 제약.
- **티어**: ★★
- **심사항목**: 3·4 (3.5 외부 데이터·상용 API 출처·라이선스·제약 · 4.x 검증·운영 현실성)
- **실행 배치**: 배치 2 (정합·규제·구현)
- **경계 메모**:
  - **D7b(여기) ↔ D7a**: *상용·유료·대체데이터=D7b / 공공·무료·개방 API=D7a.*
  - **D7b ↔ D6**: *외부에서 취득하는 상용 데이터=D7b / 행 내부 시스템 연동 표준=D6.*
  - **D7b ↔ D10**: *데이터 반출·이용조건 제약(계약·라이선스)=D7b / 기술적 반출 차단·보안 통제=D10.*

## 딥리서치 프롬프트 (이 아래 통째로 붙여넣기)

당신은 **한국 상용·대체데이터 조달/라이선스 분석가**다. 목표는, RM용 거버넌드 AI Agent 콘솔(JB LocalGuard OS)이 소상공인·전세·사기 도메인에서 보강용으로 쓸 만한 **상용·대체데이터(유료)** 를 식별하고, 각각의 **공급사·데이터 항목·요금 모델·이용조건·반출(외부 처리)·재배포·가명/집계 제약**을 표로 정리하는 것이다. 이 콘솔은 외부 프런티어 LLM의 추론은 쓰되 **고객 원본 PII·신용정보를 외부로 반출하지 않는** 설계가 핵심이므로, 각 상용 데이터의 **반출/외부처리 제약**이 우리 설계와 맞는지 특히 중요하다.

### 알아내야 할 것 (구체 질문)
1. **신용평가사(CB) 데이터**: NICE평가정보·KCB(코리아크레딧뷰로) 등이 제공하는 기업/개인사업자 신용·연체·여신 관련 상용 데이터·스코어·API. 제공 형태, 이용 자격(금융회사 한정 여부), 신용정보법상 이용 제약, 요금 모델(공개 범위), 반출·재가공 제약.
2. **카드매출·결제 대체데이터**: 카드사·VAN·PG·결제대행이 제공하는 가맹점/소상공인 **매출 추정·결제동향** 데이터(상권 매출지표 등) 상용 상품의 존재·공급사·이용조건. (히어로 케이스 "카드매출 둔화" 지표의 데이터 근거 가능성.)
3. **통신/통신과금 대체데이터**: 통신 3사·관련 사업자가 제공하는 본인확인·통신연체·유동인구(상권) 등 상용 데이터의 가용성·이용조건.
4. **상권·시세·부동산 상용 데이터**: 상권분석 상용 서비스(매출·유동인구·임대시세), 부동산 시세(KB부동산 시세 등) 데이터의 상용 제공·라이선스·API·재배포 제약. (공공 실거래가는 D7a이므로, 여기서는 **상용·가공 시세/상권**만.)
5. **문서·등기·법인 상용 데이터**: 법인등기·기업정보(예: 기업 신용·재무 요약) 상용 제공사, 대량/자동 조회 라이선스·요금.
6. **이상거래·사기 관련 상용 데이터/솔루션 데이터피드**: FDS·피싱 번호/계좌 블랙리스트 등 상용 데이터 피드의 존재·공급 주체·이용조건(가용 시).
7. **반출·이용조건 제약 종합**: 위 상용 데이터들이 (a) 외부 클라우드/외부 LLM으로의 전송을 약관상 허용하는지, (b) 가명/집계 형태 가공·재배포를 허용하는지, (c) 금융회사 내부망 한정 이용을 요구하는지. **우리 PII 비반출 설계와의 적합성**을 데이터별로 판정.
8. (덤 유도) 위 카테고리 밖에서 한국에서 실제 구매 가능한 유용한 상용/대체데이터가 있으면 동일 표로 추가하되, 요금/조건 미확인은 [미검증].

### 출처 규칙 (반드시 준수)
- ① 발견마다 **URL + 원문 짧은 인용 + 발행처·날짜 + 신뢰도(1차/2차/[미검증])**.
- ② 공급사 공식 페이지·약관·상품 소개를 1차로, 요금·제약은 가능하면 2출처 교차.
- ③ 링크 미접속·미확인 항목은 인용 금지, [미검증] 분리. **상용 요금은 공개되지 않는 경우가 많으므로 "비공개(문의)"로 정직하게 표기**하고 추정 금지.
- ④ 한국 맥락·최신 우선(현행 상품·약관).
- ⑤ 정본(_canon.md)과 충돌 시 정본 우선(SSOT).
- **도메인 가드(이 도메인 전용)**: 상용 데이터는 **반출·이용조건이 라이선스의 핵심**이므로, 약관/이용조건 원문 중 (전송·재배포·가공·이용주체 제한) 문구를 인용하라. **신용정보(CB) 데이터는 신용정보법 적용 대상**이라는 점을 함께 표기(법 조문 상세는 다른 문서가 담당하나, "신용정보법 적용" 사실은 명시). 요금 미공개는 추정하지 말고 "비공개".

### 한국 맥락·기간
- 대상: 한국에서 실제 조달 가능한 상용·대체데이터. 현행 상품 기준. 금융회사(지방은행) 이용 관점.

### 출력 형식 (반드시 이 구조 — 표 강제)
1. **핵심 발견** — 우리 설계(PII 비반출)와 잘 맞는 상용 데이터 / 반출제약으로 부적합한 데이터 구분.
2. **상용 데이터 인벤토리 표 (필수)** —
   | 데이터/상품 | 공급사 | 주요 항목 | 제공 형태(API/파일/조회) | 요금 모델 | 이용 자격 | 반출·재배포·가공 제약(원문 인용) | 신용정보법 적용 | 신뢰도 | URL |
3. **PII 비반출 적합성 표** — | 데이터 | 외부 LLM 전송 가부 | 가명/집계 처리 요건 | 내부망 한정 여부 | 우리 설계 적합도(적합/조건부/부적합) |
4. **근거표(이용조건 원문)** — | 주장 | 출처 URL | 발행처·날짜 | 신뢰도 | 원문 인용 |
5. **제품 적용 시사점** — 본선에서 상용 데이터를 "보강 옵션/확장 경로"로 어떻게 위치시킬지(공공=기본, 상용=확장).
6. **갭 / [미검증]** — 요금 비공개·조건 확인 불가 항목.

### 딥리서치 깊이 지시
공급사(NICE·KCB·카드/VAN·상권 분석사 등)의 공식 상품/약관 페이지에 접속해 **이용주체 제한·반출/전송 제약** 문구를 우선 확보하라. 요금이 비공개면 정직하게 "비공개(영업문의)"로 표기. 핵심은 "구매 가능한가"보다 **"우리 PII 비반출·외부 LLM 활용 설계와 라이선스가 충돌하지 않는가"** 이므로, 적합성 판정 표를 반드시 채워라.

## Opus 점검 로그 (2패스)
- **패스 1 (경계·정확도 적대적 비판 → 보완)**: 초안이 공공 실거래가(D7a)나 시스템 연동(D6)으로 새거나, 상용 요금을 추정해 환각을 부를 위험이 있었다. → 경계 메모로 D7a(공공)·D6(연동)·D10(기술 통제)와 분리, 질문 4번에 "공공 실거래가는 D7a, 여기는 상용 가공 시세만" 명시, 가드에 "요금 미공개는 추정 금지·'비공개' 표기"를 박았다.
- **패스 2 (제품 차별점 결속·심사 가치 재비판 → 보완)**: 상용 데이터를 단순 나열하면 우리 핵심 차별점(PII 비반출)과 연결이 끊겨 심사 어필이 약했다. → **"PII 비반출 적합성 표"(외부 LLM 전송 가부·가명요건·내부망 한정·적합도)** 를 출력 형식에 강제하고, CB 데이터의 신용정보법 적용을 인벤토리 열로 추가했다. 또 상용을 "기본(공공)에 대한 확장 경로"로 포지셔닝하도록 시사점 항목을 지정해, 심사 5(확장성)와도 연결되게 했다.
