# D2 — 금융도메인 · 여신심사 · RM

## 메타 (프롬프트 밖 — 생성·운영용)
- **도메인**: 금융 도메인 지식 — 여신·심사·사후관리·RM 역할·업계 구조·핵심 용어 (메커니즘 전담)
- **티어**: ★★
- **심사 항목**: 1(주제/문제정의·핍진성) · 3(AI Agent 설계 — 업무흐름을 에이전트 역할로 매핑)
- **실행 배치**: 배치 2 (정합·규제·구현 병렬군)
- **경계 메모**:
  - **D2 ↔ D1**: *얼마나 아픈가=D1(수치) / 어떻게 일하는가=D2(메커니즘).* **수치(연체율·피해규모·금액·건수)는 D1에서 확보 — 여기서 재수집·재인용 금지.** 본 프롬프트는 메커니즘·프로세스·구조·용어만.
  - **D2 ↔ D4**: *여신 업무 절차·심사 메커니즘 그 자체=D2 / 누가 어떤 직무로 협업하나(직무 카테고리·권한 분담)=D4.* RM "일상 업무흐름"은 D2(여신 프로세스 안의 행위), 직무 간 협업 인터페이스·의사결정 권한은 D4.
  - **D2 ↔ D9**: 사람의 업무 절차=D2 / 그것을 구현하는 코드·도구·기술스택=D9.
- **제품 연결**: JB LocalGuard OS — RM용 거버넌드 AI Agent 콘솔. 본 리서치는 "상담→심사→약정→실행→사후관리" 여신 라이프사이클을 에이전트 운영 루프(Case→AgentRun→Skill→Evidence→Approval→Audit)에 핍진하게 매핑하기 위한 도메인 베이스.

## 딥리서치 프롬프트 (이 아래 통째로 붙여넣기)

당신은 **한국 시중·지방은행의 여신(기업·소상공인 대출) 실무 메커니즘을 정확히 설명하는 금융 도메인 리서처**다. 목표는 "현직 RM(영업점 여신담당)·심사역이 읽어도 어색하지 않은 수준"의 **업무 메커니즘·프로세스·구조·용어 지식**을 1차/공개 자료에 근거해 정리하는 것이다. **수치(연체율·피해액·시장규모·금액·건수)는 이 리서치의 대상이 아니다 — 메커니즘·절차·역할·용어만 다룬다.** 수치가 불가피하게 등장하면 "예시·맥락"으로만 표기하고 출처를 달되, 핵심 산출물로 수집하지 말 것.

### 알아내야 할 것 (핵심 질문)

**A. 여신 라이프사이클 — 단계별 메커니즘**
1. 한국 은행의 기업·개인사업자(소상공인) **여신 프로세스 전체 단계**를 순서대로: **상담·신청 → 심사 → 여신 승인(의사결정 권한단계) → 약정·계약 → 실행(대출 실행) → 사후관리(사후점검)**. 각 단계에서 실제로 무엇이 일어나고, 어떤 산출물(신청서·심사보고서·약정서 등)이 생성되는가?
2. 각 단계의 **표준 소요·핸드오프 구조**: 영업점(프런트)과 본부 심사부(백)의 역할 분담, 어디서 승인 권한이 영업점 전결 → 본부 심사 → 여신위원회로 넘어가는가(여신 승인 권한 위임 체계의 일반 원리).

**B. 심사(여신심사) 의사결정 요소**
3. 여신심사에서 보는 **핵심 판단 요소**: 신용도(신용평가·신용등급/CB), 상환능력(현금흐름·소득·매출), 담보·보증(담보평가·LTV류 개념·보증서), 사업성·업종 리스크, 자금용도(운전자금 vs 시설자금)의 구분과 각각의 심사 포인트.
4. **신용평가 체계의 작동 원리**: 내부신용등급(차주등급/거래등급)과 외부 CB(신용정보회사) 점수가 심사에서 어떻게 결합되는가. 정성평가 vs 정량평가의 역할.
5. **여신 승인 등급/권한 구조**: 금액·리스크 등급에 따라 전결권이 어떻게 분기되는지의 **일반 메커니즘**(특정 은행 내부규정 추정 금지 — 공개된 일반 원리만).

**C. 사후관리 — 트리거와 행위**
6. **여신 사후관리(사후점검)의 메커니즘**: 실행 후 무엇을 모니터링하고, 어떤 **트리거(조기경보 신호·EWS 개념)**가 발생하면 어떤 행위(상시점검·재심사·자금용도 점검·연체관리·채권보전·기한이익상실)로 이어지는가.
7. **부실·연체 단계의 흐름**: 정상 → 연체 → (요주의/고정/회수의문/추정손실 등) **자산건전성 분류 단계**의 일반 개념, NPL(부실채권)·채권관리로의 이행 경로(메커니즘만 — 수치 금지).
8. 소상공인·개인사업자 차주 특유의 사후관리 관심사(매출 둔화 신호, 다중채무, 정책자금 연계 가능성 등)는 메커니즘 관점에서 무엇인가.

**D. RM(관계관리·여신담당)의 일상 업무흐름**
9. 한국 은행에서 **RM(Relationship Manager) / 기업금융 담당 / 영업점 여신담당**의 실제 일상 업무: 고객 발굴·상담, 여신 제안·서류 징구, 심사 연계, 사후관리·재여신, 교차판매. 하루·한 달 단위 업무 리듬을 메커니즘으로 서술.
10. RM이 **시간을 많이 쓰는 반복·수작업 병목**(서류 검토, 보고서 작성, 모니터링 알림 추적 등)은 일반적으로 무엇으로 보고되는가 — AI 보조의 침투 지점을 식별하기 위함.

**E. 용어집 (핍진성 핵심)**
11. 다음 용어를 **현직 통용 정의**로 짧게 정리: 여신, 수신, 운전자금/시설자금, 담보·보증·신용대출, LTV/DTI/DSR(개념 정의만), 차주신용등급, CB·신용평가, 연체·기한이익상실(EOD), 자산건전성 분류(정상/요주의/고정/회수의문/추정손실), NPL·부실채권, 충당금, 대환대출, 정책자금·보증서대출(신·기보·지역신보), 워크아웃·채권보전, 여신위원회·전결권, EWS(조기경보시스템), 사후관리.
12. *(범용 덤 유도)* 위 외에 **현직 RM·심사역이 일상적으로 쓰지만 외부인이 잘 모르는 실무 용어·관용 표현·업무 단축어**가 있으면 추가로 수집하라.

### 출처 규칙 (반드시 준수)
- **공통 5가드**: ① 발견마다 **출처 URL + 원문 짧은 인용 + 발행처·날짜 + 신뢰도(1차/2차/[미검증])** 를 붙인다. ② 핵심 주장은 **2개 출처 교차** 확인(불가 시 단일출처 [미검증] 명시). ③ **링크에 실제 접속 못 하면 인용 금지.** ④ **한국 맥락·최신 우선**(옛 제도·폐지 규정과 현행 구분). ⑤ 의심 시 보수적으로 [미검증] 표기.
- **도메인 가드**:
  - 1차로 인정: **금융감독원·은행연합회·한국금융연수원(KBI) 교재/용어집, 은행법·은행업감독규정·여신전문금융업 관련 법령(law.go.kr), 은행연합회 여신거래 표준약관, 공개된 은행 여신 업무 안내·IR/사업보고서의 업무 설명**.
  - 2차/보강: 금융 실무 해설서·업계 보고서·교육자료 (발행처·연도 명시).
  - **금지: 특정 은행의 비공개 내부 여신규정·전결 한도·심사 매뉴얼을 "추정"으로 서술하지 말 것.** 권한·등급 구조는 *공개된 일반 원리*로만 기술하고, 은행별 차이는 "은행마다 상이"로 명시.
  - **수치 재수집 금지**: 연체율·피해규모·시장금액 등은 본 리서치 산출물이 아니다(D1 담당). 부득이 등장 시 "맥락 예시"로만.
- **SSOT 우선**: 내부 정본(`_canon.md`)과 충돌하면 _canon이 승리.

## Opus 점검 로그 (2패스)

### 패스 1 — 적대적 비판 → 보완
- **비판 ① 경계 누수(수치).** 여신 메커니즘을 다루다 보면 LTV/DSR 한도 수치, 자산건전성 분류 연체일수, 충당금 적립률 등 "수치"를 끌어오기 쉽다 → D1과 중복·SSOT 충돌 위험.
  - **보완**: 프롬프트 역할문·도메인 가드·출력형식 3곳에 "수치는 D1 담당, 재수집 금지, 부득이 시 맥락 예시로만"을 명시. 용어집 항목(LTV/DTI/DSR)도 "개념 정의만"으로 한정.
- **비판 ② 내부규정 환각.** "여신 전결권한·심사 매뉴얼"은 은행 비공개 영역이라 딥리서치가 그럴듯한 가짜 한도·규정을 생성할 위험이 큼.
  - **보완**: 권한·등급 구조 질문(A2·B5)에 "특정 은행 내부규정 추정 금지, 공개된 일반 원리만, 은행별 차이는 '상이'로 명시" 가드를 직접 삽입. 도메인 가드에도 명문화.
- **비판 ③ D4와의 경계 모호.** "RM 일상 업무흐름"과 "직무 협업·권한 분담"이 겹칠 수 있음.
  - **보완**: 메타 경계에 "여신 절차 안의 RM 행위=D2 / 직무 간 협업 인터페이스·권한 분담=D4"로 분리 명시. D 블록(RM)은 "여신 프로세스 내 행위·병목"으로 스코프 한정.

### 패스 2 — 재비판 → 보완
- **재비판 ① 제품 연결 약함.** 메커니즘만 모으면 "그래서 제품에 뭐?"가 빠져 심사 3(에이전트 설계 정당화) 기여가 약해짐.
  - **보완**: 질문 A10에 "AI 보조 침투 지점(병목) 식별" 목적을 명시하고, 출력형식 4에 라이프사이클→에이전트 운영 루프·승인 게이트 매핑을 요구. _canon 운영계약(Case→AgentRun→…→Approval→Audit)과 직접 연결.
- **재비판 ② 핍진성 검증 장치 부재.** "현직이 읽어도 어색하지 않게"가 구호로만 남을 수 있음.
  - **보완**: 깊이 지시 마지막에 "현직 RM/심사역 관점 자체 점검 후 제출", 용어집에 "현직 통용 정의" + "외부인이 모르는 실무 관용어 덤 수집"을 요구해 핍진성을 산출물 형태로 강제.
- **재비판 ③ 1차 출처 풀이 막연.** "공개 자료"만으로는 딥리서치가 블로그로 흐를 수 있음.
  - **보완**: 도메인 가드에 한국금융연수원·은행연합회 표준약관·은행업감독규정·law.go.kr·감독원 등 **구체 1차 출처군**을 열거. 2차는 발행처·연도 명시 의무화.
