## 딥리서치 프롬프트 (이 아래 통째로 붙여넣기)

**역할**: 당신은 금융권 LLM/생성형 AI 도입의 **보안 위협 모델링과 방어 아키텍처를 설계하는 보안 엔지니어 겸 리서처**다. 정책이 "외부 LLM 활용을 조건부 허용"한다는 전제(이 부분은 별도 정책 트랙이 다룬다) 위에서, **"그렇다면 기술적으로 어떻게 안전하게 구현하는가"**만 깊게 파는 것이 목표다. 핵심 질문: **"은행이 망분리 환경에서 외부 프런티어 LLM의 추론력을 쓰되 고객 원본 PII·신용정보는 절대 외부로 반출하지 않으려면, 어떤 위협을 어떤 기술 통제(토큰화 게이트웨이·반출 스캔·프록시·가드레일·감사)로 막아야 하며, 그 근거가 되는 표준·선례는 무엇인가?"**

**중대 전제**:
- 이 리서치는 **법·정책의 허용/금지 판단을 내리지 않는다**(그건 별도 트랙). 본 리서치는 **위협 → 기술 통제 → 표준/선례** 의 기술 인과만 다룬다.
- "막을 수 있다"는 주장은 반드시 **구체 기술 메커니즘 + 그 한계(우회 가능성·잔존 위험)** 와 함께 보고하라. 보안에서 "완벽 차단"은 환각 신호다. 프롬프트 인젝션처럼 **현재 완전 해결이 안 된 위협**은 "완화"로만 표현하고 잔존 위험을 명시하라.

### 알아내야 할 것 (핵심 질문 — 1~8 필수, 9~11 덤 발견 유도)

**A. LLM 고유 위협 — 무엇을 막아야 하나**
1. **OWASP Top 10 for LLM Applications** 현행판(최신 연도 명시)의 10개 위협을 원문으로 확인하라. 각 항목(특히 Prompt Injection, Sensitive Information Disclosure, Insecure Output Handling, Excessive Agency, Supply Chain 등)의 정의·공격 시나리오·권고 완화책을 인용하라. **에이전트형(도구·행동 권한 보유) 시스템에 특히 위험한 항목**(Excessive Agency 등)을 식별하라.
2. **프롬프트 인젝션**(직접·간접 indirect injection)의 공격 메커니즘과 **실제 보고된 데이터 유출/탈취 사례**(연구·CVE·실사례)를 수집하라. 간접 인젝션(외부 문서·RAG 소스에 숨긴 명령)이 RAG·도구 사용 에이전트에 어떻게 작동하는지, 현재 방어책의 **한계**(완전 차단 불가)를 명시하라.
3. **민감정보(PII/신용정보) 유출 경로**: LLM이 데이터를 유출하는 기술 경로 — 프롬프트에 PII 포함, 학습 데이터 암기·재현(memorization/extraction), 로그·캐시 잔존, 모델 응답을 통한 추론, 외부 API 전송 구간. 각 경로별 측정된 사례·연구를 인용하라.

**B. 방어 아키텍처 — 어떻게 막나 (제품 4중 방어의 근거)**
4. **토큰화·가명화 게이트웨이 / 데이터 마스킹 패턴**: 외부 LLM에 보내기 전 PII를 토큰·가명으로 치환하고 응답을 역치환하는 아키텍처(de-identification proxy, format-preserving tokenization, 토큰↔원본 키 분리보관). 실제 구현 패턴·제품·레퍼런스 아키텍처와 그 **한계**(맥락 손실, 준식별자 재식별 위험, 비정형 텍스트 내 PII 탐지 누락)를 정리하라.
5. **반출(egress) 스캔·DLP·외부전송 통제**: 외부로 나가는 프롬프트/데이터를 검사해 PII·기밀을 탐지·차단하는 기술(DLP, PII detection NER, output/input filtering, 정규식+ML 하이브리드). 금융권 외부전송 통제(망분리 게이트웨이·프록시) 패턴과 정확도·오탐/미탐 한계.
6. **모델 거버넌스 — 가드레일·레드팀·감사**: (a) **가드레일/AI 방화벽**(input·output guardrails, NeMo Guardrails·Llama Guard 등 유형과 작동 방식), (b) **LLM 레드티밍**(적대적 테스트·자동화 레드팀 도구·방법론·표준), (c) **감사·관찰성**(프롬프트/응답 로깅, 결정 추적성, 감사원장 audit log 설계, tamper-evident 기록). 각각의 표준·프레임워크를 인용하라.
7. **망분리 환경에서 외부 LLM을 호출하는 기술 패턴**: 물리·논리 망분리 또는 다층보안체계(MLS) 하에서 내부망 시스템이 외부 LLM API를 호출하는 안전 아키텍처 — **단방향 게이트웨이·프록시·브로커·DMZ 중계·토큰화 게이트웨이 경유** 등. 내부 PII는 내부에 두고 비식별 토큰만 외부로 나가는 구조의 구체 토폴로지(다이어그램 수준의 컴포넌트·데이터 흐름)를 정리하라.
8. **표준·프레임워크 매핑**: 위 통제를 뒷받침하는 보안 표준·가이드 — **NIST AI RMF, NIST SP 800-53/800-218, ISO/IEC 42001·27001, MITRE ATLAS(AI 위협), KISA·금융보안원(FSEC) AI/LLM 보안 가이드**(있으면 원문). 각 표준이 위 위협·통제를 어떻게 다루는지 매핑하라.

**C. 금융권 실선례**
9. **금융권(은행·핀테크) LLM 보안 도입 사례·아키텍처**(국내외): 외부/사내 LLM을 도입하며 PII 통제·게이트웨이·가드레일을 구현한 공개 사례·기술 블로그·컨퍼런스 발표(발행처·연도·구조 명시). 국내 은행·금융보안원 발표가 있으면 우선.

**D. 범용 덤 유도**
10. *(덤)* **온프레미스/사설 LLM(sLLM) vs 외부 프런티어 LLM** 하이브리드 라우팅 보안 트레이드오프 — 민감 작업은 내부 모델, 고난도 추론만 외부로 보내는 **모델 라우팅**의 보안 설계 패턴과 한계.
11. *(덤)* **에이전트 보안 고유 이슈** — 도구 사용·자율 행동 권한을 가진 멀티에이전트의 권한 최소화(least privilege), human-in-the-loop 승인 게이트, 행동 차단(kill switch) 설계 패턴. (우리 제품의 승인우선 루프와 연결되는 부분.)

### 출처 규칙 (반드시 준수)
- **공통 5가드**: ① 발견마다 **출처 URL + 원문 짧은 인용 + 발행처·날짜 + 신뢰도(1차/2차/[미검증])**. ② 핵심 보안 주장(통제의 효과·표준 항목 번호)은 **2출처 교차**. ③ **링크 미접속 시 인용 금지.** ④ **한국 맥락·최신 우선**(LLM 보안은 변화가 빠름 → 2023~2026 우선, 표준은 현행판 명시). ⑤ 의심 시 보수적 [미검증]. 내부 정본(`_canon.md`)과 충돌하면 _canon 우선(SSOT).
- **도메인 가드**:
  - **1차로만 인정**: **OWASP**(LLM Top 10 공식 문서), **NIST**(AI RMF·SP 800 시리즈), **ISO/IEC**, **MITRE ATLAS**, **KISA·금융보안원(FSEC)** 공식 가이드, 동료평가 보안 학술논문, CVE/공식 보안 권고. 표준은 **항목 번호·버전·발행연도**를 반드시 명시.
  - **[2차/미검증]로 격하**: 보안 제품 벤더의 자가 성능 주장("우리 DLP는 99% 차단"), 마케팅 백서, 출처 불명 블로그. 본문에 인용하되 "[2차/미검증·벤더]" 꼬리표를 달고 1차 표와 분리.
  - **"완전 차단" 금지 가드**: 프롬프트 인젝션·간접 인젝션 등 **현재 미해결 위협**을 "완전 방어 가능"으로 쓰지 말 것. 모든 통제는 **완화 수준 + 잔존 위험·우회 가능성**을 함께 보고. "완벽" "100% 차단" 표현이 나오면 [미검증]으로 의심.
  - **경계 가드**: 법·정책의 허용/금지 판단(D5b)·인증 통과 절차(D12)·코어뱅킹 내부 구조(D6)는 **다루지 말 것** — 본 파일은 기술·위협·아키텍처에 한정. 특정 은행 내부망 구성의 비공개 추정 금지(공개 표준·벤더 백서·공개 사례로만).

### 한국 맥락·기간
- 대한민국 금융 보안 기준 우선(KISA·금융보안원·전자금융 보안). 국내 1차 가이드가 희소하면 글로벌 표준(OWASP·NIST·MITRE)으로 보강하되 "국내 직접 가이드 부재/존재"를 갭으로 명시.
- 기간: LLM 보안 위협·방어는 **2023~2026 최신 우선**. OWASP LLM Top 10은 현행 최신판, 인젝션 연구·사례도 최신.

### 출력 형식 (이 순서로)
1. **핵심 발견 요약** — A(위협)·B(방어 아키텍처)·C(금융 선례)·D(덤) 블록별. 각 통제는 **"효과 + 잔존 위험"** 쌍으로 서술.
2. **위협 → 통제 매핑표**: `위협(OWASP 항목·인젝션 유형·유출 경로) | 기술 통제 | 통제의 한계·잔존 위험 | 근거 표준·출처 URL | 발행처·날짜·버전 | 신뢰도`
3. **근거표(1차 전용)**: `주장(통제·표준·사례) | 출처 URL | 발행처·날짜·버전/항목번호 | 신뢰도(1차) | 원문 짧은 인용`
4. **[2차/미검증·벤더] 별도표**: `주장 | 출처 | 발행처 | 왜 격하했나` (1차와 물리적 분리).
5. **외부 LLM 호출 안전 아키텍처(텍스트 다이어그램)**: 내부망 → 토큰화 게이트웨이 → 반출 스캔 → 외부 LLM → 역치환 → 감사원장 의 컴포넌트·데이터 흐름을 컴포넌트별 책임·통제와 함께. 어느 지점에서 PII가 토큰으로 바뀌고, 어느 지점에서 차단/로깅되는지 표시.
6. **제품 적용 시사점**: 위 위협·통제가 JB LocalGuard OS의 **4중 방어**(데이터 등급제·토큰화·모델 라우팅·반출 스캔·감사원장)와 **승인우선 루프(L0–L4)** 를 어떻게 기술적으로 정당화하는지. 우리가 채택해야 할 표준·통제·아키텍처 패턴 5~8개를 우선순위와 함께. 단, 인증 통과 절차는 D12·정책 허용근거는 D5b로 표시.
7. **갭 / [미검증]**: 국내 금융 LLM 보안 가이드 공백, 미해결 위협(인젝션)의 잔존 위험, 확정 못 한 표준 항목·버전.

### 딥리서치 깊이 지시 (Gemini / Perplexity Deep Research 대상)
- 단순 "보안 위협 나열" 금지. **위협마다 그것을 막는 구체 기술 통제와 그 통제의 한계**까지 인과로 연결하라. "DLP를 쓰면 된다"로 끝내지 말고 "DLP의 미탐률·우회 시나리오"까지.
- OWASP LLM Top 10은 **현행판 원문**을 열어 항목 번호·정의를 확인하고, 우리 에이전트형 제품에 가장 위험한 항목을 식별하라.
- **프롬프트 인젝션은 "완전 해결 불가"가 현재 정설**임을 전제로, 완화 수준과 잔존 위험을 솔직히 보고하라. 과장된 "완전 차단" 주장은 1차 출처로도 의심하라.
- 토큰화 게이트웨이·반출 스캔·망분리 중계 아키텍처는 **실제 컴포넌트·데이터 흐름 수준**까지 내려가라(추상적 "안전하게 처리한다" 금지). 금융권 실선례를 최소 1건 이상 찾되 못 찾으면 갭으로 명시.
- 표준(NIST AI RMF·OWASP·MITRE ATLAS·ISO 42001·KISA/FSEC)은 **버전·발행연도·해당 항목**을 명시해 교차하라. 최소 8개 이상의 1차 표준·연구·사례 출처.

## Opus 점검 로그 (2패스)

### 패스 1 — 적대적 비판 → 보완
- **비판 ① "완전 차단" 환각 위험.** 보안 도메인은 벤더·블로그가 "우리 솔루션이 인젝션을 완전 차단"이라 주장 → 엔진이 그대로 받아쓰면 심사장에서 보안전문가에게 즉사한다. 프롬프트 인젝션은 현재 미해결이 정설.
  - **보완**: 중대 전제·"완전 차단 금지 가드"·깊이지시 3곳에 "모든 통제는 완화+잔존 위험으로만, 100% 차단 표현은 [미검증]"을 박음. 위협→통제 매핑표에 **"통제의 한계·잔존 위험" 칼럼을 강제**.
- **비판 ② D5b·D6·D12와 경계 누수.** "망분리"라는 단어가 정책(D5b)·시스템 구조(D6)·인증(D12)과 다 닿아 산만해질 위험.
  - **보완**: 경계 메모 4중 명시 + 역할문·경계 가드·출력형식 6에 "허용근거=D5b, 인증절차=D12, 코어구조=D6, 여기는 기술·위협·아키텍처만" 반복. "정책 허용 판단 안 함"을 중대 전제로 못박음.
- **비판 ③ 추상적 "안전하게 처리" 종결 위험.** 토큰화·게이트웨이를 형용사로만 쓰면 제품 4중 방어의 기술 근거가 안 된다.
  - **보완**: 질문4~7을 컴포넌트·데이터 흐름·한계까지 강제하고, 출력형식 5에 **텍스트 다이어그램(PII가 어디서 토큰으로 바뀌고 어디서 차단되는지)** 을 의무화. 깊이지시에 "실제 컴포넌트 수준까지" 명문화.

### 패스 2 — 재비판 → 보완
- **재비판 ① 벤더 보안제품 수치 혼입.** DLP·가드레일 벤더의 "99% 탐지" 같은 자가 성능치가 1차 표준과 섞이면 오염.
  - **보완**: 도메인 가드에 "벤더 자가 성능 주장=[2차/미검증·벤더]·꼬리표·물리적 분리" 규정. 출력형식 4에 별도표 분리.
- **재비판 ② 표준 버전 누락 위험.** OWASP LLM Top 10·NIST AI RMF는 개정되며 항목이 바뀌는데 버전 없이 인용하면 옛 항목을 현행처럼 쓸 위험.
  - **보완**: 도메인 가드·근거표·깊이지시에 "버전·발행연도·항목 번호 필수, 현행판 원문 확인"을 반복. OWASP는 "현행 최신판"을 명시.
- **재비판 ③ 에이전트 고유 위협 과소.** 우리 제품은 도구·행동 권한을 가진 멀티에이전트인데 일반 LLM 위협만 보면 Excessive Agency·자율행동 리스크를 놓친다(심사 5.5 직결).
  - **보완**: 질문1에 "에이전트형에 특히 위험한 항목(Excessive Agency) 식별"을, 덤 11에 "에이전트 보안·권한 최소화·승인 게이트·kill switch"를 추가해 우리 승인우선 루프와 연결. 출력형식 6에서 L0–L4 승인 레벨과 기술 통제를 잇게 함.
- **수렴**: 본 파일의 무게중심을 "위협 나열"이 아니라 **"위협 → 구체 기술 통제 → 그 한계 → 표준 근거 → 우리 4중 방어 정당화"** 의 인과 사슬 하나로 수렴. "완벽 방어"가 아니라 "방어 가능 범위와 잔존 위험을 정직하게 아는 팀"이라는 인상이 심사 5.5에서 더 강하다는 판단.
