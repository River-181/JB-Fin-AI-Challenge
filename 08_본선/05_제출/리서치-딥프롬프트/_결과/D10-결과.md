# 금융권 망분리 환경에서 외부 프런티어 LLM을 안전하게 쓰는 기술 아키텍처

## 핵심 결론

은행이 **망분리 환경에서 외부 프런티어 LLM의 추론력은 쓰되 고객 원본 PII·신용정보는 외부로 내보내지 않으려면**, 기술적으로는 **원본 데이터 비반출 원칙**을 중심에 둔 **다층 중개 구조**가 필요하다. 가장 보수적인 구현은 외부 모델이 보는 입력을 **비식별 토큰·가명·최소 맥락**으로 한정하고, 외부 호출은 **단일 프록시/브로커/DMZ 또는 CDS**를 통해서만 이뤄지게 하며, 왕복 경로 전체에 **입·출력 스캔, 도구 권한 통제, 사람 승인, 감사 로그**를 붙이는 방식이다. 이 구조의 목적은 “LLM을 믿는 것”이 아니라 **LLM을 불신한 채 통제된 주변 시스템으로 위험을 줄이는 것**이다. [\[1\]](https://genai.owasp.org/llmrisk/llm01-prompt-injection/)

다만 이 구조가 의미하는 것은 **완화**이지 **완전 차단**이 아니다. OWASP는 프롬프트 인젝션에 대해 “fool-proof methods”가 불분명하다고 명시하고, 영국 NCSC와 MITRE도 간접 인젝션은 **통제되지 않은 외부 데이터 소스**를 통해 계속 유입될 수 있어 잔존 위험이 남는다고 본다. 따라서 은행의 목표는 “프롬프트 인젝션을 0으로 만든다”가 아니라, **원본 데이터가 외부 모델에 닿지 않게 하고**, **모델이 어떤 공격을 받아도 실행 가능한 권한과 반출 가능한 데이터 양을 작게 만드는 것**이어야 한다. [\[2\]](https://genai.owasp.org/llmrisk/llm01-prompt-injection/)

실무적으로는 다음의 **제품 4중 방어**가 가장 설득력 있다. 첫째, **토큰화·가명화 게이트웨이**가 외부 호출 전 원본 PII를 치환한다. 둘째, **반출 스캔·DLP·프록시**가 외부로 나가는 프롬프트와 돌아오는 응답을 검사한다. 셋째, **가드레일·권한중개·HITL**이 LLM을 에이전트가 아니라 “권한 없는 제안 엔진”으로 취급한다. 넷째, **감사·관찰성·레드팀·공급망 검증**이 운영 중 우회와 실패를 찾아낸다. 이 네 층은 각각 독립적으로도 가치가 있지만, 어느 하나만으로는 충분하지 않다. [\[3\]](https://genai.owasp.org/llmrisk/llm022025-sensitive-information-disclosure/)

## 무엇을 막아야 하는가

OWASP의 **현행판**은 **“2025 Top 10 Risk & Mitigations for LLMs and Gen AI Apps”**이다. 공식 목록은 다음 10개다: **LLM01 Prompt Injection, LLM02 Sensitive Information Disclosure, LLM03 Supply Chain, LLM04 Data and Model Poisoning, LLM05 Improper Output Handling, LLM06 Excessive Agency, LLM07 System Prompt Leakage, LLM08 Vector and Embedding Weaknesses, LLM09 Misinformation, LLM10 Unbounded Consumption**. 은행의 망분리-외부모델 시나리오에서 특히 위험한 항목은 **LLM01, LLM02, LLM03, LLM05, LLM06, LLM08**이다. 이유는 이 여섯 가지가 각각 **원본 반출, 도구 오남용, RAG 오염, 외부 공급자/모델 리스크, 에이전트 권한 남용, 경계 우회형 검색·임베딩 취약점**으로 직결되기 때문이다. [\[4\]](https://genai.owasp.org/llm-top-10/)

**Prompt Injection**은 사용자 입력이나 외부 문서가 모델의 행위를 바꾸는 문제다. OWASP는 직접 인젝션과 간접 인젝션을 구분하고, 간접 인젝션은 웹페이지·파일 같은 외부 소스의 텍스트가 모델에 해석되어 의도치 않은 동작을 유발한다고 설명한다. 특히 **RAG와 에이전트형 시스템**에서는 “외부 콘텐츠를 읽는 행위” 자체가 공격면이 된다. 모델이 문서를 읽는 순간 그 문서는 데이터가 아니라 **새로운 지시문 후보**가 되기 때문이다. [\[5\]](https://genai.owasp.org/llmrisk/llm01-prompt-injection/)

**Sensitive Information Disclosure**는 단순히 “응답에 PII가 보였다”보다 넓다. OWASP 2025는 PII, 금융 정보, 건강 정보, 기밀 비즈니스 데이터, 보안 자격증명, 법률 문서, 심지어 독점 모델의 학습 방식과 소스코드까지 민감정보 범주에 둔다. 은행 관점에서 이것은 고객정보뿐 아니라 **신용평가 룰, 사기탐지 특징량, 내부 업무지침, 시스템 프롬프트, 토큰 사전**도 모두 보호대상이라는 뜻이다. [\[6\]](https://genai.owasp.org/llmrisk/llm022025-sensitive-information-disclosure/)

**Improper Output Handling**은 현재 금융권이 가장 과소평가하기 쉬운 항목이다. 위험은 “모델이 틀리게 말한다”가 아니라, **모델 출력이 downstream 시스템에서 코드나 명령으로 실행되는 순간** 현실 시스템 취약점으로 바뀐다는 점이다. OWASP는 그대로 전달된 모델 출력이 **RCE, XSS, SQL injection, path traversal, privilege escalation**로 이어질 수 있다고 경고한다. 즉, 외부 LLM 호출은 **데이터 경계**뿐 아니라 **명령 경계** 문제이기도 하다. [\[7\]](https://genai.owasp.org/llmrisk/llm052025-improper-output-handling/)

**Excessive Agency**는 도구 권한을 가진 에이전트형 시스템에 특히 치명적이다. OWASP는 불필요한 확장기능, 과도한 기능, 과도한 권한, 과도한 자율성을 대표 사례로 들며, 메일 요약 에이전트가 읽기만 하면 되는데 발송 권한까지 가진 경우를 전형적 위험으로 제시한다. 은행에서 이는 “조회용 에이전트”가 결제·전송·변경 API까지 쥐는 순간 곧바로 사고로 연결된다는 뜻이다. [\[8\]](https://genai.owasp.org/llmrisk/llm062025-excessive-agency/)

**Supply Chain**은 외부 프런티어 LLM만의 문제가 아니다. OWASP는 모델 공급망 위험을 제3자 모델·데이터, LoRA 어댑터, 모델 출처 불명, 라이선스 및 T&Cs 변경, 협업형 모델 병합 환경, 클라우드/온디바이스 배포면까지 확장한다. 은행이 공개 SaaS LLM을 쓸 때는 모델 그 자체뿐 아니라 **API 약관 변경, 데이터 처리정책 변경, 부가 기능, 안전 필터, 확장 도구, 로그·캐시 행태**까지 공급망 요소로 다뤄야 한다. [\[9\]](https://genai.owasp.org/llmrisk/llm032025-supply-chain/)

## 프롬프트 인젝션과 민감정보 유출은 어떻게 실제로 일어나는가

프롬프트 인젝션의 핵심은 모델이 **“명령”과 “데이터”를 완벽히 분리하지 못한다**는 점이다. OWASP는 직접 인젝션을 사용자의 악성 입력이 모델 행위를 바꾸는 경우로, 간접 인젝션을 외부 웹사이트나 파일에서 들어온 텍스트가 모델 행위를 바꾸는 경우로 설명한다. Microsoft Azure 문서도 프롬프트가 **URL 같은 외부 소스로부터 증강**될 수 있음을 문서화한다. 따라서 RAG나 도구 사용 에이전트는 “문서를 읽는다”는 이유만으로 간접 인젝션 surface를 갖는다. [\[10\]](https://genai.owasp.org/llmrisk/llm01-prompt-injection/)

이 위험은 이미 실사례로 이어졌다. NVD의 **CVE-2025-32711**은 “**Ai command injection in M365 Copilot allows an unauthorized attacker to disclose information over a network**”라고 기술한다. 즉, 생산계 엔터프라이즈 AI에서 **프롬프트 인젝션이 실제 정보유출**로 연결된 공개 CVE가 존재한다. OWASP도 2025 Prompt Injection 페이지에서 간접 인젝션이 **웹페이지 요약, RAG 문서 저장소, 이메일 어시스턴트, 멀티모달 입력**을 통해 민감정보 유출과 임의 동작으로 이어질 수 있는 시나리오를 예시로 든다. [\[11\]](https://nvd.nist.gov/vuln/detail/cve-2025-32711)

간접 인젝션이 RAG에 먹히는 이유는 간단하다. 검색 단계에서는 문서가 “관련 자료”일 뿐이지만, 생성 단계에서는 그 문서가 동일한 컨텍스트 창에 들어와 모델이 **지시문 우선순위를 잘못 해석**할 수 있다. Microsoft Research의 Spotlighting 연구는 이런 공격을 위해 외부 데이터를 별도 표식으로 감싸는 방법을 제안했고, 실험에서는 공격 성공률을 **50% 이상에서 2% 미만**으로 낮췄다고 보고했다. 그러나 이것도 **완전 해결**이 아니다. OWASP는 근본적으로 fool-proof 방법이 불명확하다고 하고, NCSC는 간접 인젝션이 **system inputs, tool responses, other agentic systems**에서도 올 수 있다고 경고한다. [\[12\]](https://www.microsoft.com/en-us/research/publication/defending-against-indirect-prompt-injection-attacks-with-spotlighting/)

민감정보 유출 경로는 적어도 다섯 가지다. 첫째, **사용자가 프롬프트에 원본 PII를 직접 넣는 경로**다. 이 경우 외부 API 호출 자체가 앱 계층에서 원문 전달을 의미한다. OpenAI는 API 데이터가 기본적으로 학습에는 쓰이지 않지만 **최대 30일 abuse monitoring logs**에 보관될 수 있다고 명시하고, Anthropic도 상업용 API 입력·출력을 **기본 30일** 보관한다고 밝힌다. Azure Foundry는 더 보수적인 호스팅 모델을 제공하지만, 그 역시 프롬프트와 응답을 서비스 제공 및 모니터링을 위해 **처리**한다. 원본문서를 보낸 순간 “모델이 기억하느냐” 이전에 **공급자 처리면에 노출**되는 셈이다. [\[13\]](https://developers.openai.com/api/docs/guides/your-data)

둘째, **학습 데이터 암기와 추출**이다. Carlini 등은 LLM 쿼리만으로 개별 학습 예시를 복구할 수 있음을 보였고, 추출된 예시에는 **이름, 전화번호, 이메일 주소** 같은 PII가 포함됐다고 보고했다. 이어 Nasr 등은 오픈모델뿐 아니라 **ChatGPT 같은 production language models**에서도 훈련데이터 추출이 가능하다고 주장했고, Lukas 등은 **black-box API access만으로 extraction, inference, reconstruction** 세 유형의 PII leakage를 정의하고 실험했다. 따라서 “외부 모델에 원본을 보내도 학습에만 안 쓰면 괜찮다”는 식의 주장은 기술적으로 과장이다. [\[14\]](https://www.usenix.org/conference/usenixsecurity21/presentation/carlini-extracting)

셋째, **로그·캐시 잔존**이다. OpenAI의 2023년 공식 사고 보고서는 Redis 기반 버그로 인해 다른 사용자의 **chat history title**과 일부 첫 메시지가 노출될 수 있었다고 인정했다. 이는 “프롬프트·응답 데이터가 모델 내부가 아니라 주변 시스템의 로그·캐시·세션 계층에서 새는 경로”가 실제로 존재함을 보여준다. 은행 설계에서 토큰화를 모델 직전만이 아니라 **프록시, 캐시, 관찰성, APM, SIEM 전처리 전**에 적용해야 하는 이유다. [\[15\]](https://openai.com/index/march-20-chatgpt-outage/)

넷째, **응답 기반 추론·재구성**이다. Lukas 등의 연구는 PII leakage를 추출뿐 아니라 **inference와 reconstruction**으로도 정의한다. 즉, 모델이 원문을 통째로 재현하지 않더라도, 외부 모델이 본 컨텍스트와 내부 속성을 조합해 **개인을 재식별하거나 특정 속성을 추정**할 수 있다. 토큰화 뒤에도 준식별자 조합이 남으면 이 경로는 사라지지 않는다. [\[16\]](https://arxiv.org/abs/2302.00539)

다섯째, **외부 API 전송 구간**이다. Azure 문서는 프롬프트와 생성물이 서비스 처리를 위해 데이터 경로를 통과함을 설명하고, OpenAI와 Anthropic은 각각 보관·삭제 정책을 명시한다. 즉, 은행이 통제하지 못하는 외부 추론 서비스를 쓸 때 원본 PII를 보내지 말아야 하는 가장 강한 기술적 이유는 “혹시 모델이 기억할까 봐” 이전에 **그 데이터가 외부 서비스 운영면에서 처리·보관될 수 있기 때문**이다. [\[13\]](https://developers.openai.com/api/docs/guides/your-data)

## 어떤 방어 아키텍처가 현실적인가

가장 현실적인 패턴은 **내부 보안 브로커를 중심으로 한 비식별 중개형 토폴로지**다. 내부 업무망에서 LLM을 직접 호출하지 않고, 모든 요청이 **보안 브로커**를 지나도록 만든다. 브로커는 요청을 받아 **데이터 분류 → PII 탐지 → 토큰화/가명화 → 최소 맥락화 → 외부 호출 → 응답 DLP 스캔 → 정책 검증 → 필요 시 내부에서만 역치환 → 감사 기록**의 순서로 처리한다. 이때 토큰 사전과 역치환 키는 **애플리케이션·프록시·관찰성 스택과 분리된 금고**에 둔다. 외부 공급자는 원본을 보지 못하고, 내부 애플리케이션도 원칙적으로 역치환 권한 없이 토큰만 소비할 수 있게 설계하는 것이 좋다. [\[17\]](https://docs.cloud.google.com/sensitive-data-protection/docs/transformations-reference)

**토큰화·가명화 게이트웨이**의 핵심은 “원문 대체물”을 일관되게 만드는 것이다. Google Sensitive Data Protection은 **deterministic encryption**과 **format-preserving encryption**을 모두 제공하며, FPE는 **같은 문자집합·같은 길이**를 유지하는 토큰을 생성하고, 원래 키를 가진 경우 재식별을 지원한다. NIST SP 800-38G는 FF1/FF3 계열의 FPE를 표준화했고, NIST SP 800-122는 PII 보호에서 **de-identification**을 명시적 수단으로 제시한다. 은행에서는 주민번호·계좌번호·전화번호처럼 포맷이 중요한 필드는 FPE 또는 포맷 유지 토큰화가 유용하고, 고객명·주소·자유서술형 메모는 룩업 기반 불투명 토큰이 더 안전한 경우가 많다. [\[18\]](https://docs.cloud.google.com/sensitive-data-protection/docs/transformations-reference)

하지만 토큰화에도 한계가 분명하다. 첫째, **맥락 손실**이다. “김○○ 고객의 최근 연체 이력과 배우자 직업”을 토큰으로 바꾸면 추론 품질이 떨어질 수 있다. 둘째, **준식별자 재식별 위험**이다. NIST 자료는 생년월일·성별·우편번호 같은 **quasi-identifiers**만으로도 재식별이 가능할 수 있음을 강조한다. 셋째, **비정형 텍스트 탐지 누락**이다. 문장 속 별칭, 오탈자, 복합 언어, 이미지 내 텍스트, 표 캡처, 음성 전사본, 금융 약어는 정규식만으로 잘 잡히지 않는다. 따라서 토큰화 게이트웨이는 **탐지기 품질 한계**를 전제로 해야 하며, 탐지 신뢰도가 낮을 때는 외부 모델로 보내지 않고 내부 모델 또는 사람 검토 경로로 **fail closed** 해야 한다. [\[19\]](https://nvlpubs.nist.gov/nistpubs/ir/2015/nist.ir.8053.pdf)

**반출 스캔·DLP·프록시**는 토큰화 다음 방어선이다. NIST는 DLP를 **data in use, in motion, at rest** 전반에서 딥패킷 검사와 문맥 분석으로 무단 전송을 막는 능력으로 정의한다. Google은 네이티브 detector와 hotword/context rule을 제공하고, 결과의 **likelihood threshold**를 조절하면 거짓양성은 줄지만 참양성도 함께 놓칠 수 있다고 설명한다. 즉, 금융권의 외부전송 통제는 보통 **정규식 + 룰 + NER/ML + 컨텍스트 룰**의 하이브리드여야 하며, 망분리 구간에서는 이를 **단일 egress proxy**에 집중시키는 것이 운영상 유리하다. [\[20\]](https://csrc.nist.gov/glossary/term/data_loss_prevention)

**프록시·DMZ·CDS**는 네트워크 경계 통제다. NIST는 DMZ를 내부와 외부 사이에서 정책을 강제하고 내부망을 외부 공격으로부터 차폐하는 구간으로 정의하며, Cross Domain Solution을 서로 다른 보안 도메인 사이의 정보 전달을 통제하는 인터페이스로 정의한다. 따라서 은행이 외부 LLM을 쓴다면, 내부 업무망은 외부 모델과 직접 통신하지 않고 **물리·논리적으로 분리된 프록시/브로커**로만 통신해야 한다. **단방향 게이트웨이(데이터 다이오드)**는 정의상 한 방향만 허용하므로, 실시간 챗봇처럼 요청-응답이 필요한 워크로드에는 단독으로 맞지 않고, **비동기 배치 요약·파일 반출형 use case**나 별도 회신 채널이 있는 고보안 파이프라인에 더 적합하다는 점을 명시해야 한다. 이 부분은 다이오드의 one-way 특성에 근거한 설계적 추론이다. [\[21\]](https://csrc.nist.gov/glossary/term/demilitarized_zone)

**가드레일과 AI 방화벽**은 브로커의 런타임 계층에 놓여야 한다. OWASP Improper Output Handling은 모델 출력을 “다른 사용자 입력처럼” 취급하고 downstream 함수 호출 전에 검증하라고 권고한다. 따라서 외부 LLM의 응답은 HTML/Markdown/SQL/경로/명령/툴 파라미터로 직접 흘려보내지 않고, **스키마 검증, allow-list, prepared statement, context-aware encoding**을 거쳐야 한다. Google Model Armor, Microsoft Prompt Shields, NVIDIA NeMo Guardrails, Meta Llama Guard 같은 도구는 이러한 input/output screening과 policy enforcement를 제공하지만, 대부분은 **공식 제품 문서 또는 벤더 연구** 수준이므로 **\[2차/벤더\]** 취급이 타당하다. 이 도구들은 분명 유용하지만, 은행이 신뢰해야 할 것은 제품명보다 **배치 위치와 실패 시 동작**이다. [\[22\]](https://genai.owasp.org/llmrisk/llm052025-improper-output-handling/)

**에이전트 권한 중개**는 별도 계층으로 분리해야 한다. OWASP Excessive Agency는 최소 확장기능, 최소 기능, 최소 권한, 사용자 컨텍스트 실행, 사용자 승인, complete mediation을 권고한다. 은행형 구현으로 옮기면, 모델은 결코 코어뱅킹 API 토큰을 직접 갖지 않고, 오직 **정해진 툴 스키마**만 호출할 수 있어야 한다. 각 툴은 **별도 서비스 계정**, **최소 scope**, **사용자 OAuth 컨텍스트**, **고위험 작업 이중승인**, **속도 제한**, **kill switch**를 가져야 한다. 모델이 “송금하라”고 제안할 수는 있어도, **결정과 서명은 정책 엔진과 사람**이 내려야 한다. [\[23\]](https://genai.owasp.org/llmrisk/llm062025-excessive-agency/)

**감사·관찰성**은 나중에 붙이는 기능이 아니라 브로커의 기본 기능이어야 한다. NIST는 audit log를 시스템 활동의 시계열 기록으로 정의한다. AU-2는 어떤 이벤트를 로그할지 정의하는 통제이고, NIST SP 800-12는 감사기록 무결성을 위해 **digital signatures**나 **write-once devices**를 예시로 든다. RFC 5848은 Signed Syslog로 **origin authentication, message integrity, replay resistance, message sequencing, missing-message detection**을 제공한다. 따라서 은행용 LLM 감사원장은 적어도 **사용자/서비스 ID, 원문 대신 토큰화된 요청 요약, 정책 결정 이유, 사용된 모델/버전, 도구 호출, 사람 승인 여부, 반출 스캔 결과, 최종 응답 분류, 이전 로그 해시 또는 서명**을 담는 **tamper-evident** 형태가 바람직하다. “tamper-proof”보다는 **tamper-evident**가 더 정확한 표현이다. [\[24\]](https://csrc.nist.gov/glossary/term/audit_log)

## 표준과 프레임워크는 어떻게 연결되는가

NIST의 **AI RMF 1.0**은 기술 통제를 직접 나열하는 체크리스트라기보다, 위험관리 체계를 **GOVERN, MAP, MEASURE, MANAGE** 네 기능으로 정리한다. 외부 LLM 안전구현 문제에 이 틀을 적용하면, **GOVERN**은 정책·역할·승인·공급자 관리, **MAP**은 데이터 분류와 위협모델, **MEASURE**는 레드팀·탐지 정확도·로그 커버리지, **MANAGE**는 운영 중 우회·사고 대응·잔존위험 수용으로 연결된다. [\[25\]](https://nvlpubs.nist.gov/nistpubs/ai/nist.ai.100-1.pdf)

NIST의 **Generative AI Profile, NIST AI 600-1**은 이 문제에 더 직접적이다. 이 문서는 GenAI 특화 위험과 조치를 제시하면서, **MS-1.1-003**에서 “**Conduct adversarial role-playing exercises, AI red-teaming, or chaos testing**”을 권고하고, **MS-1.3-005**에서 독립된 평가와 structured human feedback를, **MS-2.7-014/016**에서 배포 중 모니터링과 프롬프트 인젝션·데이터 포이즈닝·membership inference 등을 겨냥한 red-teaming을 제시한다. 또 **GV-6.2 계열**은 제3자 GAI 시스템과 데이터에 대한 문서화·fallback·continuous monitoring을 요구한다. 즉, 외부 LLM 도입은 단순 네트워크 예외가 아니라 **제3자 AI 공급망 관리 대상**이다. [\[26\]](https://airc.nist.gov/docs/NIST.AI.600-1.GenAI-Profile.ipd.pdf)

NIST **SP 800-53 Rev. 5**에서는 세부 통제를 고를 수 있다. 이 주제와 가장 직접적으로 연결되는 것은 **AC-6 Least Privilege**, **SC-7 Boundary Protection**, **AU-2 Event Logging**, **AU-9 Protection of Audit Information**, **SI-4 System Monitoring**이다. 은행형 LLM 아키텍처로 번역하면 각각 **도구 최소 권한**, **DMZ/CDS 경계 통제**, **요청·응답·정책결정 로깅**, **감사로그 무결성**, **이상행위/반출 모니터링**이다. 이들은 LLM 전용 통제는 아니지만, 외부 모델 안전구현의 가장 튼튼한 기반이다. [\[27\]](https://csrc.nist.gov/pubs/sp/800/53/r5/upd1/final)

공급망과 개발 수명주기 측면에서는 **NIST SP 800-218 SSDF**가 중요하다. SSDF는 취약한 소프트웨어 출시를 줄이고 root cause를 제거하는 secure SDLC 실천집이며, 외부 LLM 연동 코드, 프록시, 토큰화 게이트웨이, LoRA/에이전트 플러그인, 관찰성 파이프라인까지 모두 그 범위에 들어간다. 특히 은행은 **모델 구성요소 인벤토리, 서명 검증, 취약성 스캔, 패치 정책, 변경관리**를 LLM 앱에도 동일하게 적용해야 한다. [\[28\]](https://csrc.nist.gov/pubs/sp/800/218/final)

**MITRE ATLAS**와 SAFE-AI는 위협모델 관점에서 유용하다. MITRE는 **Direct Prompt Injection, Indirect Prompt Injection, Excessive Agency, LLM Plugin Compromise, LLM Data Leakage**를 명시하고, 특히 간접 인젝션 항목에서 **통제되지 않은 데이터 소스가 남는 한 한계가 있다**는 잔존위험을 적는다. 이 점은 “보안 필터를 달았으니 이제 안전하다”는 식의 과신을 막아준다. [\[29\]](https://atlas.mitre.org/pdf-files/SAFEAI_Full_Report.pdf)

ISO 쪽에서는 **ISO/IEC 42001:2023**이 AI management system 표준이고, **ISO/IEC 27001:2022**가 ISMS 요구사항 표준이다. 둘 다 기술 제품 자체를 보장하진 않지만, 외부 LLM 사용을 **조직적 통제**와 **지속적 개선**, **내부감사**, **리스크 처리 프로세스**에 편입시키는 데 적합하다. 은행처럼 운영·감사·조달·보안이 분리된 조직에서는 이 두 표준이 “토큰화 게이트웨이” 같은 점 솔루션보다 오히려 더 중요할 수 있다. [\[30\]](https://www.iso.org/standard/42001)

한국 맥락에서는 **금융위원회 2023 ‘금융분야 AI 보안 가이드라인’**, **KISA 2025 ‘인공지능 보안 안내서’**, **금융보안원 2025 AI 보안성 평가**가 가장 직접적인 공식 축이다. 금융위는 AI 모델 개발단계별 보안 고려사항과 **AI 챗봇 서비스 보안성 체크리스트**를 제시했고, KISA는 **15개 보안위협 유형과 30개 보안대책**을 제시하는 국가·공공 맥락 안내서를 배포했으며, 금융보안원은 생성형 AI 활용과 관련해 **망분리 예외 구간에서도 안전하게 AI가 활용될 수 있도록 지원**하고 **모의 공격 기반 보안성 검증**을 수행하겠다고 밝혔다. 공개 문서의 해상도는 해외 표준보다 낮지만, 한국 금융권에선 이 세 문서가 실질적 정합성 기준에 가깝다. [\[31\]](https://www.fsc.go.kr/no010101/79825?curPage=&srchBeginDt=&srchCtgry=&srchEndDt=&srchKey=&srchText=)

## 금융권 공개 선례에서 읽히는 것

국내에서는 아직 **은행 내부의 상세 LLM 보안 토폴로지**가 공개된 사례가 많지 않다. 그러나 정책·평가 측면에서는 흐름이 분명하다. 금융위는 2023년부터 금융 AI 보안 가이드라인과 챗봇 체크리스트를 내놓았고, 2025년에는 생성형 AI 기반 금융상담·Q&A·시황요약·AI 은행원 등 **혁신금융서비스**를 다수 지정하면서 “혁신과 보안의 균형을 위해 탄탄한 보안체계”를 강조했다. 금융보안원은 같은 해 **AI 모델 모의공격 기반 보안성 검증**과 **망분리 예외 구간의 안전한 AI 활용 지원**을 발표했다. 공개발표만 놓고 보면, 한국 금융권의 실선례는 “직접 외부 LLM에 붙는다”보다 **예외 구간에 대한 보안성 평가와 통제 체계**가 앞서고 있다. [\[32\]](https://www.fsec.or.kr/bbs/detail?bbsNo=11629&menuNo=69)

해외 은행권에서는 싱가포르 은행협회(ABS)의 **Handbook on Generative AI Guardrails in Banking**가 매우 실무적이다. 이 핸드북은 은행의 GenAI use case를 분류하고, guardrail과 control implementation을 **개발 생애주기 전반**에 매핑한다. 문서에는 **Human-in-the-Loop, Moderation, User Transparency and Consent, Monitoring and Validation Activities** 같은 가드레일이 명시돼 있고, 문서 추출/요약과 코드 생성 사례에 적용 방법까지 설명한다. 이는 “은행형 가드레일”이 단순 콘텐츠 필터가 아니라 **운영·승인·피드백·검증을 포함한 체계**임을 보여주는 좋은 선례다. [\[33\]](https://www.abs.org.sg/docs/library/handbook-on-generative-ai-guardrails-in-banking.pdf)

개별 은행 사례로는 **Morgan Stanley**가 가장 공개도가 높다. Morgan Stanley는 GPT-4 기반 내부 어시스턴트가 **공개 인터넷이 아니라 내부 Morgan Stanley 콘텐츠에서만 응답**하도록 설계했다고 밝히며, “appropriate controls”를 언급했다. 구체 토큰화 구조까지 공개하진 않았지만, 이 사례는 금융권이 외부 최고성능 모델을 쓰더라도 **지식 경계와 데이터 소스 경계**를 명확히 나누는 방식을 선호한다는 점을 보여준다. [\[34\]](https://www.morganstanley.com/press-releases/key-milestone-in-innovation-journey-with-openai)

**BBVA**는 2026년 OpenAI와의 공개 사례에서 AI 도입 전략을 **trust, governance, structured learning** 세 축으로 설명했고, 보안·법무·컴플라이언스·기술 조직이 함께 trusted AI tools를 제공하도록 맞췄다고 밝혔다. 이 또한 상세한 망분리 설계를 공개한 것은 아니지만, 금융기관의 외부 frontier model 도입이 실제로는 **모델 자체보다 거버넌스와 안전한 사용 경계 설계**에 더 크게 좌우된다는 점을 보여준다. [\[35\]](https://openai.com/index/bbva/)

여기서 중요한 해석은 하나다. 공개 선례들은 거의 예외 없이 **원본 고객데이터를 외부 LLM으로 직접 보내는 아키텍처를 자랑하지 않는다**. 대신 **내부 콘텐츠 한정, guardrail, governance, monitoring, human oversight, evaluation** 같은 표현이 반복된다. 이는 기술적으로도 자연스럽다. 외부 모델이 강할수록, 금융기관은 더더욱 **원본 비반출·권한 최소화·중앙중개**에 매달릴 수밖에 없다. [\[36\]](https://openai.com/index/morgan-stanley/)

## 외부 프런티어 LLM을 쓸 때 권장되는 설계 원칙

가장 권장할 만한 패턴은 **하이브리드 라우팅**이다. **원본 PII·신용정보가 포함된 작업은 내부 sLLM 또는 비생성 규칙엔진**에서 처리하고, 외부 프런티어 LLM에는 **토큰화된 일부 컨텍스트와 추론 질문만 보낸다**. 예를 들어 여신심사 메모 초안 생성 같은 작업이라면, 내부 시스템이 먼저 고객명·주민번호·계좌번호·정확 주소·거래상세를 토큰으로 치환하고, 외부 모델에는 “고객 A123의 최근 상환패턴 및 리스크 요약을 바탕으로 설명문 초안을 작성하라” 수준만 전달한다. 응답이 돌아오면 내부에서만 역치환 가능 여부를 판정하고, 고위험 항목은 사람 검토 뒤 반영한다. 이 패턴은 토큰화·DLP·최소권한 설계와 가장 잘 맞는다. [\[37\]](https://docs.cloud.google.com/sensitive-data-protection/docs/transformations-reference)

에이전트형 시스템에서는 **도구 사용 자체를 별도 보안 경계**로 보는 것이 중요하다. 모델은 텍스트를 생성할 뿐이고, **실제 행동은 툴 브로커**가 한다는 원칙을 지켜야 한다. 각 툴은 명시적 스키마, allow-list 파라미터, 최소 scope, 사용자 컨텍스트, 승인 라인을 가져야 하며, 송금·계좌변경·고객통지·파일다운로드·지식베이스 색인 갱신처럼 영향이 큰 작업은 **기본 거부 후 승인**이 맞다. OWASP가 말한 “Require user approval”, “Complete mediation”, “Minimize extension permissions”은 은행형 에이전트 보안의 핵심 설계 규칙이다. [\[23\]](https://genai.owasp.org/llmrisk/llm062025-excessive-agency/)

운영상으로는 **레드팀과 가드레일 평가를 배포 전후에 반복**해야 한다. NIST AI 600-1은 red teaming을 일회성 이벤트가 아니라 측정·관리 활동으로 위치시키고, OWASP GenAI Red Teaming Guide도 모델 평가, 구현 테스트, 인프라 평가, 런타임 행위 분석을 함께 보라고 권한다. 금융권용으로 번역하면, 테스트셋에는 적어도 **직접/간접 프롬프트 인젝션, 멀티언어 우회, base64/emoji 난독화, 문서 요약형 hidden instruction, tool call injection, PII echo, SQL/HTML/Markdown payload, excessive agency 시나리오**가 들어가야 한다. [\[38\]](https://airc.nist.gov/docs/NIST.AI.600-1.GenAI-Profile.ipd.pdf)

마지막으로, **공급자 선택보다 브로커 설계가 더 중요하다**는 점을 분명히 해야 한다. Azure Foundry처럼 공급자 비학습·격리 호스팅 모델을 제공하는 형태는 분명 장점이 있지만, 그것만으로 은행의 보안 문제가 해결되지는 않는다. 반대로 OpenAI나 Anthropic에서 zero data retention 같은 옵션이 있어도, **프롬프트 인젝션**, **도구 남용**, **준식별자 재식별**, **로그 누수**, **응답 기반 코드실행** 문제는 남는다. 공급자는 최후의 안전판이 아니라 **브로커 외부의 가변 요소**로 취급하는 편이 안전하다. [\[39\]](https://developers.openai.com/api/docs/guides/your-data)

## 근거 메모와 한계

아래는 이번 결론을 떠받치는 **핵심 1차 출처 메모**다. 링크는 각 인용 표기를 통해 열람할 수 있다.

- **OWASP GenAI Security Project, 2025, 1차**. 짧은 인용: “**it is unclear if there are fool-proof methods**.” 프롬프트 인젝션은 완전 차단이 아니라 완화 대상이라는 이번 보고서의 가장 중요한 근거다. [\[40\]](https://genai.owasp.org/llmrisk/llm01-prompt-injection/)
- **NVD, 2025-06-11, 1차**. 짧은 인용: “**Ai command injection in M365 Copilot**.” 프롬프트 인젝션이 실제 production data disclosure로 이어진 공개 CVE다. [\[41\]](https://nvd.nist.gov/vuln/detail/cve-2025-32711)
- **Carlini et al., USENIX Security, 2021, 1차 학술**. 짧은 인용: “**recover individual training examples**.” LLM memorization/extraction이 이론이 아니라 실증된 공격임을 보여준다. [\[42\]](https://www.usenix.org/conference/usenixsecurity21/presentation/carlini-extracting)
- **Nasr et al., arXiv, 2023, 1차 학술**. 짧은 인용: “**extract gigabytes of training data**.” production LM에서도 추출 위협이 남음을 강화한다. [\[43\]](https://arxiv.org/abs/2311.17035)
- **NIST AI 600-1, 2024, 1차 표준**. 짧은 인용: “**AI red-teaming**.” 배포 전후 적대적 평가를 운영 필수 통제로 끌어올린다. [\[44\]](https://airc.nist.gov/docs/NIST.AI.600-1.GenAI-Profile.ipd.pdf)
- **MITRE SAFE-AI / ATLAS, 2025 게시본, 1차 프레임워크**. 짧은 인용: “**there is a limit to how effective the controls can be**.” 특히 간접 인젝션과 excessive agency에서 잔존 위험을 명시한다. [\[29\]](https://atlas.mitre.org/pdf-files/SAFEAI_Full_Report.pdf)
- **OpenAI Developer Docs, 현행 문서, 1차**. 짧은 인용: “**retained for up to 30 days**.” 외부 API 처리면 노출을 설명한다. [\[45\]](https://developers.openai.com/api/docs/guides/your-data)
- **Anthropic Privacy Center, 2026-06 갱신, 1차**. 짧은 인용: “**delete inputs and outputs … within 30 days**.” 공급자별 retention 차이를 보여준다. [\[46\]](https://privacy.claude.com/en/articles/7996866-how-long-do-you-store-my-organization-s-data)
- **Microsoft Foundry Data Privacy, 2026-05경 현행, 1차**. 짧은 인용: “**Prompts and completions are processed**.” 외부/호스티드 모델도 서비스 처리면을 가진다는 점을 문서화한다. [\[47\]](https://learn.microsoft.com/en-us/azure/foundry/responsible-ai/openai/data-privacy)
- **RFC 5848, 2010, 1차 표준**. 짧은 인용: “**message integrity … detection of missing messages**.” tamper-evident audit trail 설계의 기술 근거다. [\[48\]](https://datatracker.ietf.org/doc/html/rfc5848)

이번 리서치의 한계도 분명하다. 첫째, **국내 은행의 실제 내부망·프록시·토큰화 게이트웨이 구성은 공개자료가 부족**해, 본 보고서는 공개 표준·공식 발표·공개 사례를 바탕으로 한 **권고 아키텍처**를 제시한다. 둘째, **한국어 비정형 금융문서에서의 PII 탐지 정밀도**를 금융권 공통 기준으로 제시한 공개 1차 자료는 확인 범위 내에서 제한적이었다. 셋째, **프롬프트 인젝션은 현재도 미해결 문제**이므로, 어떤 제품·가드레일도 단독으로 “안전”을 보장하지 않는다. 그러므로 은행이 현실적으로 취해야 할 태도는, 외부 모델의 지능을 활용하되 **원본 비반출·최소권한·중앙중개·사람승인·감사증적**으로 실패 비용을 낮추는 것이다. [\[49\]](https://www.fsec.or.kr/bbs/detail?bbsNo=11629&menuNo=69)
