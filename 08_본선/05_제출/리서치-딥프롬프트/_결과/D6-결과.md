# **한국 은행 시스템 표준 구조 분석 및 RM용 거버넌스 AI Agent 콘솔 연동 방안**

## **핵심 발견**

* **계정계·정보계·채널계의 유기적 분리 및 연동 구조**: 한국 은행 IT 아키텍처는 데이터 정합성을 보장하는 실시간 거래 처리 중심의 계정계(SoR), 대량의 데이터를 수집·가공하여 리스크 예측 및 경영 분석을 수행하는 정보계, 사용자 접점의 인터페이스와 화면 출력을 제어하는 채널계로 삼차원 분리되어 연동된다1. 계정계가 실시간 거래의 최신 장부 역할을 맡고, 정보계는 배치(Batch) 및 실시간 변경 데이터 캡처(CDC) 스트리밍을 통해 비동기식으로 데이터를 적재하는 상호 보완적 아키텍처를 취하고 있다3.  
* **코어뱅킹 자바 현대화 및 지방은행의 MDD 선도 기술**: 메인프레임 중심의 기존 환경에서 유닉스(UNIX) 다운사이징 단계를 거쳐, 최근에는 오픈소스 기반의 자바(Java) 프레임워크와 모델 기반 개발(MDD) 방식을 채택하는 차세대 시스템 아키텍처가 전형적인 표준으로 자리 잡았다6. 지방은행인 전북은행(2013년 가동)과 광주은행(2016년 가동)은 대형 시중은행보다 선제적으로 LG CNS 주도의 자바 기반 코어뱅킹 현대화 프로젝트를 성공적으로 완수하였으며, 이를 통해 높은 유지보수 효율성과 빠른 금융상품 팩토리 대응 역량을 실증한 바 있다8.  
* **표준 연동 인터페이스의 공개 범위와 내부망의 한계**: 금융결제원 오픈뱅킹 표준 API(잔액, 거래내역, POST 기반의 계좌실명조회 등) 및 금융보안원의 마이데이터 표준 API(수신계좌 도메인의 bank.deposit scope 기반 세부 리소스)는 금융권 대외 접점의 데이터 규격을 상세히 정의한다11. 그러나 이는 은행 외부와의 합의된 프로토콜일 뿐이며, 은행 내부망 시스템과 직접 연결하기 위해서는 내부 통합 미들웨어인 MCI, EAI 및 전용 API 게이트웨이의 내부 표준 전문 규격에 어댑터를 맞추는 설계가 필수적이다15.  
* **AI 콘솔(JB LocalGuard OS)의 3대 핵심 이식 설계 및 PII 경계**:  
  1. *정보계 Read-only 연동*: 운영 데이터베이스에 직접 부하를 주는 것을 원천 방지하기 위해, 계정계 트랜잭션 로그를 실시간으로 리드하는 로그 기반 CDC(Debezium 등) 파이프라인을 구축하여 AI 콘솔에 위험 분석용 원시 데이터 스트림을 실시간 비동기로 인입시킨다4.  
  2. *RM 채널 UI 임베딩*: 영업점 RM 단말기의 표준 플랫폼인 넥사크로(Nexacro) 내에 웹뷰(WebView) 컴포넌트를 이식하고, 단말 시스템과 AI 콘솔 간 postMessage 기술을 사용하여 안전하게 양방향 데이터를 중계한다17.  
  3. *승인 행동 Write 연동*: AI 콘솔이 단독으로 계정계 원장이나 발송계에 직접 데이터를 쓰는 구조를 차단하고, RM이 단말기에서 '최종 승인'을 처리할 때 단말 런타임이 채널계 통합 인터페이스(MCI)를 기동하여 사내 표준 전문 트랜잭션을 실행하게 만든다15.  
  * *PII 비반출 보안 가명화 경계*: 은행 내부망 내부의 토큰화 프록시(Tokenization Proxy) 세그먼트에서 고객의 개인정보 및 신용정보를 비가역적인 토큰(Token)으로 상시 변환하여 AI 에이전트 콘솔에 전달하며, AI 콘솔은 식별할 수 없는 토큰만을 기계 학습 및 검증에 할용하도록 통제한다.  
* **지방은행 공동망 계획의 무산과 이에 따른 아키텍처적 유의점**: JB금융그룹 산하 전북은행과 광주은행은 당초 시너지 창출과 전산센터 현대화를 목적으로 전용 데이터센터(IDC)를 통한 통합전산센터 구축 투자 협약을 맺는 등 공동 인프라 구성을 추진하였다21. 하지만 원뱅크화 및 IT 독립성 축소를 우려한 노동조합의 거센 반발로 인해 2025년 3월 공동 전산센터 구축 계획을 전면 철회하고 각각의 개별 전산망을 독립적으로 구축하는 방향으로 전격 선회하였다23. 이에 따라 외부 스타트업의 솔루션은 양 지방은행의 이원화된 시스템 환경에 별도로 호스트를 구성하고 독립적으로 연동해야 하는 개별 최적화 과제를 안게 되었다23.

## **아키텍처 매핑**

아래 매핑 분석은 한국 시중·지방은행의 표준 시스템 인프라 아키텍처 구조 내부에서 JB LocalGuard OS 에이전트 콘솔이 기술적으로 통합되는 지점과 세부 통신 규격, 그리고 개인식별정보(PII)의 안전한 경계를 매핑하여 규정한다.

| **정보계 (Information Tier)** | 로그 기반 CDC 스트림 (Kafka Queue 토픽 기반 메시징)4 | **Read-only**: 계정계 원장 데이터의 변경 이력(수신이체, 여신 트랜잭션 등) 발생 시 데이터베이스 커밋 로그에서 이벤트를 비동기적으로 실시간 스캔 및 수집4. | **내부 망 데이터 정제 서버 (In-Bank Agent)**: 원본 데이터베이스에서 데이터가 수집되는 최초 파이프라인 단에 "가명처리 중계 에이전트"를 배치하여 이름, 계좌번호, CI값을 원천 수집하지 않고, 오직 실시간 위험 패턴 판독에 필요한 고유 패턴 피처 및 해싱 처리된 토큰 ID만 가공하여 AI 서버 측으로 스트리밍함5. | 분석용 실시간 FDS(이상거래탐지시스템) 및 대용량 DW 데이터 쿼리 수행 시 트랜잭션 처리(OLTP) 성능에 부정적 마찰을 주지 않기 위해 실시간 트랜잭션 로그 기반 CDC 스트리밍을 채택함4. |
| **미들웨어 (EAI / ESB / MCI Tier)** | MCI(Multi-Channel Integration) 채널 허브 및 EAI 표준 전문 변환 프로토콜15 | **Write**: RM의 승인 프로세스 완료 시, 거래 통제 및 알림 처리 요청 프로토콜을 내부 전사 표준 전문 형태로 변환하여 내부 업무 시스템으로 전계15. | **미들웨어 기반 역매핑 역전**: AI 콘솔에서 승인되어 반출되는 메시지는 식별 불가능한 가상 토큰만을 적재함. EAI 및 MCI 인터페이스가 은행 내부 보안망 내부 저장소를 조회하여 해당 가상 토큰에 대응하는 원본 계좌정보 및 통신채널 정보를 다시 덮어씌워 내부 발송 서버 및 코어뱅킹으로 전송함15. | MCI는 단말, 콜센터 등 다양한 채널의 메시지를 통제하고, EAI는 내부 업무 단 간 이기종 프로토콜(XML, JSON 등) 변환을 일괄 수행하므로 솔루션 간 결합도를 최소화함15. |

## **상세 기술 분석 및 구조적 고찰**

### **계정계, 정보계, 채널계 아키텍처의 분리와 동기화 메커니즘**

한국 은행 IT 아키텍처의 전형적인 분리 구현은 비즈니스 목적에 따른 부하 제어 및 무장애 설계에 기반한다. 계정계는 입출금, 이체, 대출 한도 산출 등 데이터 무결성이 철저히 지켜져야 하는 금융 비즈니스 로직을 수행하며 일차 장부 역할을 수반한다1. 이 계정계의 마스터 데이터(Master Table)인 거래 원장 테이블들은 실시간 온라인 트랜잭션(OLTP) 부하를 받기 때문에 가용성을 최대로 보호받아야 한다1.

\[계정계 Core DB (OLTP)\]  
         │  
         │ (실시간 트랜잭션 로그 발생)  
         ▼  
\[Debezium CDC (로그기반 캡처)\] ───► \[Tokenization Proxy (가명처리/해싱)\]  
                                                     │  
                                                     ▼ (비식별 가명 데이터 전송)  
                                         \[Apache Kafka (메시지 큐)\]  
                                                     │  
                                                     ▼  
                                      \[JB LocalGuard OS (AI 콘솔)\]

일반적인 통념상 정보계는 배치(Batch) 위주의 사후 데이터 분석 영역으로 알려져 있으나5, 현대 고도화된 은행 시스템은 변경 데이터 캡처(Change Data Capture, CDC) 및 실시간 동기화 데이터 파이프라인(ODS, Operational Data Store)을 추가하여 실시간 이상징후 탐지(FDS) 인프라를 동시에 구현한다3. 이 과정에서 데이터베이스의 트랜잭션 복제 로그 파일(PostgreSQL의 WAL, MySQL의 Binlog 등)을 직접 비동기 리드하는 로그 기반 CDC를 도입함으로써 가용성에 타격을 주지 않는 실시간 Read-only 경로가 완성되는 원리다5. 채널계는 프레젠테이션 레이어(PT)의 역할을 전담하여, 고객 인터페이스 및 내부 창구 직원의 통합 단말(Presentation Tier) 소프트웨어를 구동하고 계정계의 원천 데이터를 전문 통신 형식으로 가공 및 변환하여 노출하는 책임을 수행한다2.

### **한국 코어뱅킹 현대화 추세 및 벤더 참조 아키텍처**

한국 은행권 코어뱅킹 현대화는 시스템 이중화 및 유연한 비즈니스 로직 적용을 목표로 추진되어 왔다. 과거 메인프레임 환경에서 유닉스 유연 전환(Downsizing) 과정을 거치며 개방형 시스템으로 변천하였으며9, 현재는 완전한 클라우드 및 자바(Java) 기반 차세대 뱅킹 플랫폼인 마이크로서비스 아키텍처(MSA) 및 오픈 API 연계 고도화 패러다임으로 나아가고 있다6.

\[레거시 아키텍처 (메인프레임 / 유닉스 C)\]  
  \- 밀결합된 소스 코드 구성 및 유지보수 한계

\[현대화 자바 기반 아키텍처 (Java 기반 프레임워크 & MDD)\] \[cite: 7, 8\]  
  \- 비즈니스 모델링 설계 중심 아키텍처 (Model Driven Development)  
  \- 상품 팩토리 구조화: 복잡한 코드 수정 없이 파라미터 조정으로 신상품 개발 \[cite: 8, 10\]  
  \- 점진적 시스템 오픈 아키텍처 지원 (무중단 가동 전환 성공사례 실재)

삼성SDS, LG CNS, SK C\&C 등 국내 선도 IT 서비스 대기업들은 독자적인 금융 자바 개발 프레임워크(예: LG CNS의 DevOn-Java 등)와 모델기반개발(MDD/MDA) 방법론을 제시해 왔다7. 이 참조 아키텍처 모델은 계정계 비즈니스 설계를 소스코드로 100% 자동 생성하는 레벨 3 수준의 빌드 환경을 구축하여 일관적인 개발 품질을 확보하고 레거시 개발자 의존도를 제어한다8. 특히 전북은행과 광주은행 차세대 프로젝트는 이 MDD 방법론의 정수를 활용해 기존 2년 이상 걸리는 뱅킹 재구축 사이클을 16\~20개월 내외로 압축 수행하는 기록적인 시너지 성과를 이뤄내기도 했다7.

은행의 경계 외부망과 연결되는 가장 체계적인 데이터 연동 인터페이스는 금융결제원 오픈뱅킹 시스템과 금융보안원의 금융 분야 마이데이터 표준 API 규격을 지목할 수 있다30. 오픈뱅킹 공동업무 표준 API는 이용기관이 획득한 핀테크이용번호와 접근토큰(Access Token)을 바탕으로 잔액조회, 거래내역조회를 원격 구동할 수 있게 하며12, 특히 예금주의 정상 유무를 점검하는 '계좌실명조회 API'의 경우 아래 테이블과 같이 POST 요청 방식 및 2-legged OAuth 2.0 표준 규격을 명시하여 운영된다14.

## **제품 적용 시사점**

### **시사점 1: CDC 파이프라인과 내부 가명처리 중계 서버를 결합한 정보계 Read-Only 이식 서사**

JB LocalGuard OS가 실시간 금융 거래상의 리스크 징후를 추적하기 위해 정보계 데이터 스트림을 실시간 확보하되, 고유식별정보(PII) 누출 제로 규칙을 준수하기 위한 최적화 연동 설계이다28.

\[계정계 코어 트랜잭션 DB\]  
         │ (실시간 Write 트랜잭션 수행)  
         ▼  
\[Debezium 로그기반 캡처 커넥터\] (DB 부하 영향 원천 차단)  
         │  
         ▼ (가공 전 실시간 트랜잭션 원시 데이터)  
\[은행 내부 가명처리 중계 Node\]  
         │ (PII 데이터 상시 해싱 및 UUID 변환 필터 가동)  
         ▼ (식별 불가하게 완벽 조치된 가명 위험 토큰 스트림)  
\[Apache Kafka 메시지 버스\] ──► \[JB LocalGuard OS AI 분석 콘솔 엔진\]

이 연동 아키텍처 구조 하에서, AI 에이전트 콘솔 시스템은 은행 내부망 외곽 또는 격리 클라우드 환경 내에 완벽하게 디커플링된 형태로 가동되며, 데이터베이스에 쿼리 부하를 단 1%도 인가하지 않은 채 완벽하게 비식별화 처리된 시계열 트랜잭션 패턴 스트림만을 입력값으로 제공받아 기계 학습 및 검증 예측을 지속 고도화한다4.

단말기에 로드된 WebView 인스턴스 영역은 격리된 샌드박스로 가동되어 외부 AI 서버 단의 악성스크립트 실행 공격 시도를 차단한다25. 넥사크로 컨텍스트 내부와 LocalGuard OS 웹 클라이언트 간 실시간 데이터 바인딩 제어가 요구되는 타이밍에는 넥사크로 전용 연동 가이드인 postMessage 브릿지와 onusernotify 리스너를 실행하여 오직 "가상 토큰 UUID" 변수만 실시간 인계하고, 단말기의 네이티브 로컬 메모리 내부 장부에서만 이를 실명으로 매핑하여 직원에게 렌더링하도록 격리 제어한다18.

### **시사점 3: 인간 승인 단계(Human-in-the-Loop)의 비대칭 제어 및 MCI/EAI 우회 차단 이식 서사**

AI 에이전트 분석 결과 명백한 위험 케이스로 식별되어 실시간 이체 제한 조치를 시도하거나 경고 SMS 발송 행동을 기동할 때, 외부 솔루션이 은행 내부 핵심 수동 제어망에 임의로 Write를 발생시키는 것은 내부 통제 정책상 거부된다. 이를 해결하기 위해 승인 기반 비대칭 연동 경로를 구현한다28.

* **프로세스 흐름**:  
  1. AI 거버넌스 에이전트 콘솔이 특정 가명 토큰 건에 대해 "이체 전면 차단 권고" 혹은 "경고 발송 필요" 위험 의견 초안 및 조치 리포트를 자동으로 렌더링한다28.  
  2. 영업점 직원이 콘솔 내 '승인(Action Execution)' 아이콘을 누르는 순간, WebView 영역은 자체적인 Write 트랜잭션을 은행 내부로 직접 요청하는 대신, 넥사크로 단말 시스템으로 postMessage(action\_token, "APPROVED") 제어 토큰 신호만 상향 전송한다18.  
  3. 신호를 넘겨받은 단말 네이티브 런타임이 사내 인증 처리를 거쳐 기존 은행의 내부 표준 전문 채널 중계기인 MCI(Multi Channel Integration) 채널 허브 측으로 정상적인 원장 차단 및 계정계 트랜잭션 요청을 정규 송출한다15. 이 아키텍처로 인해 에이전트 콘솔 시스템은 일체의 은행 계정계 원장 데이터베이스 쓰기 권한을 필요로 하지 않는 "무결격 무권한 분석기" 상태를 보존할 수 있다.

### **시사점 4: 금융 특화 현실적 마찰점(배치 윈도우, 계정계 변경 동결, 물리 망분리) 극복 서사**

실제 금융 환경 도입 단계에서 빈번히 보고되는 전형적 현장 마찰 요소를 회피하기 위해 맞춤형 우회 설계를 도입한다.

* **배치 윈도우(Batch Window) 지연 대응**: 정보계 시스템의 대량 적재 처리는 주로 야간 주기에 대규모 배치 파일 작업으로 실행되어5, 주간 실시간 판독이 누락되는 리스크를 극복하기 위해 CDC 실시간 이관 채널과 주기적 마이너 배치(Micro-Batch) 보정이 결합된 하이브리드 아키텍처를 도입한다4.  
* **계정계 API 변경 동결(Freeze) 기간 우회**: 은행 차세대 구축 작업 또는 명절 등 주요 마이그레이션 중에는 계정계 소스 및 연동 관문 변경이 전면 금지된다6. LocalGuard OS는 계정계 내부 소스나 구조 수정을 0%로 강제하고 오직 기존에 이미 생성되어 실시간 출력되는 DB 리두 로그(CDC) 및 단말 내부 프론트엔드 연동단(넥사크로 웹뷰)만 접점으로 삼으므로, 동결 정책의 영향 범위 밖에서 상시 독립적 무중단 롤아웃이 보장된다5.  
* **엄격한 금융 망분리 경계 우회**: AI 서버는 물리적으로 완전 차단된 은행 내부 비보안망 외부 세그먼트 혹은 규제 샌드박스 망 내에 배치하고, 오직 사내 가명처리 프록시 서버 단 한 대만 DMZ 및 FEP 게이트웨이를 경유하여 제한된 특정 토픽 데이터만 단방향 송수신하도록 보장하여 금융감독당국의 망분리 통제 심사를 통과한다2.

## **갭 / \[미검증\]**

* **지방은행 단말 PC 성능 편차 및 구형 웹 뷰 엔진 사양**: 비록 넥사크로 플랫폼이 표준 브라우저 환경을 공식 보증하고 있으나17, 전북은행과 광주은행이 차세대 전환 단계에서 최초 가동 및 보급한 사내 단말 런타임 클라이언트(NRE) 내부의 웹 랜더링 엔진(WebView) 버전이 최신 ECMAScript 및 CSS 변수 레이아웃 사양을 완전하게 디버그 없이 구현할 수 있는 최신 크로미움 기반 엔진인지, 혹은 구형 MSIE 엔진에 구속되어 있는 상태인지는 개별 행의 단말 PC 자산 배포 명세에 속하여 외부 공인 자료만으로는 명확히 판단하기 어렵다 \[미검증\].  
* **MCI 및 EAI의 사내 표준 전문 암호화 알고리즘 세부 사양**: 은행 내부망에서 실질적으로 활용되고 있는 MCI 및 EAI 엔진의 사내 표준 데이터 스키마 내에 정의된 개별 변수 매핑 테이블 및 실시간 메시지 정수 암호화 키 관리 매뉴얼은 비공개 내부 규칙이므로 연동을 지원하는 외부 솔루션 엔지니어가 사전에 파악하기 불가능하다15. 이는 프로젝트 수행 초기 단계의 금융 전용 보안 연계 검토 절차를 거쳐 매핑 명세를 취득함으로써 보강되어야 한다2.  
* **지방은행 개별 전산센터 신축 연동 타당성**: 광주은행과 전북은행의 개별 IDC 신설 추진에 따라, 각 센터에 위치한 백업 및 실시간 Active 서버들의 다중 리포지토리 구성과 두 행의 물리적 전용 거점을 연계할 라우팅 가상 터널(SDN) 스펙은 2029년 센터 가동 일정 전까지 비공개 상태로 추정되므로 추후 실사 단계를 생략할 수 없다23.
