# 케이스 단위 메모리를 가진 거버넌드 AI 에이전트 콘솔을 위한 학술 리서치

## 핵심 발견 요약

### 메모리 유형론

학술·인지모형 관점에서 에이전트 메모리는 단순히 “짧은 컨텍스트 vs 긴 컨텍스트”로 끝나지 않는다. 고전 CBR와 인지 아키텍처, 최근 LLM 에이전트 서베이를 함께 보면, 설계상 최소한 **작업기억/단기기억**, **일화기억**, **의미기억**, **절차기억**을 분리하는 편이 재사용성과 거버넌스 모두에 유리하다. Tulving 계열 구분은 “경험(event)으로서의 기억”과 “일반화된 사실 지식”을 분리했고, Soar 같은 인지 아키텍처는 이를 **procedural–semantic–episodic** 장기기억과 **working memory**로 구체화했다. 최근 LLM 에이전트 서베이도 이를 거의 그대로 재해석해, 저장 형태는 달라도 기능적으로는 **경험적 메모리(experiential)**, **사실 메모리(factual/semantic)**, **작업 메모리(working)**를 따로 관리해야 한다고 본다. 이는 “한 케이스의 판단 이력”을 저장하려는 제품 설계와 정합적이다. 즉, 케이스 실행 로그는 일화기억, 정제된 규칙·정책·요약은 의미기억, 승인 절차·워크플로는 절차기억으로 나누는 편이 맞다. [\[1\]](https://pmc.ncbi.nlm.nih.gov/articles/PMC2952732/?utm_source=chatgpt.com)

최근 장기기억 에이전트 연구는 메모리 라이프사이클을 거의 공통적으로 **쓰기(write) → 읽기(retrieve) → 갱신(update) → 요약·추상화(consolidation/reflection) → 망각·보존 정책(forgetting/retention)** 으로 본다. 예를 들어 MemoryBank는 장기기억과 망각 메커니즘을 전면에 두고, Reflexion은 실패 경험을 “반성 텍스트”로 바꿔 다음 에피소드의 의사결정에 재주입하며, ACL 2026 서베이는 이를 **Storage → Reflection → Experience**라는 진화 단계로 정리한다. 따라서 “실행 로그를 그대로 다 보존”하는 방식보다, **원본 일화 + 구조화 요약 + 재사용 규칙**을 분리 저장하는 아키텍처가 문헌적으로 더 강하게 지지된다. [\[2\]](https://arxiv.org/abs/2305.10250)

### CBR와 케이스 표현

고전 CBR의 핵심은 여전히 유효하다. Aamodt와 Plaza의 4R 사이클은 **Retrieve–Reuse–Revise–Retain**이며, 새 문제는 과거 사례를 가져와 재사용하고, 현실·교사·후속 검토로 수정한 뒤, 유용한 경험을 다시 케이스베이스에 축적하는 방식으로 학습한다. 이 프레임은 “한 케이스의 지식이 어떻게 다음 케이스에 쓰이는가”를 설명하는 가장 직접적인 선행연구 틀이다. 중요한 점은 CBR이 단순 검색이 아니라 **검색된 사례를 수정·검증·축적**하는 학습 패러다임이라는 점이다. [\[3\]](https://www.iiia.csic.es/~enric/papers/AICom.pdf)

케이스의 내용도 오래전부터 비교적 안정적으로 정리되어 왔다. Bergmann, Kolodner 계열 정리는 케이스가 최소한 **문제 상황**, **제안된 해결**, **실행 결과**, 그리고 가능하다면 **설명·교훈**을 가져야 생산적으로 재사용된다고 본다. 이 구조는 사용자가 제안한 “문제·근거·판단·결과·다음 행동” 스키마와 거의 평행하다. 특히 “결과”와 “교훈”을 따로 둔다는 점은 다음 케이스 재사용에서 결정적이다. 단순히 최종 답만 저장하면 CBR이 아니라 예시 저장소에 머물지만, **왜 그렇게 판단했는지와 이후 어떤 결과가 났는지**까지 포함하면 진짜 사례 기반 추론이 가능해진다. [\[4\]](https://maxapress.com/data/article/ker/preview/pdf/S0269888906000555.pdf)

### 지식 데이터화와 provenance

케이스 단위 지식을 데이터화할 때는 문헌상 크게 네 계열이 반복된다. 첫째, **속성-값형 스키마**는 구현이 쉽고 검색이 빠르다. 둘째, **프레임·객체·관계형 구조**는 케이스 내부의 관계를 보존해 재사용과 설명에 유리하다. 셋째, **텍스트/약구조 표현**은 문서 기반 케이스 수집에 강하다. 넷째, **지식그래프·온톨로지 기반 표현**은 출처·관계·버전 이력을 붙이기 쉽다. 특히 CBR 표현 연구는 “표현 방식과 유사도 계산은 강하게 결합되어 있다”고 본다. 즉, 검색성을 원하면 벡터·속성 스키마가 낫고, 설명가능성과 규제 대응을 원하면 관계형·그래프형이 낫다. 실무적으로는 **스키마형 최소필드 + 원문텍스트 + 그래프형 provenance**의 삼중 구조가 가장 설득력 있다. [\[5\]](https://maxapress.com/data/article/ker/preview/pdf/S0269888906000555.pdf)

provenance는 이 연구 주제에서 부가 기능이 아니라 핵심 기능이다. W3C 계열에서 provenance는 데이터 생산에 관여한 **entities, activities, people**에 대한 정보이며, 품질·신뢰성 평가에 사용된다. 최근 provenance–XAI 리뷰는 provenance를 복잡한 AI의 설명과 신뢰성을 높이는 매개로 다루고, 지식그래프 추적성 연구는 “현재 스냅샷만 가진 KG”는 규제·감사 요구에 취약하다고 지적한다. 다시 말해, 케이스 메모리에 근거·출처·생성자·수정자·시각·변경 이력을 함께 저장하지 않으면, 그 저장소는 장기지식이 아니라 사실상 비감사성 캐시다. [\[6\]](https://tinnguyen-lab.com/home/files/dint_a_00119.pdf)

### 메모리 활용과 학습

메모리는 단순 회상 장치가 아니라 판단 개선 메커니즘으로 쓰인다. Generative Agents는 에이전트의 모든 경험을 자연어 기록으로 저장하고 이를 고수준 반성으로 합성해 이후 계획에 사용한다. Reflexion은 실패 뒤 언어적 피드백을 생성해 **episodic memory buffer**에 넣고 다음 시도에서 더 나은 결정을 유도한다. MemInsight와 ACL 2026 서베이는 이 흐름을 보다 일반화하여, 저장된 경험이 곧바로 쓰이지 않고 **요약·선별·경험 추상화**를 거쳐 더 높은 재사용 가치를 가진 메모리로 변환되어야 한다고 본다. 이는 케이스 단위 콘솔에서 **AgentRun 로그와 Skill/Policy 레벨 교훈을 분리**해야 한다는 설계 근거가 된다. [\[7\]](https://arxiv.org/abs/2304.03442)

CBR와 LLM/RAG의 결합은 이미 “과거 케이스를 검색해 새 질의에 재활용”하는 방향으로 구체화되고 있다. Wilkerson과 Leake는 일반적인 텍스트 조각 대신 **cases rather than snippets**를 LLM에 주입하고 CBR 절차를 명시적으로 프롬프트하는 실험을 제안했고, CBR-RAG는 법률 QA에서 CBR의 색인어휘와 유사도 지식을 RAG 검색 단계에 결합했다. 즉, 현대적 패턴은 “문서 RAG”보다 “**케이스 RAG**”에 가깝다. 이것이 바로 사용자가 말한 “한 케이스의 지식을 데이터화해 다음 케이스에 재활용”하는 패턴의 현재 학술적 구현이다. [\[8\]](https://homes.luddy.indiana.edu/leake/papers/p-24-02.pdf)

### 고위험 도메인과 한국 맥락

고위험 도메인에서는 케이스 메모리의 가치가 더 커진다. 의료에서는 근거기반 의사결정과 source verification, tamper-evident audit logging을 결합한 임상 의사결정 프레임이 제안되었고, 금융에서는 설명가능한 CBR가 “자연스러운 경제적 해석”을 주는 장점 때문에 금융위험 탐지에 연구되어 왔다. 감사·감사위원회 맥락에서도 과거 사례를 활용한 위험평가 CBR 시스템이 일찍부터 제안되었다. 다만 **“LLM 에이전트의 케이스 메모리”를 금융 규제 실무와 직접 연결한 1차 논문은 아직 희소**하며, 현재 근거는 주로 CBR·설명가능성·provenance·audit logging의 인접 문헌을 조합해 얻어진다. 이 점은 명시적으로 남겨야 한다. [\[9\]](https://www.frontiersin.org/journals/artificial-intelligence/articles/10.3389/frai.2026.1737532/full)

한국 맥락에서는 “자동화된 결정”에 대한 설명요구와 재검토, 그리고 **기준·기초정보의 개요** 제공 의무가 이미 법·서식 차원에서 분명하다. 개인정보보호법 제37조의2와 관련 시행령·고시는 자동화된 결정의 설명 요구권을 규정하고, 신용정보법 관련 서식은 자동화평가 결과, 주요 기준, 이용된 기초정보의 개요를 안내하도록 한다. 이는 케이스 메모리에 **근거, 기준, 입력정보, 판단시점, 재산출 가능성**을 함께 저장하는 설계가 한국 규제 맥락과도 잘 맞는다는 점을 뒷받침한다. [\[10\]](https://www.law.go.kr/LSW//lsLinkCommonInfo.do?ancYnChk=&chrClsCd=010202&lsJoLnkSeq=1029334889&utm_source=chatgpt.com)

## 제품 적용 시사점

### JB LocalGuard OS의 케이스 메모리와의 매핑

아래 매핑은 제시된 내부 구조 **Case → AgentRun → Skill → Evidence → Approval → Audit**를 외부 선행연구의 공통 원리와 연결한 것이다. 비교 대상인 **HagentOS**의 외부 정식 정의는 이 리서치에서 검증하지 않았으므로, 비교는 “범용 세션 중심 에이전트 런타임” 대비의 원리 수준으로 제한한다. 해당 고유명사 자체는 \[미검증\]으로 둔다. [\[52\]](https://aclanthology.org/2026.findings-acl.2069/)

| JB LocalGuard OS 요소 | 학술적으로 대응되는 개념 | 설계 정당화 |
|----|----|----|
| Case | CBR의 “new case / learned case” | 케이스를 최상위 식별자로 두는 것은 4R 사이클의 출발점과 종착점을 시스템 데이터모델로 구현한 것이다. 문제, 해결, 결과, 교훈을 한 단위로 묶어야 이후 재사용이 가능하다. [\[53\]](https://www.iiia.csic.es/~enric/papers/AICom.pdf) |
| AgentRun | 일화기억의 실행 에피소드 | AgentRun은 세션 로그가 아니라 **episodic trace**로 보는 편이 정확하다. 실행 시도, 실패, 반성, 승인 요청까지를 사건 단위로 저장해야 reflection과 revise가 가능하다. [\[54\]](https://openreview.net/pdf?id=vAElhFcKW6) |
| Skill | 절차기억 + 재사용 가능한 적응 지식 | Skill을 단순 tool wrapper가 아니라 “검증된 절차적 기억”으로 다루면, 케이스에서 배운 교훈을 다음 실행 절차로 승격시키는 구조가 된다. 이는 procedural memory 설계와 합치한다. [\[16\]](https://arxiv.org/pdf/2205.03854?utm_source=chatgpt.com) |
| Evidence | provenance가 붙은 의미기억/근거 그래프 | Evidence를 별도 1급 객체로 두는 것은 매우 강한 설계 포인트다. 판단 텍스트 안에 출처를 섞어 넣는 것보다, Evidence를 독립 노드로 두고 Case/Run/Judgment와 연결해야 검색·감사·복구가 쉬워진다. [\[22\]](https://tinnguyen-lab.com/home/files/dint_a_00119.pdf) |
| Approval | Revise 단계의 제도화 | 고전 CBR의 revise는 현실 검증이나 교사 평가를 뜻한다. Approval을 이 revise 단계의 제도적 구현으로 보면, “에이전트가 제안하고 인간/정책이 수정·확정한다”는 거버넌스 논리를 학술적으로 설명할 수 있다. [\[55\]](https://www.iiia.csic.es/~enric/papers/AICom.pdf) |
| Audit | Retain + change history | Audit은 단순 사후조회가 아니라, retain된 케이스가 어떤 근거와 변경 이력을 통해 현재 상태에 왔는지를 남기는 change history 계층이어야 한다. 그래야 케이스가 “재사용 가능한 조직 기억”이 된다. [\[56\]](https://www.utwente.nl/en/eemcs/fois2024/resources/papers/dibowski-full-traceability-and-provenance-for-knowledge-graphs.pdf) |

### 차별화 논리

“케이스 단위 거버넌드 메모리”라는 차별은 학술적으로 충분히 정당화 가능하다. 첫째, 최근 agent memory 문헌은 메모리를 아키텍처의 핵심으로 보지만, 많은 구현은 여전히 “개인화 기억” 혹은 “반성 텍스트 저장” 중심이다. 반면 케이스 중심 설계는 **판단 가능한 업무 단위**를 메모리의 기본 엔티티로 삼는다. 둘째, CBR는 원래부터 “문제–해결–결과–교훈”을 단위로 학습해 왔기 때문에, 케이스 중심 메모리는 장기기억을 **검색 가능한 업무 사례 라이브러리**로 변환한다. 셋째, provenance와 audit trail을 내장하면 이 메모리는 단순 personalization이 아니라 **설명 가능한 조직 기억**이 된다. 넷째, Approval과 Audit을 메모리 모델 안에 포함시키면, 학습·재사용과 규제·감사가 분리되지 않고 한 스키마에서 연결된다. [\[57\]](https://arxiv.org/abs/2404.13501?utm_source=chatgpt.com)

실무적 문장으로 바꾸면, 이 선행연구가 지지하는 차별 포인트는 다음과 같다.\
케이스를 최상위 엔티티로 두고, 그 아래에 실행 일화, 구조화 근거, 승인, 감사 이력을 묶는 제품은 **세션 중심 메모리보다 재사용성이 높고**, **문서 중심 RAG보다 설명가능성이 높으며**, **개인화 메모리보다 규제 적합성이 높다**는 것이다. 특히 한국의 자동화된 결정 설명요구권과 신용정보 자동화평가 설명 체계는, “왜 그런 결론이 나왔는가”를 **사후에 다시 설명할 수 있는 메모리 구조**를 사실상 요구한다. [\[58\]](https://www.law.go.kr/LSW//lsLinkCommonInfo.do?ancYnChk=&chrClsCd=010202&lsJoLnkSeq=1029334889&utm_source=chatgpt.com)

## 갭과 미검증

직접적인 갭도 분명하다. 우선, **금융·법률·의료의 규제 실무와 “LLM 에이전트 장기 케이스 메모리”를 정면으로 연결한 1차 실증연구는 아직 많지 않다.** 현재 가장 강한 근거는 CBR, provenance, KG traceability, audit logging, XAI, clinical decision support를 가로지르는 인접 문헌의 합성이다. 따라서 “케이스 단위 메모리가 규제 대응에 유리하다”는 명제는 강한 설계 가설이지만, 대규모 현장 실증은 앞으로 더 필요하다. [\[59\]](https://www.frontiersin.org/journals/artificial-intelligence/articles/10.3389/frai.2026.1737532/full)

둘째, 2023~2026의 LLM agent memory 문헌은 매우 빠르게 변하고 있으며, 중요한 서베이와 프레임워크 중 일부는 여전히 arXiv·OpenReview·워크숍 단계다. 본 보고서에서는 그런 항목을 가능한 한 보수적으로 \[미검증\] 처리했다. 대표적으로 **Generative Agents의 arXiv 페이지 기반 메타서술**, **MemoryBank**, 일부 2025~2026 서베이와 보안 리뷰는 유용하지만, 보고서 본문에서 제도 설계의 “보조 근거”로만 사용해야 한다. [\[60\]](https://arxiv.org/abs/2304.03442)

셋째, 비교 대상으로 언급된 **HagentOS** 및 기타 정의되지 않은 고유명사는 이 리서치 범위에서 외부 1차 정의를 검증하지 않았다. 따라서 본 보고서의 비교는 특정 제품이 아니라 **“세션 중심 범용 에이전트 런타임” 대 “케이스 단위 거버넌드 메모리 콘솔”**의 일반 원리 비교로 읽는 것이 안전하다. 이 항목은 \[미검증\]이다.
