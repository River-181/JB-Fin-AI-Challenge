# 한국 공공·금융 데이터 API 조달 분석 리포트

## 핵심 발견

RM용 거버넌드 AI Agent 콘솔에서 **즉시 실무 투입 가능성**이 높은 것은 `국토교통부 실거래가 Open API`, `국세청 사업자등록정보 진위확인·상태조회 API`, `소상공인시장진흥공단 상가(상권)정보 API`, `금융위원회 금융통계 Open API`, 그리고 `HUG의 전세보증·보증사고 파일데이터 및 자동변환 API`입니다. 이들은 모두 공식 포털에서 인증 절차, 갱신 주기, 비용, 신청 가능 트래픽 또는 이용허락범위를 직접 확인할 수 있었고, 대부분 `이용허락범위 제한 없음`으로 표시되어 MVP 부록 표에 바로 넣을 수 있는 수준입니다. 특히 실거래가 계열 API는 아파트·연립다세대·단독다가구의 매매·전월세를 모두 커버하고, 국세청 API는 휴·폐업/과세유형을 `1회 100건, 1일 100만건`까지 조회할 수 있어 소상공인 라인에서 실전성이 높습니다. [\[1\]](https://www.data.go.kr/data/15126468/openapi.do?recommendDataYn=Y)

반대로 **주의 API**는 세 부류입니다. 첫째, **등기(IROS/인터넷등기소)**는 `등기정보광장 Open API`가 존재하지만, 확인된 범위는 주로 “등기현황” 중심이며, 전세 권리분석의 핵심인 **개별 부동산 등기사항증명서 본문을 API로 기계 취득하는 공식 경로는 이번 점검에서 확인하지 못했습니다**. 공식 확인이 된 것은 오히려 `등기기록 열람 1통당 700원`, `등기사항증명서 발급 1통당 1,000원`이라는 유상 열람·발급 체계입니다. 둘째, **금융결제원 통합포털 API**는 공개 포털은 있으나 이용대상이 “금융회사, 핀테크, 연구기관 등”이고, 신청·계약·적합성 심사·운영키 발급 흐름이 전제되어 있어 일반적인 퍼블릭 오픈데이터와는 성격이 다릅니다. 셋째, **한국은행 ECOS Open API**는 인증키 신청과 저작권 귀속 문구는 공식적으로 확인되지만, 일괄적인 호출 한도와 재배포 조건의 현행 원문을 본 세션에서 직접 열람 확인하지 못해 해당 부분은 \[미검증\]으로 남겨야 안전합니다. [\[2\]](https://data.iros.go.kr/iu/fa/openFaq.do)

또 하나의 핵심 포인트는 **지방행정 인허가 데이터의 취득 경로가 바뀌었다**는 점입니다. 예전 `localdata.go.kr` 개발센터는 2026년 4월 16일부로 폐쇄되었고, 공식 안내는 `지방행정 인허가정보`와 `생활편의정보`를 이제 `data.go.kr`에서 이용하라고 명시합니다. 따라서 부록에는 더 이상 localdata 직접 연계를 기본 경로로 적지 말고, **행정안전부의 data.go.kr Open API들**을 표준 경로로 적는 것이 맞습니다. [\[3\]](https://www.localdata.go.kr/devcenter/apiGuide.do?menuNo=20002)

## 제품 적용 시사점

JB LocalGuard OS의 **RAG 레이어**에는 변동성이 낮고 재활용성이 큰 공개 데이터를 우선 적재하는 편이 맞습니다. 가장 먼저 적재할 축은 `RTMS 실거래가`, `HUG 전세보증/사고`, `소진공 상가정보`, `행안부 인허가`, `금융위 금융통계`입니다. 이 다섯 종류는 모두 구조화가 잘 되어 있고, 주소·행정구역·주택유형·업종코드 등 공통 키가 있어 사내 데이터와 조인하기 쉽습니다. 반면 등기사항증명서 전문은 유상·건별 취득 구조라서 전체 적재형 RAG보다는 **사건기반 온디맨드 취득 후 단기 캐시**가 더 타당합니다. [\[77\]](https://www.data.go.kr/data/15126468/openapi.do?recommendDataYn=Y)

**룰엔진 입력 설계**는 세 라인으로 분리하는 것이 적절합니다. 소상공인 라인에서는 `휴·폐업`, `영업상태`, `업종·입지`, `은행권 매크로 지표`, 필요시 `정책금리/시장금리`를 넣고, 전세 라인에서는 `전세가율`, `최근 실거래 대비 보증금`, `HUG 사고율`, `보증가입 이력`, `고위험 지역 플래그`를 넣어야 합니다. 사기 라인에서는 금융결제원 데이터가 확보되기 전까지는 `사업자 정상여부`, `통신판매업 인허가`, `사기 다발 통계`, `계좌·ATM 접근성 같은 준공공 정보`를 보조지표로만 쓰는 것이 현실적입니다. [\[78\]](https://www.data.go.kr/data/15081808/openapi.do)

**PII 비반출 전제**에서는 외부 취득 데이터를 사내 고객 데이터와 직접 합친 원본을 외부로 내보내지 말고, 내부에서만 `주소 표준화 → 키 생성 → 가명 결합 → 점수 산출` 순서로 처리해야 합니다. 예를 들어 전세 라인에서는 고객 성명이나 주민등록번호를 외부 API에 보내지 않고, 계약주소를 도로명주소 표준화한 뒤, 내부에서만 부동산 식별키를 만들어 RTMS/HUG/등기열람 결과를 결합하는 방식이 적합합니다. 소상공인 라인에서는 사업자등록번호도 원문 보관 대신 **해시·부분마스킹·상태 플래그화**로 다운스트림에 넘기는 편이 안전합니다. [\[79\]](https://business.juso.go.kr/jst/jstAddressLinkage)

운영 프로세스 관점에서는 **2단계 의사결정 구조**가 가장 현실적입니다. 1단계는 무료·대량 가능한 공공 API로 예비 스코어를 만들고, 2단계는 경계선 이상 건에 한해 사람 승인 아래 유상 등기 열람이나 계약형 금융 API를 호출하는 방식입니다. 이렇게 해야 등기비용과 계약형 API 비용을 통제하면서도 RM에게 설명 가능한 근거를 남길 수 있습니다. 특히 전세사기 대응에서 IROS 열람의 유상성과 자동화 제약 불확실성이 크기 때문에, 모든 건을 선제적으로 긁어오는 구조는 추천하기 어렵습니다. [\[80\]](https://data.iros.go.kr/cm/et/openUtnRgltn.do?sParam=1)

## 갭과 미검증

이번 조사에서 가장 큰 갭은 **개별 부동산 등기사항증명서의 대량 자동취득 가능성**입니다. 공식적으로 확인된 것은 유상 열람·발급 수수료와 등기정보광장 현황 Open API뿐이었고, 개별 권리관계 본문을 API로 안정적으로 대량 취득할 수 있다는 공식 문구는 확보하지 못했습니다. 따라서 전세 권리분석의 핵심 로직을 전부 자동 API에 기대는 설계는 현재 근거상 과감합니다. MVP 부록에는 이 항목을 `사람 승인형 유상 보강 절차`로 적는 편이 보수적이고 정확합니다. [\[81\]](https://data.iros.go.kr/cm/et/openUtnRgltn.do?sParam=1)

**한국은행 ECOS Open API**도 주의가 필요합니다. 인증키 신청 경로와 저작권 귀속 문구는 공식적으로 확인했지만, 사용자가 요구한 수준의 `현행 호출 한도`, `재배포 가능 여부`, `상업적 이용 허용 범위`를 본 세션에서 공식 원문으로 끝까지 열람하지 못했습니다. 이 부분은 부록 표에서 `[미검증]`으로 두고, 실제 제안서 제출 직전 한국은행 약관 페이지를 브라우저 수동 확인으로 다시 캡처하는 것이 맞습니다. [\[82\]](https://ecos.bok.or.kr/api/)

**HUG 자체 포털의 Open API 목록 및 신청 경로**는 검색엔진 미리보기로는 존재가 확인되지만, 공식 페이지 본문 열람은 타임아웃으로 실패했습니다. 다만 data.go.kr에 등록된 HUG 파일데이터와 자동변환 API는 충분히 확인되므로, 당장 제품에는 이 경로를 우선 사용하고, HUG 자체 포털은 차후 재검증 대상으로 두는 편이 안전합니다. 특히 “정보공개 포털 신청 중단 예정, HUG 빅데이터 개방 포털 권장”이라는 문구는 검색 스니펫 수준에서만 확인되었으므로, 본문 부록에서는 `[미검증]` 주석을 붙여야 합니다. [\[83\]](https://www.khug.or.kr/openapi/web/se/ap/seap000002.jsp)

**금융결제원 통합포털 API**는 기술적으로 매우 유용하지만, 공개 오픈데이터라기보다 계약형 인프라에 가깝습니다. 회원가입, 오픈 API 신청, 계약 절차, 적합성 심사, 테스트 수행, 운영키 배부가 공식 절차로 확인되었고, 수수료 존재도 약관 스니펫에서 확인되지만 정확한 금액표와 호출 한도는 이번 점검으로 확정하지 못했습니다. 따라서 부록 표에는 “실제 발급·호출 가능하나, 기관 온보딩과 계약이 필요하며 비용/한도는 별도 계약서 기준”으로 적는 것이 가장 사실에 가깝습니다. [\[84\]](https://openapi.kftc.or.kr/intro/useProcedure)

마지막으로, 사용자 지시의 **정본(canon.md) SSOT 충돌 검증**은 이번 세션에 해당 파일이 제공되지 않아 수행할 수 없었습니다. 따라서 본 리포트는 공개 공식 소스 기준의 조달 분석본이며, 제출 직전에는 보유 중인 canon.md와의 불일치 여부를 별도로 대조해야 합니다.
