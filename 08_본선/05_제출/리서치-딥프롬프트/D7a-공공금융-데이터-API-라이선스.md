# D7a — 공공금융 · 데이터 · API · 라이선스

## 메타
- **도메인**: 실제 취득 가능한 **공공·금융 데이터·개방 API** — 한국은행 ECOS, 국토부 실거래가, 등기(인터넷등기소/IROS), HUG, data.go.kr, 금감원·국세청 등. 소상공인·전세·사기 도메인별 활용 매핑.
- **티어**: ★★★
- **심사항목**: 3·4 (3.5 외부 데이터·API 출처·라이선스·제약 · 4.x MVP 데이터 가용성 검증)
- **실행 배치**: 배치 1 (사실 선행 — 후속 문서가 인용할 데이터 가용성 확보)
- **경계 메모**:
  - **D7a(여기) ↔ D6**: *밖에서 가져오는 외부 데이터=D7a / 행 안에서 흐르는 내부 연동·표준 인터페이스=D6.*
  - **D7a ↔ D7b**: *공공·무료·개방 API=D7a / 상용·유료·대체데이터=D7b.* 유료 상용 데이터는 여기서 다루지 않는다.
  - **D7a ↔ D1**: *데이터/API 가용성·라이선스=D7a / 피해 규모 통계 수치=D1.* 수치 재수집 금지.

## 딥리서치 프롬프트 (이 아래 통째로 붙여넣기)

당신은 **한국 공공·금융 데이터 API 조달 분석가**다. 목표는, RM용 거버넌드 AI Agent 콘솔(JB LocalGuard OS)이 **소상공인 금융부담 / 전세사기 / 보이스피싱·이상거래** 세 도메인에서 실제로 끌어다 쓸 수 있는 **공공·금융 개방 데이터·API**를 빠짐없이 식별하고, 각 API의 **호출 한도·라이선스·갱신주기·비용·인증절차·제약**을 표로 확정하는 것이다. 본선 기능명세서 부록(외부데이터/API/라이선스 표)에 그대로 붙일 수 있는 정확도를 요구한다.

### 알아내야 할 것 (구체 질문)
대상 API/데이터(최소 이 목록은 모두 확인하고, 누락된 유용한 공공 API가 있으면 덤으로 추가):
1. **한국은행 ECOS Open API** — 금리·통화·경기 통계. 인증키 발급, 호출 한도, 라이선스(이용약관/재배포 가능 여부), 갱신주기.
2. **국토교통부 실거래가 공개시스템 / 실거래가 Open API (RTMS)** — 아파트·연립·단독 매매·전월세 실거래가. data.go.kr 제공 여부, 인증키, 트래픽 제한, 라이선스.
3. **등기 정보 — 인터넷등기소 / IROS(부동산등기) 및 등기정보 제공**: 부동산 등기사항(소유권·근저당 등) 열람·발급의 API/전자적 취득 가능성, 비용(건당 수수료), 자동화 허용 여부·약관상 제약, 대량 조회 가능성. (전세 권리분석 핵심 — 가용성과 제약을 특히 정밀히.)
4. **HUG(주택도시보증공사) 데이터** — 전세보증·전세사기·보증사고 관련 공개 데이터/API/통계 다운로드 가용성, 갱신주기, 라이선스.
5. **data.go.kr(공공데이터포털)** — 위 외 소상공인·상권·사업자·금융 관련 표준데이터셋/오픈API: 예) 소상공인시장진흥공단 상가(상권)정보, 국세청 사업자등록 상태조회, 행정안전부 인허가 등. 각각 인증·한도·라이선스·갱신.
6. **금융감독원 / 금융위 / 금융결제원 공개 데이터** — 금융통계정보시스템(FISIS), 금감원 공시·통계 API, 보이스피싱·민원 통계 등 공개 취득 경로.
7. **국세청** — 사업자등록 상태조회 API(휴·폐업 여부), 현금영수증/세금 관련 공개 API의 가용성·인증·제약.
8. **도메인별 활용 매핑**: 위 데이터들을 ① 소상공인(매출·상권·휴폐업·연체위험), ② 전세(실거래가·전세가율·등기 권리·보증사고), ③ 사기(이상거래·피싱 통계·계좌·번호) 세 라인에 각각 어떻게 결합할 수 있는지.
9. (덤 유도) 위 목록 밖에서, 한국 공공/준공공 영역에 **실제 발급·호출 가능한** 유용한 개방 API가 더 있으면(예: 우체국·통신과금·KISA 관련 공개 데이터) 동일 표 형식으로 추가하라.

### 출처 규칙 (반드시 준수)
- ① 발견마다 **URL + 원문 짧은 인용(약관·한도 조항) + 발행처·날짜 + 신뢰도(1차/2차/[미검증])**.
- ② 핵심 사실(호출 한도·라이선스·비용)은 **공식 페이지 원문으로 직접 확인**하고, 가능하면 2출처 교차.
- ③ **링크 실접속 확인 필수.** 접속·확인 못 한 API/조항은 인용하지 말고 [미검증]으로 분리.
- ④ 한국 맥락·최신 우선. 약관/요금은 가능한 한 현행본 기준(개정일 명시).
- ⑤ 정본(_canon.md)과 충돌 시 정본 우선(SSOT). 피해 규모 등 통계 수치는 여기서 재산정하지 말 것(D1 담당).
- **도메인 가드(중위험·이 도메인 전용)**: **라이선스 조항은 원문을 그대로 인용**하라(특히 재배포·상업적 이용·자동화/크롤링 허용 여부, 일 호출 한도). 인증 절차(API키 발급 경로·신청 자격)를 단계로 명시. **미확인 API·미접속 링크는 반드시 [미검증]** 표기. 추정 한도/요금 금지.

### 한국 맥락·기간
- 대상: 대한민국 공공·금융 개방 데이터/API. 현행 약관·요금 기준. 지방은행(전북·광주) RM 업무 적용 관점.

### 출력 형식 (반드시 이 구조 — 표 강제)
1. **핵심 발견** — 즉시 쓸 수 있는 API top과 주의 API(라이선스/한도 제약 큰 것) 요약.
2. **API/데이터 인벤토리 표 (필수)** —
   | API/데이터셋 | 제공기관 | 주요 항목 | 인증/발급 절차 | 호출 한도 | 갱신주기 | 비용 | 라이선스·재배포·자동화 제약(원문 인용) | 신뢰도 | URL |
3. **도메인 활용 매핑 표** — | 도메인(소상공인/전세/사기) | 결합 데이터 | 산출 지표(예: 전세가율, 휴폐업, riskScore 입력) | 제약·주의 |
4. **근거표(라이선스 원문)** — | 주장 | 출처 URL | 발행처·날짜 | 신뢰도 | 원문 인용 |
5. **제품 적용 시사점** — JB LocalGuard OS RAG·룰엔진에 어떤 데이터를 어느 라인에 투입할지, PII 비반출 전제에서 외부 취득 데이터의 가명/집계 처리 포인트.
6. **갭 / [미검증]** — 접속 실패·확인 불가·자동화 금지로 MVP에서 제외해야 할 항목.

### 딥리서치 깊이 지시
각 API의 **공식 이용약관·요금·트래픽 정책 페이지에 실제 접속**해 한도·라이선스 문구를 원문으로 확보하라. 특히 등기(IROS/인터넷등기소)는 자동화·대량조회 허용 여부와 건당 수수료가 전세 라인 실현가능성을 좌우하므로 정밀히 다뤄라. "있다더라" 수준 추정은 금지하고, 확인된 것만 표에 넣고 나머지는 갭으로 분리하라.

## Opus 점검 로그 (2패스)
- **패스 1 (정확도·환각·심사 4 적합성 적대적 비판 → 보완)**: 초안은 API를 나열하되 **라이선스/한도가 모호하면 MVP 검증(심사 4)에서 깨질** 위험, 등기 같은 핵심 데이터의 자동화 제약을 얼버무릴 위험이 있었다. → 출처 규칙에 "약관 원문 인용·실접속 필수·미확인 [미검증]" 강제, 등기(IROS) 자동화·수수료를 별도 질문(3번)으로 격상, 출력에 **표 9열 인벤토리**를 필수화해 기능명세서 부록으로 직결되게 했다.
- **패스 2 (D7b·D1 경계, 누락 적대적 재비판 → 보완)**: (a) 상용·유료 데이터(신용평가·카드매출)가 끼어들 위험 → 경계 메모·가드에서 **유료 상용은 D7b로 배제**. (b) 피해 통계 수치를 또 모으면 D1과 중복 → "수치 재산정 금지(D1)" 명시. (c) 사업자 휴폐업·상권 같은 **소상공인 핵심 공공 API가 빠질** 위험 → 질문 5·7에 국세청 사업자상태·소상공인 상권정보를 명시 추가. 도메인 활용 매핑 표를 분리해 세 라인(소상공인/전세/사기)에 데이터가 빠짐없이 붙는지 자가 점검되게 했다.
