# D14 · 데이터 품질·편향·공정대출 + 한국어 금융 도메인 모델/데이터

> ⚠️ 대외비 — 6/29 공식발표 전 비공개.

**도메인** 데이터·모델 신뢰성 · **티어** ★★ · **심사** 3(구현)·4(검증)·5(리스크) · **배치** 3 · **엔진** 🟠 GPT
**경계**: D7a/D7b=데이터 *취득·라이선스* / **D14=데이터 *품질·편향·공정성* + KO 금융모델 *성능·가용성***. D10=보안(인젝션·유출)과 구분: D14는 *공정성·정확도·데이터결함*.

## 딥리서치 프롬프트 (이 아래 통째로 붙여넣기)

당신은 **금융 AI 신뢰성·데이터 거버넌스 리서처**입니다. 목표는 "RM용 거버넌드 AI Agent 콘솔(소상공인 여신·전세·사기 판단을 보조, 외부 LLM 사용·PII 비반출)"이 **실제로 신뢰할 만큼 데이터·모델이 받쳐주는가**를 입증·반증하는 것입니다. 세 축: (1) 데이터 품질, (2) 공정성·편향(특히 여신/신용 의사결정), (3) 한국어 금융 도메인 모델·데이터 가용성. 마케팅 수사 금지 — 결함·한계·반례를 적극 수집.

### 핵심 질문
**A. 공정성·차별 금지 (여신/신용 AI)**
- 한국의 **공정대출·AI 신용평가 공정성** 관련 규제·가이드(금융위 「금융분야 AI 가이드라인/운영 가이드」, 신용정보법상 자동화평가 대응요구권 §36조의2, 개인정보보호법 자동화 의사결정 대응권). 조문·발효일.
- AI 신용평가의 **편향·차별 사례**와 측정지표(disparate impact, equal opportunity 등). 해외 참고(美 ECOA/FCRA, SR 11-7 모델리스크) — 한국 적용 시사점.
- 설명요구·이의제기 절차가 우리 "승인·감사" 루프와 어떻게 맞물리나.

**B. 데이터 품질·결함이 의사결정에 주는 영향**
- 소상공인(매출·상권)·전세(시세·등기)·사기(이상거래) 도메인 입력데이터의 **결측·지연·오염·표류(drift)** 실태와, 그것이 모델 정확도·오탐에 주는 영향(연구·사례).
- 금융 데이터 품질관리·검증·모니터링 **모범사례·프레임**(데이터 거버넌스, DQ 지표).

**C. 한국어 금융 도메인 모델·데이터 가용성**
- 공개된 **한국어 금융 코퍼스·벤치마크**(예: 금융 NLP 데이터셋·리더보드), **금융 특화 LLM·임베딩**의 존재·성능·라이선스.
- 한국어 금융 RAG/판단에 실제 쓸 수 있는 모델·데이터의 **성능 수준과 공백**. 파인튜닝용 데이터 확보 가능성.

**D. 덤**: 위에 없지만 "데이터·모델이 실제 작동을 받친다/못 받친다"를 보여주는 신호(공공 데이터 품질 이슈, 모델 평가 사례 등).

### 출처 규칙
1. 발견마다 URL+원문 짧은 인용+발행처·날짜+신뢰도(1차/2차/[미검증]).
2. 규제·가이드는 **금융위·금감원·KISA·law.go.kr 1차**, 조문·발효일 명시.
3. 모델·벤치마크는 **논문·공식 repo·리더보드 1차**, 성능수치는 출처·평가셋 명시. 벤더 주장=[2차].
4. **반례·실패·한계를 1개 이상 의무 수집**(없으면 환각 의심).
5. 링크 실접속 못 하면 인용 금지. 한국 맥락·최신.

### 출력 형식
1. **핵심 발견** (공정성/데이터품질/KO모델 3축, 각 2~3 bullet)
2. **근거표** — `주장 | 출처 URL | 발행처·날짜 | 신뢰도 | 원문 인용`
3. **제품 적용 시사점** — 공정성·정확도 리스크를 우리 승인·감사 루프가 어떻게 완화하나 / KO모델 공백을 어떻게 메우나
4. **갭·[미검증]** — 못 찾은 것·약한 출처
5. **환각 점검 로그** — 반례 수집 여부
