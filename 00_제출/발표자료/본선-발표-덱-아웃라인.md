## 핵심 스토리라인 (3줄)

1. **문제**: 지역 금융 RM은 위험 신호가 흩어진 채 수십~수백 케이스를 혼자 대응하며, 소상공인·전세·사기 피해는 매년 최고 기록을 갱신 중이다.
2. **해법 + 최대 차별점**: JB LocalGuard OS는 14종 전문 AI 에이전트가 위험을 조기에 종합 판단하되, **고객 원본 PII는 외부 LLM에 절대 반출하지 않는 4중 거버넌스**로 금융 규제 현장에서 바로 도입 가능한 운영체계다.
3. **증명**: 브라우저에서 지금 돌아가는 MVP로 골든 패스 3종을 보여 주고, PoC→파일럿→내부 적용→상용 로드맵과 JB 공표 추진 방향이 정합함을 논증한다.

### S4. 최대 차별점 — PII 비반출 거버넌스 (4중 방어)

**핵심 메시지**: 외부 프런티어 LLM의 추론력은 활용하되, 고객 원본 PII·신용정보는 절대 외부로 나가지 않는다 — 신용정보법 §40조의2 · 개인정보보호법 §28조의4/5 정합.

| 항목 | 내용 |
|------|------|
| 4중 방어 레이어 | **①데이터 등급제** (public/internal/confidential/restricted(PII)) → 라우팅 결정 |
| | **②토큰화/가명처리** — PII를 토큰으로 치환, 토큰↔원본 키는 국내 볼트에만 보관 |
| | **③모델 라우팅** — 원본 PII 작업→국내·온프레 모델 / 비식별 요약·분류→외부 LLM(토큰만) |
| | **④반출 스캔 + 감사 원장** — 전송 페이로드 패턴 검증 + 무결성 해시 기록 |
| 법령 근거 | 신용정보법 §40조의2 ①②(분리보관·안전조치) · ⑥⑦(재식별 금지·중단·회수·파기) · §3조의2(특별법 우선) |
| | 개인정보보호법 §28조의4(안전조치) · §28조의5(재식별 금지) |
| 규제 환경 | 금융위 망분리 개선 로드맵(2024-08-13, MLS 기반 생성형 AI 클라우드 허용) → "외부 LLM 쓰되 PII 통제"의 정책적 정당화 |
| | 금융보안원 금융 클라우드 이용 가이드(2025 개정) |
| 시각자료 | **4중 방어 레이어 다이어그램** (원본 있음 `mermaid/data.png`) · 토큰화 전/후 비교 스냅샷 |
| 대응 심사항목 | 2.4 · 3.3 · 5.1 · 5.5 |

> 이 슬라이드는 발표 전체에서 **가장 강조해야 할 핵심**. 시연 전 머릿속에 심어 두는 용도.

**6분 메모**: 4중 방어 레이어 시각 1장 + 법령명 언급 30초 (30초)

### S10. 리스크 대응 (5.5 전담)

**핵심 메시지**: 개인정보·보안·환각·설명가능성·책임소재 5가지 리스크에 각각 구체적인 대응이 있고, 법령·규제 정합성까지 확보했다.

| 리스크 | 대응 방안 |
|--------|----------|
| 개인정보·보안 | PII 비반출 4중 방어 (S4) · 신용정보법·PIPA 정합 · 망분리 로드맵 |
| 환각(Hallucination) | 에이전트는 결론만 아니라 **근거·불확실성·다음 확인 항목** 의무 출력 · 근거 없으면 추정 금지 |
| 설명가능성 | 점수 분해 · 출처 칩 · 승인 라우팅으로 "왜 이 결론인가" 항상 제시 |
| 책임소재 | 승인 게이트 + 무결성 감사 원장(누가·무엇·언제·근거 추적) |
| 저작권/출처 | 외부 데이터 라이선스·출처 명시(공공누리·상업이용 조건 확인) |
| 고위험 자동 차단 | riskScore ≥ 85 외부 행동 차단 · fraud 감지 시 L4 외부 접촉 차단 |

| 항목 | 내용 |
|------|------|
| 시각자료 | 리스크 매트릭스 또는 5×2 표 (신규 제작) |
| 대응 심사항목 | 5.5 · 2.4 · 3.5 |
