## 1. Summary (서비스 요약)

**JB LocalGuard OS**는 지역 금융 고객의 위험 신호를 하나의 `Case`로 모으고, 14종의 전문 AI Agent가 스킬을 장착해 **판단 → 행동 초안 → 검증/개선**을 수행하되, **고객 대상 행동은 사람 승인 전까지 차단**하는 금융 AI Agent 운영 콘솔이다.

- **누구를 위한 것인가**: 전북은행·광주은행·JB우리캐피탈의 RM·기업금융·사후관리·준법 담당자(1차 사용자)와, 그 뒤의 지역 소상공인·개인사업자·전세 고객(최종 수혜자).
- **무엇을 푸는가**: 기사·정책공고·시세·등기·상담기록·사기경보가 흩어져 있어 담당자가 위험을 **조기에 모아 판단하고 다음 행동으로 연결**하기 어렵다.
- **어떻게 다른가 (차별점)**: 외부 프런티어 LLM의 추론력은 활용하되 **고객 원본 PII·신용정보는 절대 외부로 반출하지 않는다**(데이터 등급제·토큰화·모델 라우팅·반출 스캔·감사원장의 4중 방어). 단순 챗봇이 아니라 **승인·감사 가능한 내부 운영체계**라 금융권 실도입 가능성이 높다.
- **검증 가능성**: 정적 MVP(브라우저)에서 케이스 선택 → AgentRun → 승인 → 감사 로그 갱신이 실제로 재현된다. 골든 패스 3종(`?demo=sme|jeonse|phishing`).

## 2. 문제 정의 (Problem Definition)

### 2.1 대상 사용자와 사용 상황
- **1차 사용자**: 지역은행/캐피탈의 RM, 기업금융 심사역, 사후관리 담당, 준법·리스크 담당.
- **사용 상황**: 담당자 1인이 수십~수백 케이스를 보유. 위험 징후는 기사·정책·시세·등기·상담·거래·사기경보 등 **출처가 분산**되어 있고, 조기에 종합 판단할 시간이 없다.
- **최종 수혜자**: 전북·광주/전남 기반 소상공인·개인사업자·전세 계약 고객(특히 청년·고령).

### 2.2 현행 Pain Point (검증된 데이터)
- **소상공인 자금압박**: 자영업자 대출 잔액 **1,064.2조원**, 취약 자영업자 **42.7만명(13.7%)**, 취약 차주 연체율 **11.16%**. 2024년 폐업 **100.8만명**(첫 100만 돌파, 폐업률 9.04%, 사업부진 50.2%). → 금리 부담·매출 둔화가 동시에 오는데 담당자는 **사후관리 누락·대응 지연**이 발생. *(한국은행 2024말, 국세청 2024 — 출처: `01_research/pain-point-evidence.md`)*
- **전세사기**: 피해자 누적 인정 **약 3.9만건**, HUG 사고액 **2024년 4.49조원**(역대 최고). 전세가율·권리관계·보증보험 정보가 분산되어 고객·담당자 모두 사전 판단이 어렵다. *(국토부, HUG)*
- **금융사기(보이스피싱)**: 2025년 1~11월 피해액 **1.13조원(+56% YoY)**, 50대 이상 53%. 비정상 콜백·외부 접촉을 조기에 차단할 통제가 필요. *(금융위·경찰청)*

### 2.3 왜 JB금융그룹·금융업 관점에서 중요한가
- JB는 전북·광주/전남이라는 **지역 밀착 기반**과 소상공인·개인사업자·전세 고객 접점이 강하다(전북은행 소상공인 대환대출 취급 **2024년 687억, 1위**). 차주 리스크 조기 탐지·사후관리는 곧 **건전성·상생금융·고객 유지**로 직결된다.
- JB금융그룹은 2025-12-26 **네이버클라우드와 MOU**를 맺고 기업여신 **상담→심사→사후관리** 전반의 AI 적용을 단계적으로 검토한다고 공표했다. 본 제품은 그 **공표된 추진 방향의 실행 수단**이다.
- 김기홍 회장은 2026년을 **"AX(AI 전환) 원년"**으로 선언했다 — 그룹 최우선 전략과 정합.

### 2.4 실제 서비스 과제로의 전환 가능성
문제는 "분산된 위험 신호의 조기 종합·승인형 대응"이라는 **운영 과제**로 명확히 환원되며, 이는 신규 시스템이 아니라 **기존 RM 업무 흐름(상담·심사·사후관리)에 삽입되는 운영 콘솔**로 구현 가능하다.

## 3. 제안 솔루션 개요 (Solution Overview)

### 3.1 핵심 아이디어
`Case → AgentRun → Agent → Skill → Evidence → Approval → Audit` 운영 계약 위에서, **판단·행동·검증을 가진 멀티 에이전트**가 케이스를 처리한다. 챗봇이 고객에게 바로 답하는 대신, **담당자에게 근거 있는 판단과 승인 가능한 행동 초안**을 제공한다.

### 3.2 차별점 ― 데이터 거버넌스 (PII 비반출) 〔최대 차별점·리스크 답변〕
외부 LLM을 쓰되 원본 PII는 외부로 나가지 않는 **4중 방어**:
1. **데이터 등급제**: 모든 필드에 `public/internal/confidential/restricted(PII)` 등급. 등급이 모델 라우팅·반출 가능 여부를 결정.
2. **토큰화/가명처리**: 외부 호출 직전 PII를 토큰으로 치환. 토큰↔원본 매핑은 **국내 볼트**에만 존재.
3. **모델 라우팅**: 원본 PII 필요 작업 → 국내·온프레 모델, 비식별 요약·분류 → 외부 프런티어 모델(토큰 입력만), 공개정보 분석 → 외부 모델.
4. **반출 스캔 + 감사 원장**: 외부 전송 페이로드를 사전 스캔(주민번호/계좌/전화 패턴 + 토큰 검증), 모든 접근·전송·재식별을 무결성 해시 원장에 기록.

근거(검증): 은행은 **신용정보법 §40조의2**(특별법 우선, §3조의2)로 가명처리·**재식별 금지(⑥⑦)**·분리보관(①②)을 규율, 개인정보보호법 **§28조의4(안전조치)·§28조의5(재식별 금지)** 보충. 환경: **전자금융감독규정 §15조①(망분리)**, 금융위 **망분리 개선 로드맵(2024-08-13, 다층보안체계·생성형 AI 클라우드 이용 허용)**, 금융보안원 **금융 클라우드 이용 가이드(2025)**. *(상세: `05_evidence/legal-citation-verification.md`)*

### 3.3 시스템 구성 (개요)
UI(4-zone 콘솔) → 케이스 런타임 → 오케스트레이터 → **전문 Agent 메시 + Skill Registry** → Evidence/거버넌스 게이트 → Approval → Audit. *(상세 다이어그램: `07_아키텍처/`)*

### 3.4 왜 이 접근인가
- 멀티 에이전트(스킬 장착형): **기능 추가 = 스킬 추가**로 확장. 
- 승인 우선 자동화: 완전 자동 발송 금지 → 금융권 내부통제와 정합.
- Evidence·Audit이 일급 객체 → 설명가능성·책임소재 확보.

> 대응: 2.4, 3.1, 3.4, 5.1, 5.5

## 4. 주요 기능 정의 (Key Features)

핵심 기능 **6가지 — 모두 MVP에서 동작 확인**:

| 기능 | 설명 | 동작하는 에이전트 |
| --- | --- | --- |
| **케이스 운영보드/대시보드** | 케이스를 위험·상태·SLA로 큐잉, 칸반 이동이 실제 상태 전환 hook을 발화 | 운영 조율 에이전트 |
| **AgentRun 실행** | 자연어 지시→담당 에이전트가 단계별 로그를 시간순 생성(판단·근거·confidence) | 상환위험 분류/정책금융 매칭/위험신호 조기감지 |
| **스킬 저장소** | 25종 스킬에 위험도·승인 정책 부여, 에이전트에 장착(확장·내부통제 단위화) | 전 에이전트 |
| **데이터 거버넌스 패널** | 토큰화 전/후, 모델 라우팅(국내/외부), 반출 스캔 결과, 근거 법령 칩 | 준법 검토 에이전트 |
| **승인 게이트 + 감사 원장** | 고객 대상 행동 전 사람 승인, 승인/반려/상위검토를 무결성 해시 원장에 기록 | 승인 게이트 / RM·준법 최종 승인자 |
| **전세 Shield 라인** | 전세가율·권리관계·임차인 손실위험·계약 체크리스트·은행 연계 분리 판단 | 전세위험 관리 리드 + 4개 전문 에이전트 |

각 기능은 케이스 상세 페이지에서 **산출물(MD 문서)**로 연결되어 실제 업무 결과물을 생성한다.

## 5. 데이터 및 기술 활용 (Data & Tech Plan)

### 5.1 데이터 (공개·공식 출처 우선, 출처 전부 명시)
- **공공/공식**: 공공데이터포털(사업자상태·상권), 한국은행 ECOS(금리·지표), HUG 전세보증, 국토교통부 실거래가, 금융위/금감원 공시·경보. 라이선스(공공누리 등)·제약 명시. *(전체 인벤토리·라이선스: `01_research/data-api-license-inventory.md`)*
- **내부(가상/시뮬레이션)**: 케이스·상담·거래 맥락은 MVP에서 데모/시뮬레이션으로 표기하여 실데이터와 구분.
- **출처 칩**: 판단 신호마다 `공공데이터/시뮬레이션/추정` 출처 등급을 표시.

### 5.2 기술 (AI Agent 구조 — 판단·행동·검증)
- **멀티 에이전트 오케스트레이션**: 순차(위험감지→분류→초안→준법검토)·병렬(분류·정책·사기 동시)·에스컬레이션(사기 시 외부 차단)·검토 루프(준법 반려→재작성).
- **RAG**: 공공/내부 문서 검색 기반 근거 연결(본선 목표). **Rule Engine**: 승인 레벨 L0–L4·결정 규칙(riskScore≥85 외부행동 차단 등). **모델 라우팅**: 거버넌스 §3.2.
- **현재 MVP**: Vanilla JS/CSS 정적 앱, 네트워크 API 없이 브라우저 상태로 운영 루프 재현. **본선 목표**: 정적 함수 계약(`computeRiskDecision`·`buildDashboardData`·`auditChainRecords`)을 서버 API로 1:1 승격.

### 5.3 외부 데이터·오픈소스·상용 API의 출처·라이선스·제약
- 공공 API: 인증키·호출 제한·공공누리 표기 의무. 등기/실거래는 **자동 대량수집 제약** → 사람 트리거/캐시.
- 상용 LLM API: 상업적 이용 약관·데이터 전송 정책상 **원본 금융 PII 전송 제약** → 본 제품의 PII 비반출 거버넌스의 직접적 근거.
- 오픈소스: 정적 MVP는 외부 의존성 최소(vanilla). 빌드/테스트는 Playwright(Apache-2.0)·python-pptx(MIT). *(상세 표: `01_research/data-api-license-inventory.md`)*

## 6. 사용자 시나리오/유즈케이스 (User Scenario)

### 6.1 대표 유스케이스 — 전주 중앙로 카페 (JBG-104, `?demo=sme`)
1. RM이 **전주 중앙로 카페**(개인사업자, 전북은행, riskScore 88, 운전자금 1.8억·카드매출 둔화) 케이스를 연다.
2. **AgentRun** 실행 → **상환위험 분류 에이전트**가 금리 부담·매출 둔화를 위험 요인으로 분해(근거·confidence 표시).
3. **정책금융 매칭 에이전트**가 대환/정책금융 후보와 필요 서류 체크리스트 생성.
4. **RM 보좌 에이전트**가 고객 콜백 초안 작성 → **준법 검토 에이전트**가 과장·금융조건 확정 표현·PII 노출 검토.
5. 외부 LLM 호출 구간에서는 **데이터 거버넌스**가 PII를 토큰화하고 반출 스캔 통과를 확인.
6. **RM 최종 승인자**가 승인 → 상태 `Approved`, **감사 원장**에 누가·무엇을·언제·어떤 근거로가 기록.

### 6.2 보조 유스케이스
- **전세 보호** (`?demo=jeonse`): 전세위험 관리 리드가 전세가율·등기 권리·임차인 손실위험·계약 체크리스트·은행 연계를 분리 판단, 고위험 매물 경고.
- **소상공인/보이스피싱** (`?demo=phishing`): 이상거래 탐지·차단 에이전트가 비정상 콜백을 감지해 **외부 접촉 차단**, 에스컬레이션 메모 생성.

> 대응: 1.2, 2.2, 3.1, 4.2, 4.3

## 7. 기대 효과 및 향후 확장성 (Expected Impact)

### 7.1 고객가치·업무효율·리스크 감소 (KPI)
- 위험 인지→대응 착수 시간 **50% 단축**, 판단 **100% 근거 연결**, 고객 대상 행동 **100% 승인 게이트 통과**, 사후관리 누락 **0건** 목표, 전세 위험 케이스 **100% 체크리스트 연결**.
- 수익기회: 대환·정책금융 검토 기회 발굴, 사기 피해 차단으로 손실·분쟁 비용 절감.

### 7.2 확장성
- **계열사 확장**: 케이스 스키마·스킬 추가만으로 광주은행·JB우리캐피탈·자산운용으로 확장.
- **업무영역 확장**: 정책금융·전세·사기에서 기업여신 심사·사후관리·WM으로 스킬 추가 확장.
- **고객군 확장**: 소상공인→중소기업→가계→전세 고객.

### 7.3 발전 경로 (PoC → 파일럿 → 내부 적용 → 고객 서비스화)
1. **PoC(현재)**: 정적 MVP로 운영 루프·거버넌스 검증.
2. **파일럿**: 1개 영업본부 RM 대상, 공공 데이터 API + 온프레 모델 연동.
3. **내부 적용**: 사후관리·심사 보조 도구로 내재화(네이버클라우드 MOU 방향과 정합).
4. **고객 서비스화**: 승인된 안내·전세 위험 경고를 고객 채널로 확장.

### 7.4 운영 리스크 대응 (개인정보·보안·저작권·환각·설명가능성·책임소재)
- **개인정보·보안**: PII 비반출 거버넌스(신용정보법 §40-2·PIPA §28-4/§28-5), 망분리 정합.
- **저작권/출처**: 외부 데이터는 라이선스·출처 명시, 인용 추적.
- **환각**: 에이전트는 결론만 내리지 않고 **근거·불확실성·다음 확인 항목**을 남기며, 근거 없으면 추정 금지·사람 검토로 전환.
- **설명가능성**: 모든 판단이 점수 분해·출처 칩·승인 라우팅으로 "왜 이 결론인가"를 제시.
- **책임소재**: 승인 게이트 + 무결성 감사 원장으로 누가 승인/반려했는지 추적.

> 대응: 2.3, 2.5, 5.2, 5.3, 5.4, 5.5
