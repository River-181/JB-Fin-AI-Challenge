# 사용자 스토리

> JB LocalGuard OS · 골든 패스 3종 기반 사용자 스토리 + 인수 기준 (심사 1.2 사용자·상황 정의 · 2.2 업무 프로세스/고객 여정 접점)
> 사실/수치 단일 출처: [`../docs/_canon.md`](../_canon.md) · 데모: `?demo=sme` · `?demo=jeonse` · `?demo=phishing`

## 사용자 역할

- **RM(고객관리 담당자)**: 케이스 검토·고객 안내·승인 1차 주체. 데모상 `RM 최종 승인자`(Human RM Lead) / `RM 담당자`.
- **준법/리스크 담당자**: 법률·권리·금융조건 표현 검토. `준법 최종 승인자`(Human Compliance Lead).
- **수혜 고객**: 지역 소상공인·개인사업자, 전세 임차인, 보이스피싱 위험 고객.

인수 기준은 실행 앱에서 실제로 확인 가능한 동작으로 작성한다(심사 4.2).

## A. 히어로 — 전주 카페 소상공인 자금압박 (`?demo=sme`)

대상 케이스: **전주 중앙로 카페** · 코드 `JBG-104` · 계열사 전북은행 · 개인사업자 · riskScore **88** · 상태 `Approval Pending` · 노출 운전자금 1.8억·카드매출 둔화 · pain 금리 부담 + 매출 둔화.

> **RM으로서** 매출이 둔화된 사업자의 상환 위험과 정책금융 가능성을 한 번에 보고 싶다, **그래야** 케이스마다 자료를 찾아 헤매지 않고 빠르게 상담 준비를 할 수 있다.

- 인수 기준 1: 대시보드에서 운영 지시("전주 카페 케이스의 금리 부담, 정책금융 후보, 고객 안내 초안을 정리해줘")를 입력해 실행하면 AgentRun이 생성되고 실시간 실행 목록에 표시된다.
- 인수 기준 2: 상환위험 분류 에이전트가 상환 스트레스 신호를, 정책금융 매칭 에이전트가 후보 프로그램을 산출하며 각 판단에 근거 출처와 confidence가 남는다.
- 인수 기준 3: 위험 점수가 산정되어 승인 레벨(L3 구간 RM+준법)로 라우팅되고, 고객 안내 초안은 `Approval Pending`으로 승인 큐에 올라간다.

> **RM으로서** 고객에게 나가는 안내를 내가 승인하기 전에는 자동 발송되지 않기를 원한다, **그래야** 표현·조건 오류로 인한 책임 위험을 통제할 수 있다.

- 인수 기준 4: 승인 화면(또는 케이스 상세)의 `승인`은 `Approval Pending` 상태에서만 활성화된다.
- 인수 기준 5: `승인`을 누르면 상태가 `Approved`로 바뀌고, 행위자(RM 담당자)·시각이 활동 이력과 감사 로그에 자동 기록된다.
- 인수 기준 6: `반려` 시 상태는 `Rejected`로 바뀌고 고객 발송 없이 수정 요청만 남는다.

## B. 전세 보호 — 보증금 손실 위험 (`?demo=jeonse`)

대상: 전세사기 대응 라인. 대표 전세 케이스에 전세위험 관리 리드 + 4개 전문 에이전트(전세가율 분석·등기 권리 분석·임차인 손실위험·계약 체크리스트) 배정.

> **RM으로서** 고객이 계약하기 전에 전세가율과 권리관계 위험을 정량으로 보고 싶다, **그래야** 깡통전세·보증금 손실 위험을 사전에 경고할 수 있다.

- 인수 기준 1: 전세 보호 화면의 진단 폼에 전세보증금·주변 매매가·고객 총자산·월 소득·권리관계 신호를 입력하고 `진단 실행`하면 전세가율·자산노출·월 부담률이 계산된다.
- 인수 기준 2: 진단 결과로 위험 점수가 산정되고 케이스 상태가 `Approval Pending`으로 전환된다.
- 인수 기준 3: 등기부/보증보험 원문 확인·은행 상담 연결 게이트가 `pending`으로 남아 계약 전 사람 검토가 강제된다(권리관계가 근저당/신탁 의심이면 자동 pending).

> **전세 고객으로서** 위험하면 계약을 멈추라는 안내와 함께 안전 계약 체크리스트·은행 상담 연결을 받고 싶다, **그래야** 보증금을 지킬 수 있다.

- 인수 기준 4: 산출물로 위험 진단 리포트·계약 전 체크리스트·특약 문구 초안·은행 상담 연결 카드가 생성된다.
- 인수 기준 5: 특약 문구는 "초안으로만 제공" 게이트로 표시되어 확정 안내로 발송되지 않는다.
- 인수 기준 6: HUG 안심전세·국토교통부 전세 정책이 근거 커넥터로 연결되어 판단 근거를 추적할 수 있다.

## C. 보이스피싱 — 외부 접촉 차단 (`?demo=phishing`)

대상: 이상거래 탐지·차단 라인. 이상거래 탐지·차단 에이전트 + 준법 검토 에이전트.

> **RM으로서** 보이스피싱 의심 신호가 있으면 고객에게 어떤 자동 메시지도 나가지 않기를 원한다, **그래야** 사기범에게 악용될 여지를 차단하고 고객을 보호할 수 있다.

- 인수 기준 1: fraud/콜백 위험 pain이 있는 케이스는 조치 유형이 `fraud`로 분류되어 이상거래 탐지·차단 에이전트가 고객 외부 접촉 전에 먼저 실행된다.
- 인수 기준 2: 위험 점수가 높으면(L4 구간) 매트릭스상 "외부 발송 차단/자동 차단 제안"으로 라우팅되고, 고객 접촉 차단 게이트가 `blocked`로 표시된다.
- 인수 기준 3: 외부 발송 없이 보안팀 보고 메모·고객 발송 차단 기록·준법 검토 메모가 산출물로 생성된다.

> **준법 담당자로서** 차단 결정과 근거가 감사 로그에 남기를 원한다, **그래야** 사후에 책임 소재와 판단 근거를 증빙할 수 있다.

- 인수 기준 4: 차단·승인·반려 actor와 시각이 활동 이력·감사 로그에 자동 기록된다.
- 인수 기준 5: 반려해도 고객 발송은 계속 차단 상태로 유지되고 보안팀 수동 검토로만 넘어간다.

## 공통 인수 기준 (운영 계약)

모든 스토리에 공통 적용. 운영 계약: `Case → AgentRun → Agent → Skill → Evidence → Approval → Audit`.

- 고객 대상 행동은 사람 승인 전 자동 실행되지 않는다(완전 자동 발송 금지).
- 모든 판단은 Evidence(근거·출처)와 Audit(감사 로그)으로 연결되며, 에이전트는 결론뿐 아니라 근거·불확실성·다음 확인 항목을 남긴다.
- 원본 PII·개인신용정보는 외부 LLM으로 전송되지 않는다(거버넌스 게이트 통과). PII 포함 커넥터는 외부 모델 분석이 차단된다.
