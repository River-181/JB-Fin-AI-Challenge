# 07 · 데이터 거버넌스 / 개인정보 보호 (핵심 차별점)

> 모듈: `governance/` · 모든 외부 LLM 호출·플러그인 조회의 관문

## 문제 정의

에이전트 추론은 외국(외부) 프런티어 LLM API로 실행될 수 있다. 그러나 금융 고객의 원본 개인정보(PII)·신용정보를 외부로 그대로 전송하면 **개인정보보호법·신용정보법·전자금융거래법·금융권 망분리/클라우드 가이드라인**에 저촉된다. 따라서 "외부 모델은 쓰되 원본 PII는 절대 외부로 내보내지 않는" 구조가 필요하다.

## 4중 방어 설계

### 1) 데이터 등급제 (Data Tiers)
모든 필드에 등급을 부여한다. 등급이 모델 라우팅·반출 가능 여부를 결정한다.

| 등급 | 예시 필드 | 외부 반출 |
| --- | --- | --- |
| `public` | 공시자료, 기사, 법령, 정책 | 허용 |
| `internal` | 케이스 코드, 리스크 점수, 상태 | 허용(비식별 상태) |
| `confidential` | 매출 추이, 대출 잔액(범위화) | 토큰화 후 허용 |
| `restricted` (PII) | 성명, 주민번호, 계좌, 연락처, 주소 | **반출 금지** |

### 2) 비식별 / 토큰화 (Tokenization)
외부 모델 호출 직전 PII를 토큰으로 치환한다. 토큰↔원본 매핑은 **국내 PII 볼트**(온프레/도메스틱)에만 존재한다.

```
원본:  "김민수(900101-1******) 010-1234-5678, 전주시 완산구 ..."
토큰화: "{{CUST_7F2}} {{PHONE_A1}} {{ADDR_3C}}"  ← 외부 모델은 이것만 본다
재식별: 국내 볼트에서만 토큰→원본 (사람 승인·감사 후)
```

가명처리 근거(검증 완료 · 은행은 신용정보법이 특별법 우선, 신용정보법 §3조의2):
- 토큰↔원본 키 **분리보관**·안전조치·처리기록: **신용정보법 §40조의2 ①②⑧** (보충: 개인정보보호법 **§28조의4**)
- **재식별 금지**·재식별 시 즉시 중단·회수·파기: **신용정보법 §40조의2 ⑥⑦** (보충: 개인정보보호법 **§28조의5**)
- 규제 환경: **전자금융감독규정 §15조 ①(3·5호)** 망분리, **금융분야 망분리 개선 로드맵**(금융위 2024-08-13, 다층보안체계 MLS·생성형 AI 클라우드 이용 허용), **금융분야 클라우드컴퓨팅서비스 이용 가이드**(금융보안원 2025 개정). 상세 인용 형식은 [`docs/_canon.md`](../../_canon.md) §4 참조.

### 3) 모델 라우팅 (Model Router)
추론을 민감도로 분기한다.

| 작업 | 라우팅 |
| --- | --- |
| 원본 PII가 필요한 식별/대조 | **국내·온프레 모델** (외부 금지) |
| 비식별 요약·분류·초안 작성 | 외부 프런티어 모델 (토큰화된 입력) |
| 법령/정책/뉴스 등 공개정보 분석 | 외부 모델 (PII 없음) |

### 4) 외부 반출 제한 + 감사 원장
외부로 나가는 모든 페이로드를 전송 전 스캔(주민번호/계좌/전화 패턴 정규식 + 토큰 검증)하여 PII 누출을 차단하고, **모든 접근·전송·재식별을 감사 원장(무결성 해시)에 기록**한다. (기존 audit-ledger 스킬과 연결)

## 데이터 인터페이스

```js
governance = {
  tiers: { fieldName: "public|internal|confidential|restricted" },
  classify(field) -> tier,
  tokenizePII(text) -> { masked, map },     // map은 볼트행, UI엔 미표시
  route(taskKind, hasPII) -> "onprem|external",
  egressScan(payload) -> { safe: bool, hits: [] },   // 외부 반출 스캔
  log(event) -> auditEntry
}
```

## 데모 가시화 (거버넌스 패널 / 케이스 상세 내 섹션)
- 토큰화 **전/후** 텍스트 비교 (원본은 마스킹 표시)
- 이번 케이스에서 어떤 작업이 **국내 vs 외부** 모델로 갔는지 라우팅 표
- 외부 반출 스캔 결과(통과/차단 건수)
- 근거 법령·가이드라인 칩 (플러그인 법령 커넥터와 링크)

## 갈아끼우기
- 등급 규칙(`tiers`)·라우팅 정책(`route`)은 데이터/함수 교체만으로 변경.
- 토큰화 엔진은 인터페이스(`tokenizePII`)만 지키면 실제 비식별 엔진으로 교체 가능.

## 심사 효과
⑤ 리스크 관리의 핵심 답변이자, 금융권 실도입 가능성(규제 준수)을 증명하는 최대 차별점.
