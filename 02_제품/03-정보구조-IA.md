# 정보구조 (IA)

> JB LocalGuard OS · 내비게이션 구조, 4-zone 레이아웃, 객체 모델과 화면 매핑 (심사 3.4 시스템 구성·기능 구조의 구현 가능성)
> 사실/수치 단일 출처: [`../docs/_canon.md`](../_canon.md) · 내비 정의: [`../app/app.js`](app/app.js) `navigation` 배열

## 1. 내비게이션 트리 (실제 사이드바)

사이드바는 3개 섹션·15개 항목으로 구성된다(`navigation` 배열, app.js 상단). 각 항목은 `renderWorkbench()`의 `pages` 라우팅 테이블에서 렌더 함수로 연결된다.

```
JB LocalGuard OS
├─ 운영 업무
│  ├─ 대시보드        (dashboard)  — 전체 현황
│  ├─ 알림함          (inbox)      — 긴급 알림
│  ├─ 케이스          (cases)      — 위험 케이스   ─┐ 클릭 시
│  │   └─ 케이스 상세 (case-detail) — 자율운영 루프 ─┘ 진입
│  ├─ 승인            (approvals)  — 사람 검토
│  ├─ 실행 이력       (runs)       — 에이전트 로그
│  └─ 전세 보호       (jeonse)     — 전세사기 대응
├─ AI 조직
│  ├─ 에이전트 팀     (agents)     — 담당 업무
│  ├─ 조직도          (orgchart)   — 보고 체계
│  ├─ 자동화          (routines)   — 정기 실행
│  └─ 운영 목표       (goals)      — 성과 기준
└─ 시스템 관리
   ├─ 스킬            (skills)     — 기능 묶음
   ├─ 플러그인        (plugins)    — 법령·정책·뉴스·DB (MCP 커넥터)
   ├─ 활동 이력       (activity)   — 처리 기록
   ├─ 비용            (budget)     — 예산 사용률
   └─ 설정            (settings)   — 운영 정책
```

- **운영 업무**: 일상 케이스 처리 흐름(인지→검토→승인). 골든 패스 3종이 이 섹션에서 시작·종료된다.
- **AI 조직**: 에이전트 운영의 메타 정보(누가/어떻게/얼마나 일하는가).
- **시스템 관리**: 능력(스킬)·연결(플러그인)·기록(활동)·비용·정책.

`케이스 상세`는 내비에 직접 노출되지 않고 케이스 카드 클릭으로 진입하는 종속 화면이다(`pages["case-detail"]`).

## 2. 4-Zone 레이아웃

운영 콘솔은 4개 고정 영역으로 구성된다.

```
┌──────────┬─────────────────────────────────────────┬──────────────┐
│ Sidebar  │ Topbar (지시 입력 · 검색 · 데모 코치마크)        │              │
│ (내비게이션)├─────────────────────────────────────────┤ Properties   │
│          │                                         │ (케이스·근거· │
│ 3섹션     │ Workbench                               │  감사 맥락)    │
│ 15항목    │ (대시보드·케이스·승인·전세·스킬·플러그인 등)        │  on/off 토글  │
│          │                                         │  항목 접기/펼침 │
└──────────┴─────────────────────────────────────────┴──────────────┘
```

- **Sidebar**: 내비게이션 트리(`renderNavigation`).
- **Topbar / 지시 영역**: 운영 지시 입력과 데모 코치마크(`demoCoachMarkup`). 대시보드 상단의 자연어 명령 입력으로 AgentRun을 생성한다.
- **Workbench**: 활성 화면 본문(`renderWorkbench` → `pages[activeView]`).
- **Properties**: 선택된 케이스/에이전트/스킬/근거의 맥락 패널(`renderProperties`). on/off 토글(`propertiesOpen`), 내부 항목 접기/펼치기(`collapsedPanelKeys`).

## 3. 객체 모델 (운영 계약)

canon §8 운영 계약: `Case → AgentRun → Agent → Skill → Evidence → Approval → Audit`. 고객 대상 행동은 승인 전 차단되며, 모든 판단은 Evidence와 Audit으로 연결된다.

```
┌────────┐   생성    ┌──────────┐  실행한다  ┌────────┐  장착   ┌────────┐
│  Case  │──────────▶│ AgentRun │──────────▶│ Agent  │────────▶│ Skill  │
│ 위험 단위 │          │  실행 기록  │          │ 전문 AI │         │ 처리능력 │
└───┬────┘           └────┬─────┘           └────────┘         └────────┘
    │                     │ 근거 참조
    │                     ▼
    │                ┌──────────┐
    │                │ Evidence │ 근거·출처
    │                └────┬─────┘
    │  제안 행동           │
    └────────────────────▶┌──────────┐  통과/반려  ┌────────┐
                          │ Approval │──────────▶│ Audit  │ 무결성 기록
                          │ 사람 승인  │           │ 감사 원장 │
                          └──────────┘           └────────┘
```

| 객체 | 정의 | 핵심 필드(앱) | 생성·변경 함수 |
| --- | --- | --- | --- |
| Case | 고객·매물·사업자 위험 작업 단위 | code·customerName·affiliate·region·riskScore·status·pains·gates·audit | `moveCaseToColumn`·`buildManualCase` |
| AgentRun | 에이전트가 맥락·스킬을 장착해 실행한 기록 | id·caseId·agentName·status·startedAt | `startAgentRun`·`dispatchCommand` |
| Agent | 전문 AI 에이전트(표시명) | name·role·skills·reportsTo·status | `agents` 배열·`agentLabel` |
| Skill | 에이전트가 장착하는 처리 능력 | slug·type·approvalPolicy·riskLevel | `skillRack`·`skillBodyPanel` |
| Evidence | 판단 근거·출처(기사·공식·경보·내부 이벤트·플러그인) | id·type·source·url·implication | `evidence`·`pluginRegistry` |
| Approval | 고객 대상 행동의 사람 승인 단계 | status·level·route | `approveCase`·`rejectCase`·`approvalLevelFor` |
| Audit | 행위자·행동·근거·시각의 무결성 기록 | timestamp·actor·action·evidence | `auditChainRecords`·`activity` |

## 4. 화면 ↔ 객체 매핑

| 화면 | 주 객체 | 보조 객체 |
| --- | --- | --- |
| 대시보드 | Case(집계)·AgentRun | Audit·비용 |
| 케이스 / 케이스 상세 | Case | AgentRun·Agent·Evidence·Approval·Audit |
| 승인 | Approval | Case·Audit |
| 실행 이력 | AgentRun | Case |
| 전세 보호 | Case(전세)·Agent | Skill·Evidence·Approval |
| 에이전트 팀 / 조직도 | Agent | Skill·Approval(승인자 위계) |
| 스킬 | Skill | Approval(정책 매트릭스) |
| 플러그인/MCP | Evidence(커넥터) | 거버넌스 게이트 |
| 활동 이력 | Audit | 전 객체 |
| 비용 | AgentRun(토큰)·Agent | — |
| 설정 | (조직·정책·연동) | Approval 정책·Evidence 어댑터 |

## 5. 승인 레벨 계층

`computeRiskDecision` → `approvalLevelFor`가 신호 합산 점수와 조치 유형으로 L0~L4를 산정한다. 상세 매트릭스는 [화면별 기능 명세 §공통 승인 레벨 매트릭스](01-화면별-기능명세.md#공통-승인-레벨-매트릭스). 조직도의 사람 승인자(RM 최종 승인자·준법 최종 승인자)가 위계 정점에서 고객 대상 행동을 통제한다.
