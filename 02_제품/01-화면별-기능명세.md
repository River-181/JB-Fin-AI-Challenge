# 화면별 기능 명세

> JB LocalGuard OS · 실행 앱의 실제 화면 단위 명세 (심사 4.3 기능명세 구체성 · 3.4 구현 가능성)
> 사실/수치 단일 출처: [`../docs/_canon.md`](../_canon.md) · 화면 라우팅: [`../app/app.js`](app/app.js) `renderWorkbench()`
> 에이전트 명칭은 모두 **표시명**(canon §2)으로 표기한다. 영문 내부명은 `agentLabel()`(app.js)로 표시명에 매핑된다.

이 문서는 내비게이션 15개 화면 중 운영자가 직접 다루는 핵심 화면을 다룬다. 각 화면은 `각 화면 = 함수` 형태로 `renderWorkbench()`의 `pages` 라우팅 테이블에 등록되어 있다. 승인 레벨은 공통 매트릭스(L0~L4, [05](#공통-승인-레벨-매트릭스) 참조)를 따른다.

---

## 1. 대시보드 (dashboard)

- **목적**: 운영 조율 에이전트 상태와 케이스 흐름, 비용·효과를 한 화면에서 파악한다.
- **핵심 요소**: 운영 지시 입력창(`commandMarkup`), 지표 그리드(`renderMetrics`), 의사결정 요약(`dashboardDecisionView`), 실시간 실행/지역별 위험도/처리 흐름/위험도 순위/최근 처리 이력, 비용 해석(`dashboardCostView`)·서비스 사이클·월별 추세·데이터 신뢰도·데이터 상태·최근 케이스 패널.
- **사용자 행동**: 자연어 운영 지시 입력 → 실행 버튼 클릭 → AgentRun 생성. 지표·패널 클릭으로 맥락 이동.
- **연결 함수**: `dashboardPage()`, `renderMetrics()`, `buildDashboardData()`, `dispatchCommand()` → `startAgentRun()`. (app.js)
- **승인레벨 관련성**: 지시 실행은 케이스로 귀결되며 결과 행동은 승인 게이트로 보내진다. 대시보드 자체는 L0(관찰·로그) 영역.

## 2. 케이스 (cases)

- **목적**: 지역 금융 위험 케이스를 목록/보드로 읽고 상태를 이동한다.
- **핵심 요소**: 검색창(고객·지역·위험 원인·에이전트), 목록/보드(Kanban) 전환 스위치, 케이스 보드(`renderBoard`). 상태는 `New`·`Agent Running`·`Approval Pending`·`Approved`·`Escalated`·`Rejected`.
- **사용자 행동**: 검색어 입력으로 즉시 필터(`caseSearchQuery`), 보드 모드 전환, 카드 드래그로 컬럼(상태) 이동, 카드 클릭으로 상세 패널 갱신.
- **연결 함수**: `casesPage()`, `renderBoard()`, `moveCaseToColumn(caseId, column)`, `selectCase(caseId)`. (app.js)
- **승인레벨 관련성**: 보드의 `Approval Pending` 컬럼이 승인 큐의 입구. 고위험(점수 높음) 케이스는 승인·준법 통과 전 외부 행동이 차단된다.

## 3. 케이스 상세 (case-detail)

- **목적**: 단일 케이스의 자율운영 루프 전체(감지→근거→분석→매칭→초안→승인 게이트→감사)를 한 화면에서 추적한다.
- **핵심 요소**: 브레드크럼, 헤더(고객·코드·계열사·지역·위험도 칩·상태), 자율운영 타임라인 7단계, 담당 에이전트 루프 카드, 근거 커넥터(플러그인) 칩, 산출물 카드, 게이트 행, 감사 로그.
- **사용자 행동**: `에이전트 실행`·`산출물 생성`·`승인`·`반려`·`감사 JSON` 버튼. 산출물/플러그인 칩 클릭으로 근거 확인. 승인/반려 버튼은 `Approval Pending` 상태에서만 활성화된다.
- **연결 함수**: `caseDetailPage(caseItem)` (**modules.js**), `startAgentRun()`, `approveCase()`, `rejectCase()`, `deliverablesForCase()`, `auditChainRecords()`. (app.js/modules.js)
- **승인레벨 관련성**: 타임라인의 `승인 게이트`(준법 검토 에이전트)·`감사 기록` 단계가 `Approval Pending`이면 미완료로 표시된다. 점수×조치유형으로 L0~L4가 산정된다.

## 4. 승인 (approvals)

- **목적**: 고객 대상 행동·법률/권리 리스크·금융조건 표현을 사람 승인 전까지 차단한다.
- **핵심 요소**: 탭(전체/대기/승인/반려), 승인 대기 항목 목록(`approvalsView`), 항목별 승인·반려 버튼.
- **사용자 행동**: 탭 전환(`approvalTab`), 항목의 `승인` 클릭 → 상태 `Approved`·감사 로그 추가·실행 종료, `반려` 클릭 → 상태 `Rejected`.
- **연결 함수**: `approvalsPage()`, `approveCase(item)`, `rejectCase(item)`, `closeRunsForCase()`. (app.js)
- **승인레벨 관련성**: 이 화면이 승인 게이트의 사람 통제 지점. L1(RM 원클릭)~L3(RM+준법)~L4(외부 발송 차단) 라우팅이 실제 버튼 동작으로 재현된다.

## 5. 전세 보호 (jeonse)

- **목적**: 전세사기 위험 신호·자산노출·계약 전 체크리스트·은행 연계를 전용 에이전트 라인으로 분담한다.
- **핵심 요소**: 대표 전세 케이스 카드, 진단 입력 폼(전세보증금·주변 매매가·고객 총자산·월 소득·권리관계 신호), 진단 결과 KPI(전세가율·자산노출·월 부담률), 기능 그리드(전세위험 관리 리드 + 전세가율 분석/등기 권리 분석/임차인 손실위험/계약 체크리스트/은행 연계 에이전트).
- **사용자 행동**: 폼에 값 입력 → `진단 실행` 제출 → 전세가율/자산노출/주거비 부담 계산 후 `Approval Pending` 리포트 생성. 기능 카드 클릭으로 담당 에이전트·스킬 확인.
- **연결 함수**: `jeonsePage()`, `jeonseView()`, `runJeonseDiagnosis(form)` → `estimatedJeonseScore()` → `createAnalysisResult()`. (app.js)
- **승인레벨 관련성**: 진단 결과는 등기부/보증보험 원문 확인·은행 상담 연결 게이트를 `pending`으로 남겨 계약 전 사람 검토(contract 조치 L1~L3)를 강제한다.

## 6. 에이전트 팀 (agents)

- **목적**: AI 에이전트의 상태·역할·장착 스킬·보고 체계를 업무 범주별로 확인한다.
- **핵심 요소**: 에이전트 헬스 요약(실행 중/승인 대기/유휴/대기 큐·예산 도넛), 업무 범주별 팀 그룹(운영 조율·위험/금융 판단·전세 보호·준법 통제·고객/은행 연계), 보완 필요 사항(`agentReadinessView`).
- **사용자 행동**: 에이전트 카드 클릭 → 오른쪽 상세 패널에 역할·스킬·보고 대상 표시.
- **연결 함수**: `agentsPage()`, `agentHealthView()`, `agentsView()`, `selectAgent(agentId)`, `agentLabel()`. (app.js)
- **승인레벨 관련성**: 준법 검토 에이전트·승인 게이트가 통제 그룹에 속하며, `pending_approval` 상태 에이전트는 승인 대기 건과 연동된다.

## 7. 조직도 (orgchart)

- **목적**: 각 AI 에이전트가 어떤 상위 에이전트 또는 사람 승인자에게 보고하는지 시각화한다.
- **핵심 요소**: 보고 체계 트리(`reportsTo` 기반). 최상위는 RM 최종 승인자·준법 최종 승인자, 그 아래 운영 조율 에이전트, 다시 전문 에이전트.
- **사용자 행동**: 노드 클릭 → 에이전트 상세(역할·스킬) 확인.
- **연결 함수**: `orgChartPage()`, `orgChartView()`, `selectAgent()`. (app.js)
- **승인레벨 관련성**: 사람 승인자가 트리 정점에 있어 "고객 대상 행동은 사람 승인 전 차단"이라는 통제 위계를 구조로 보여준다.

## 8. 스킬 (skills)

- **목적**: 에이전트에 장착되는 금융·리스크·계약·준법 스킬 패키지와 승인 정책을 확인한다.
- **핵심 요소**: 스킬 저장소 목록(슬러그·목적·위험도·사람 승인 요건), 승인 정책 매트릭스(`approvalMatrixView`), 스킬 본문 패널(목적·입력 데이터 등급·처리 절차·임계값·근거).
- **사용자 행동**: 스킬 카드 클릭 → 본문·근거 출처·위험 등급 확인.
- **연결 함수**: `skillsPage()`, `skillsView()`, `approvalMatrixView()`, `selectSkill(slug)`, `skillBodyPanel()` (modules.js). (app.js/modules.js)
- **승인레벨 관련성**: 각 스킬의 `approvalPolicy`(internal only / RM review / approval required / blocks external action / mandatory)가 행동 시 요구 승인 레벨을 결정한다.

## 9. 플러그인 / MCP (plugins)

- **목적**: 에이전트가 일하기 위해 접근하는 법령·정책·뉴스·JB 금융 DB 연결점을 관리한다. 모든 연결은 데이터 거버넌스를 통과한다.
- **핵심 요소**: 커넥터 카드(종류·상태·스코프·거버넌스 등급 배지), 상세 패널(제공 데이터 샘플·거버넌스 처리 방식·테스트 조회·연결/해제). `restricted` 등급(PII 포함)은 토큰화 + 국내·온프레 모델 라우팅 + 외부 반출 스캔이 강제.
- **사용자 행동**: 커넥터 카드 클릭 → 상세 확인, `테스트 조회`·`연결/연결 해제`.
- **연결 함수**: `pluginsPage()`, `pluginDetailMarkup()` (**modules.js**), `selectPlugin()`. (modules.js)
- **승인레벨 관련성**: 거버넌스 게이트가 외부 LLM 조회의 관문. PII 커넥터는 외부 모델 분석 자체가 차단되어 데이터 등급제 통제가 실행 시점에 작동한다.

## 10. 활동 이력 (activity)

- **목적**: 에이전트 착수, 승인 생성, 상위 검토, 승인/반려 기록을 시간순으로 본다.
- **핵심 요소**: 시각·행위자(표시명)·행동·케이스 코드로 구성된 활동 장부(`activityView`).
- **사용자 행동**: 기록 열람(읽기 전용). 승인·실행·전세 진단 등 모든 상태 변경이 자동으로 한 행씩 추가된다.
- **연결 함수**: `activityPage()`, `activityView()`. 기록은 `approveCase`·`rejectCase`·`startAgentRun`·`runJeonseDiagnosis` 등이 `activity.unshift()`로 적재. (app.js)
- **승인레벨 관련성**: 승인/반려/차단 actor와 시각이 남아 감사 추적성(KPI 100%)을 뒷받침한다.

## 11. 비용 (budget)

- **목적**: 에이전트별 예산뿐 아니라 예상 비용·절감 가능 비용·비용 대비 효과를 함께 본다.
- **핵심 요소**: 월간 비용 판단(`dashboardCostView`), 토큰 사용량(일/주/월), 월별 추세, 에이전트별 사용률(`budgetView`).
- **사용자 행동**: 사용률·추세 열람으로 운영 비용 대비 회피 손실(ROI) 해석.
- **연결 함수**: `budgetPage()`, `dashboardCostView()`, `tokenStatsView()`, `budgetView()`. (app.js)
- **승인레벨 관련성**: 비용 자체는 L0(관찰). 다만 외부 모델 호출 비용은 거버넌스 라우팅(국내·온프레 vs 외부 토큰 입력)과 연동되는 운영 지표.

## 12. 설정 (settings)

- **목적**: 조직 프로필, 승인 정책, 외부 연동 어댑터를 관리한다.
- **핵심 요소**: 조직 프로필(전북은행·광주은행·JB우리캐피탈 데모 전환), 승인 정책(L0~L4 자동화 레벨·금지 자동 실행 항목), 외부 연동(뉴스·공식자료·RM 상담 기록·보안 경보 어댑터), 데모 상태 초기화.
- **사용자 행동**: 설정 카드 열람, `데모 상태 초기화` 버튼으로 로컬 저장 상태 리셋.
- **연결 함수**: `settingsPage()`, `settingsView()`, `resetDemoState()`. (app.js)
- **승인레벨 관련성**: 승인 정책 항목이 L0~L4 매트릭스와 "금지 자동 실행"(고객 대상 행동 차단) 원칙을 운영 설정으로 노출한다.

## 공통 승인 레벨 매트릭스

`computeRiskDecision(item)`이 신호별 기여 점수를 합산해 위험 점수를 만들고, 조치 유형(customerNotice / contract / fraud)과 점수 구간으로 `approvalLevelFor(score)`가 레벨을 산정한다. 표시 함수는 `approvalMatrixView()`.

| 레벨 | 점수 | 고객 안내 | 계약/전세 | 사기 차단 | 판단 이유 |
| --- | --- | --- | --- | --- | --- |
| L0 | 0–39 | 내부 기록만 | 내부 기록만 | 모니터링 | 위험 낮음 |
| L1 | 40–59 | RM 원클릭 승인 | RM 확인 | 보안 확인 | 단순 안내 가능 |
| L2 | 60–79 | RM 편집 후 발송 | 체크리스트 확인 | 고객 접촉 보류 | 고객 영향 있음 |
| L3 | 80–89 | RM+준법 승인 | 법률/보증 원문 확인 | 보안팀 승인 | 금융·계약 리스크 높음 |
| L4 | 90–100 | 외부 발송 차단 | 사람 결정 전 차단 | 자동 차단 제안 | 치명 리스크 |

> 주의: 레벨 산정 점수는 raw `riskScore`가 아니라 신호 기여 합산 점수다(`computeRiskDecision` → `approvalLevelFor`). 매트릭스는 점수 구간·조치 유형 기준으로만 해석한다.
