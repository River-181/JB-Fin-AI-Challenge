## 1. 차별성의 두 축

LocalGuard OS는 두 가지를 동시에 만족시켜 기존 접근과 갈라진다:

1. **PII 비반출 데이터 거버넌스** — 외부 프런티어 LLM의 추론력은 활용하되, 고객 원본 PII·신용정보는 절대 외부로 반출하지 않는다. (4중 방어)
2. **승인·감사형 운영체계** — 고객 대상 행동은 사람 승인 전까지 자동 실행을 차단하고, 모든 판단을 Evidence·Audit으로 연결한다.

이 두 축은 "성능 좋은 AI"가 아니라 **"금융권이 실제로 도입할 수 있는 AI"**라는 포지션을 만든다.

---

## 2. 비교 대상별 차별성

### 2.1 vs 일반 챗봇 (generic chatbot)
- 챗봇은 고객/사용자에게 **즉답**한다. 금융 현장에서는 즉답 자체가 리스크다 — 잘못된 금융 조건 안내, PII 노출, 책임소재 불명.
- LocalGuard OS는 즉답 대신 **근거 있는 판단 초안 + 승인 가능한 행동안**을 담당자에게 제공한다. 고객 대상 행동은 승인 게이트(L0–L4)를 통과해야 실행된다.
- 챗봇은 대화 종료 후 흔적이 약하다. LocalGuard OS는 모든 AgentRun이 **무결성 해시 감사 원장**에 남는다.

### 2.2 vs 단일 분석 리포트 (single analysis report)
- 단일 리포트는 "분석 결과"에서 멈춘다. 다음 행동(대환 검토, 정책금융 매칭, 고객 안내, 준법 검토)은 사람이 수작업으로 잇는다 → 단절·누락.
- LocalGuard OS는 `Case → AgentRun → Agent → Skill → Evidence → Approval → Audit` 운영 계약으로 **분석에서 승인·실행·사후관리까지 끊김 없이 연결**한다.
- 리포트는 정적이다. LocalGuard OS는 케이스 상태·SLA를 큐잉해 **사후관리 누락 0건**을 목표로 한다.

### 2.3 vs RPA (룰 기반 자동화)
- RPA는 정해진 규칙대로 동작을 자동 실행한다. 비정형 위험 신호(기사·상담맥락·시세 변화)의 **판단**은 못 하고, 자동 실행은 금융 내부통제와 충돌한다.
- LocalGuard OS는 멀티 에이전트가 **판단·행동 초안·검증**을 수행하되, 실행 직전 **사람 승인**을 강제한다. 즉 "자동 실행"이 아니라 "**판단 보조 + 승인형 자동화**".
- RPA는 근거·불확실성을 남기지 않는다. LocalGuard OS의 에이전트는 결론만이 아니라 **근거·불확실성·다음 확인 항목**을 남긴다 (환각·설명가능성 대응).

| 비교 항목 | 일반 챗봇 | 단일 분석 리포트 | RPA | **JB LocalGuard OS** |
| --- | --- | --- | --- | --- |
| 산출물 형태 | 대화 응답 | 1회성 리포트 | 정해진 동작 실행 | 케이스별 판단·근거·행동 초안 |

| 고객 대상 행동 통제 | 즉답(통제 약함) | 해당 없음 | 무인 자동 실행 | **승인 전 차단 (L0–L4 게이트)** |

| 사후관리 추적 | 없음 | 없음 | 부분 | **케이스 상태·SLA 큐잉 (누락 0건 목표)** |
| 금융권 실도입 정합 | 낮음 | 보통 | 중간 | **높음 (내부통제·규제 정합)** |

## 4. PII 비반출 데이터 거버넌스 — 4중 방어 (핵심 차별점 심화)

외부 LLM을 쓰되 원본 PII가 외부로 나가지 않는 4중 방어:

1. **데이터 등급제**: 모든 필드에 `public/internal/confidential/restricted(PII)` 등급. 등급이 모델 라우팅·반출 가능 여부를 결정.
2. **토큰화/가명처리**: 외부 호출 직전 PII를 토큰으로 치환. 토큰↔원본 매핑은 **국내 볼트**에만 분리 보관.
3. **모델 라우팅**: 원본 PII 필요 작업 → 국내·온프레 모델; 비식별 요약·분류 → 외부 프런티어 모델(토큰 입력만); 공개정보 분석 → 외부 모델.
4. **반출 스캔 + 감사 원장**: 외부 전송 페이로드를 사전 스캔(주민번호/계좌/전화 패턴 + 토큰 검증), 모든 접근·전송·재식별을 무결성 해시 원장에 기록.

### 법령 근거 (반드시 이 형태로만 인용 — canon §4)
- 은행/금융회사는 **신용정보법이 특별법으로 우선**(신용정보법 §3조의2), 개인정보보호법은 보충 적용.
- 가명처리 + 토큰↔원본 키 **분리보관**·안전조치·처리기록: **신용정보법 §40조의2 ①②⑧** (보충: 개인정보보호법 **§28조의4**).
- **재식별 금지**·재식별 시 즉시 중단·회수·파기: **신용정보법 §40조의2 ⑥⑦** (보충: 개인정보보호법 **§28조의5**).
- 규제 환경: **전자금융감독규정 §15조①(망분리)**, 금융위 **금융분야 망분리 개선 로드맵(2024-08-13, 다층보안체계 MLS 기반 생성형 AI 클라우드 이용 허용)**, 금융보안원 **금융분야 클라우드 이용 가이드(2025 개정)**. → "외부 LLM을 쓰되 PII는 통제한다"는 설계의 정책적 정당화.

## 5. 승인·감사형 운영체계 — 새로운 업무 방식으로의 전환 (criterion 5.2)

- **기존 업무 방식**: 담당자가 분산 신호를 수작업 종합 → 판단 → 행동 → 사후관리 추적(누락 빈번).
- **전환된 업무 방식**: 담당자는 케이스 보드에서 AgentRun을 실행하고, 에이전트가 만든 **근거 있는 초안을 검토·승인**하는 역할로 이동. 반복 종합·초안 작성은 에이전트가, 판단 책임·승인은 사람이.
- 이는 챗봇식 "AI가 대신 한다"가 아니라, **"AI가 준비하고 사람이 승인한다"**는 금융 친화적 협업 모델이다. 담당자 1인의 케이스 처리 폭이 넓어지고(Triage time 50% 단축), 모든 행동이 추적 가능(Approval safety·Evidence traceability 100%)해진다.

> 대응: 5.2 (새로운 고객경험/업무방식 전환)
