| **재식별 금지**·재식별 발생 시 즉시 중단·회수·파기 | **신용정보법 §40조의2 ⑥⑦** | 개인정보보호법 **§28조의5** |

| 동의 없는 가명정보 처리(통계·연구·공익) | — | 개인정보보호법 **§28조의2** |

## 8. 운영 계약 (앱과 문서 공통)
`Case → AgentRun → Agent → Skill → Evidence → Approval → Audit`
- 고객 대상 행동은 승인 전 자동 실행 금지. 고위험 fraud는 외부 접촉 차단.
- 승인 레벨 L0(관찰/로그)–L1/L2(RM 검토/승인)–L3(준법/법률)–L4(차단·상위검토).
- 모든 판단은 Evidence와 Audit으로 연결. 에이전트는 결론만 아니라 근거·불확실성·다음 확인 항목을 남김.

## 9. 기술 스택 / 구현 사실 (3.3, 3.4, 4.2용)
- 현재 MVP: Vanilla JS/CSS/HTML 정적 앱 (`app/index.html`, `app/app.js`, `app/modules.js`, `app/styles.css`), `python3 -m http.server`로 구동, 네트워크 API 없이 브라우저 상태로 운영 루프 재현.
- 검증: `scripts/verify_static.py`, Playwright e2e(19 테스트), `node --check`.
- 본선 목표: 정적 함수 계약(`computeRiskDecision`, `buildDashboardData`, `auditChainRecords`, `moveCaseToColumn` 등)을 서버 API로 1:1 승격, RAG(공공/내부 문서)·Rule Engine(승인 레벨)·멀티 에이전트 오케스트레이션.
